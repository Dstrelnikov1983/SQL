Тема 1. Представления (Views)

Именованные запросы, сохранённые в базе данных

Что такое представление?

  • Виртуальная таблица — не хранит данные (обычное VIEW)
  • Именованный запрос — SELECT, сохранённый в каталоге БД
  • Обращение к VIEW аналогично обращению к таблице
  • Упрощает сложные запросы и обеспечивает безопасность

CREATE VIEW: синтаксис

-- Базовый синтаксис
CREATE [ OR REPLACE ] VIEW имя_представления
    [ (список_столбцов) ]
AS
    SELECT ...;

-- Пример для «Руда+»: сводка добычи по шахтам
CREATE OR REPLACE VIEW v_mine_production_summary AS
SELECT
    m.mine_name,
    d.year,
    d.month_name,
    COUNT(*)              AS records_count,
    SUM(fp.tons_mined)    AS total_tons,
    AVG(fp.tons_mined)    AS avg_tons_per_shift,
    SUM(fp.trips_count)   AS total_trips
FROM fact_production fp
JOIN dim_mine m     ON fp.mine_id = m.mine_id
JOIN dim_date d     ON fp.date_id = d.date_id
GROUP BY m.mine_name, d.year, d.month_name, d.month
ORDER BY m.mine_name, d.year, d.month;

Использование представления

-- Теперь используем VIEW как обычную таблицу
SELECT mine_name, total_tons
FROM v_mine_production_summary
WHERE year = 2024 AND month_name = 'Январь';

-- Можно JOIN-ить с другими таблицами
SELECT vps.mine_name, vps.total_tons,
       COUNT(DISTINCT e.equipment_id) AS equipment_count
FROM v_mine_production_summary vps
JOIN dim_mine m ON vps.mine_name = m.mine_name
JOIN dim_equipment e ON m.mine_id = e.mine_id
WHERE vps.year = 2024
GROUP BY vps.mine_name, vps.total_tons;
Как работает: PostgreSQL подставляет SQL из VIEW вместо имени — оптимизатор видит полный запрос и строит план как для обычного SELECT.

VIEW для безопасности данных

-- Скрываем персональные данные операторов
CREATE VIEW v_operator_public AS
SELECT
    operator_id,
    tab_number,
    SUBSTRING(last_name, 1, 1) || '***' AS last_name,
    position,
    qualification,
    mine_id,
    status
FROM dim_operator;

-- Выдаём права только на VIEW, не на таблицу
GRANT SELECT ON v_operator_public TO analyst_role;
REVOKE SELECT ON dim_operator FROM analyst_role;
Практика «Руда+»: аналитики MES-системы видят только обезличенные данные операторов через VIEW. Полные ФИО доступны только кадровой службе.

Обновляемые представления

Когда VIEW можно обновлять (INSERT/UPDATE/DELETE)?

  • VIEW основано на одной таблице
  • Нет GROUP BY, HAVING, DISTINCT, UNION
  • Нет агрегатных функций и подзапросов в SELECT
  • Все NOT NULL столбцы присутствуют (для INSERT)
-- Обновляемое VIEW: активное оборудование
CREATE VIEW v_active_equipment AS
SELECT equipment_id, equipment_name, inventory_number,
       equipment_type_id, mine_id, status
FROM dim_equipment
WHERE status = 'active';

-- Можно обновлять через VIEW!
UPDATE v_active_equipment
SET status = 'maintenance'
WHERE equipment_id = 5;

WITH CHECK OPTION

-- Защита: запрет вставки/обновления данных вне условия VIEW
CREATE VIEW v_active_equipment_safe AS
SELECT equipment_id, equipment_name, inventory_number,
       equipment_type_id, mine_id, status
FROM dim_equipment
WHERE status = 'active'
WITH CHECK OPTION;

-- Теперь это ЗАПРЕЩЕНО — статус не 'active':
UPDATE v_active_equipment_safe
SET status = 'decommissioned'
WHERE equipment_id = 5;
-- ERROR: new row violates check option for view
WITH CHECK OPTION гарантирует, что через VIEW можно вставить/обновить только те строки, которые удовлетворяют условию WHERE представления.

Материализованные представления

VIEW (обычное)

  • Не хранит данные
  • Выполняет запрос каждый раз
  • Всегда актуальные данные
  • Нет накладных расходов на хранение

MATERIALIZED VIEW

  • Хранит результат на диске
  • Выполняет запрос при REFRESH
  • Данные могут устареть
  • Можно создавать индексы!

MATERIALIZED VIEW: пример «Руда+»

-- Ежедневная сводка по качеству руды — тяжёлый запрос
CREATE MATERIALIZED VIEW mv_daily_ore_quality AS
SELECT
    d.full_date,
    m.mine_name,
    sh.shaft_name,
    g.grade_name,
    COUNT(*)            AS samples_count,
    AVG(q.fe_content)   AS avg_fe,
    MIN(q.fe_content)   AS min_fe,
    MAX(q.fe_content)   AS max_fe,
    AVG(q.moisture)     AS avg_moisture
FROM fact_ore_quality q
JOIN dim_date d       ON q.date_id = d.date_id
JOIN dim_mine m       ON q.mine_id = m.mine_id
JOIN dim_shaft sh     ON q.shaft_id = sh.shaft_id
LEFT JOIN dim_ore_grade g ON q.ore_grade_id = g.ore_grade_id
GROUP BY d.full_date, m.mine_name, sh.shaft_name, g.grade_name;

-- Создаём индекс на материализованном представлении!
CREATE INDEX idx_mv_quality_date ON mv_daily_ore_quality(full_date);
CREATE INDEX idx_mv_quality_mine ON mv_daily_ore_quality(mine_name);

Обновление материализованного представления

-- Полное обновление (блокирует чтение)
REFRESH MATERIALIZED VIEW mv_daily_ore_quality;

-- Конкурентное обновление (не блокирует чтение!)
-- Требует UNIQUE INDEX
CREATE UNIQUE INDEX idx_mv_quality_uniq
    ON mv_daily_ore_quality(full_date, mine_name, shaft_name, grade_name);

REFRESH MATERIALIZED VIEW CONCURRENTLY mv_daily_ore_quality;
Для «Руда+»: материализованные представления обновляются по расписанию (pg_cron) — например, каждый час для телеметрии и раз в сутки для сводок по качеству руды. 
-- Автоматическое обновление через pg_cron
SELECT cron.schedule('refresh_ore_quality',
    '0 */1 * * *',  -- каждый час
    'REFRESH MATERIALIZED VIEW CONCURRENTLY mv_daily_ore_quality');

Вопрос для обсуждения

Когда использовать MATERIALIZED VIEW вместо обычного VIEW?

Обсудите в парах (2 минуты):

  • Таблица fact_equipment_telemetry содержит 50 млн строк
  • Сводный отчёт по средним показаниям датчиков за месяц запрашивается 100 раз в день
  • Данные обновляются каждые 30 секунд

Как бы вы решили эту задачу?

Ответ: MATERIALIZED VIEW с обновлением раз в час (CONCURRENTLY) — компромисс между актуальностью и производительностью. 100 запросов будут читать предварительно вычисленный результат.

Ограничения MATERIALIZED VIEW в PostgreSQL

ОграничениеСутьОбходной путь
Нет авто-обновления Данные устаревают сразу после создания / последнего REFRESH pg_cron, внешний планировщик (Airflow, cron)
REFRESH блокирует чтение Обычный REFRESH ставит exclusive lock — SELECT ждут REFRESH ... CONCURRENTLY + уникальный индекс
Полный пересчёт Нет инкрементального обновления — пересчитывается весь запрос Партиционирование, расширение pg_ivm (экспериментальное)
Нет DML Нельзя выполнять INSERT / UPDATE / DELETE Использовать обычную таблицу + триггеры
Нет параметров Нельзя передать аргументы (в отличие от функций) Табличные функции (RETURNS TABLE)
Занимает место Данные + индексы хранятся физически Мониторинг: pg_total_relation_size()
Зависимости ALTER TABLE базовой таблицы не обновляет MV — нужно пересоздать Планирование миграций с учётом MV
Для «Руда+»: MV по качеству руды (~100 тыс. строк) обновляется за секунды. Но MV по телеметрии (~50 млн строк) потребует CONCURRENTLY и займёт минуты.

PostgreSQL vs MS SQL Server

Материализованные представления: два подхода

PostgreSQL
MATERIALIZED VIEW		MS SQL Server
Indexed View
Обновление Ручное (REFRESH) Автоматическое (синхронное)
Актуальность Данные устаревают между REFRESH Всегда актуальные
Влияние на запись Нет влияния на INSERT/UPDATE/DELETE Замедляет каждую запись в базовые таблицы
Гибкость запросов Любые SELECT, JOIN, подзапросы Много ограничений: нет LEFT JOIN, UNION, подзапросов, оконных функций
Агрегатные функции Все Только SUM, COUNT_BIG (AVG через SUM/COUNT_BIG)
Контроль обновления Полный — когда и как часто Нет — обновляется при каждой записи
Требования Уникальный индекс для CONCURRENTLY WITH SCHEMABINDING + уникальный кластерный индекс

MS SQL Server: Indexed View (пример)

-- SQL Server: автоматически обновляемое представление
CREATE VIEW dbo.v_mine_production_summary
WITH SCHEMABINDING  -- обязательно!
AS
SELECT
    mine_id,
    SUM(tons_mined)   AS total_tons,
    COUNT_BIG(*)      AS records_count  -- обязательно вместо COUNT(*)
FROM dbo.fact_production
GROUP BY mine_id;
GO

-- Создание кластерного индекса «материализует» представление
CREATE UNIQUE CLUSTERED INDEX IX_v_mine_prod
ON dbo.v_mine_production_summary(mine_id);

-- Теперь данные обновляются автоматически при каждом
-- INSERT/UPDATE/DELETE в fact_production
Ограничения Indexed View: нельзя использовать LEFT/RIGHT JOIN, OUTER APPLY, подзапросы, UNION, DISTINCT, TOP, оконные функции, HAVING, CUBE, ROLLUP. В Standard Edition — нужен хинт WITH (NOEXPAND).
Вывод: PostgreSQL даёт гибкость и контроль, SQL Server — автоматическую актуальность ценой жёстких ограничений и нагрузки на запись.

Тема 2. Табличные функции

Функции, возвращающие наборы строк

Зачем нужны табличные функции?

  • Параметризация — в отличие от VIEW, принимают аргументы
  • Инкапсуляция логики — сложный SQL внутри функции
  • Переиспользование — вызов из любого запроса
  • Используются в FROM как обычные таблицы

RETURNS TABLE: базовый синтаксис

-- Функция: производственный отчёт для конкретной шахты
CREATE OR REPLACE FUNCTION fn_mine_production_report(
    p_mine_id   INT,
    p_date_from INT,
    p_date_to   INT
)
RETURNS TABLE (
    shift_name      VARCHAR,
    operator_name   TEXT,
    equipment_name  VARCHAR,
    total_tons      NUMERIC,
    total_trips     INT,
    avg_fuel        NUMERIC
) AS $$
BEGIN
    RETURN QUERY
    SELECT
        s.shift_name,
        o.last_name || ' ' || o.first_name,
        e.equipment_name,
        SUM(fp.tons_mined),
        SUM(fp.trips_count)::INT,
        AVG(fp.fuel_consumed_l)
    FROM fact_production fp
    JOIN dim_shift s    ON fp.shift_id = s.shift_id
    JOIN dim_operator o ON fp.operator_id = o.operator_id
    JOIN dim_equipment e ON fp.equipment_id = e.equipment_id
    WHERE fp.mine_id = p_mine_id
      AND fp.date_id BETWEEN p_date_from AND p_date_to
    GROUP BY s.shift_name, o.last_name, o.first_name, e.equipment_name;
END;
$$ LANGUAGE plpgsql;

Вызов табличной функции

-- Вызов как подзапрос в FROM
SELECT *
FROM fn_mine_production_report(1, 20240101, 20240131);

-- Фильтрация и сортировка результата
SELECT shift_name, operator_name, total_tons
FROM fn_mine_production_report(1, 20240101, 20240331)
WHERE total_tons > 100
ORDER BY total_tons DESC;

-- JOIN с другими таблицами
SELECT r.*, m.mine_name
FROM dim_mine m
CROSS JOIN LATERAL fn_mine_production_report(
    m.mine_id, 20240101, 20240131
) r;
CROSS JOIN LATERAL позволяет вызвать функцию для каждой строки левой таблицы — мощный паттерн для параметризованных отчётов.

RETURNS SETOF: возврат набора записей

-- Альтернативный синтаксис: RETURNS SETOF record-type
CREATE OR REPLACE FUNCTION fn_equipment_status_report(
    p_mine_id INT
)
RETURNS SETOF dim_equipment AS $$
BEGIN
    RETURN QUERY
    SELECT *
    FROM dim_equipment
    WHERE mine_id = p_mine_id
      AND status = 'active'
    ORDER BY equipment_name;
END;
$$ LANGUAGE plpgsql;

-- Использование
SELECT equipment_name, inventory_number, model
FROM fn_equipment_status_report(1);
SETOF тип_таблицы — возвращает строки в формате существующей таблицы. Удобно, когда функция — это «фильтр» для таблицы.

SQL-функции: лаконичный вариант

-- Чистая SQL-функция (без plpgsql)
CREATE OR REPLACE FUNCTION fn_downtime_by_reason(
    p_date_from INT,
    p_date_to   INT
)
RETURNS TABLE (
    reason_name     VARCHAR,
    category        VARCHAR,
    total_events    BIGINT,
    total_hours     NUMERIC,
    avg_duration_h  NUMERIC
) AS $$
    SELECT
        dr.reason_name,
        dr.category,
        COUNT(*),
        ROUND(SUM(dt.duration_min) / 60.0, 1),
        ROUND(AVG(dt.duration_min) / 60.0, 2)
    FROM fact_equipment_downtime dt
    JOIN dim_downtime_reason dr ON dt.reason_id = dr.reason_id
    WHERE dt.date_id BETWEEN p_date_from AND p_date_to
    GROUP BY dr.reason_name, dr.category
    ORDER BY total_hours DESC;
$$ LANGUAGE sql;

-- SQL-функции проще и могут быть «встроены» (inlined) оптимизатором
SELECT * FROM fn_downtime_by_reason(20240101, 20240630);

LANGUAGE plpgsql vs LANGUAGE sql

Характеристика LANGUAGE sql LANGUAGE plpgsql
Инлайн-оптимизация Да — оптимизатор «встраивает» тело в вызывающий запрос Нет — выполняется как «чёрный ящик»
Производительность Обычно выше (оптимизатор видит всю картину) Может быть ниже из-за барьера оптимизации
Синтаксис Голый SELECT, без BEGIN/END BEGIN ... RETURN QUERY ... END
Переменные Нет Да (DECLARE ... BEGIN ... END)
Управляющие конструкции Нет IF/ELSE, LOOP, FOR, WHILE
Обработка ошибок Нет EXCEPTION WHEN ...
Несколько RETURN QUERY Нет (один SELECT) Да — условная логика возврата
Правило: если функция — один SELECT с параметрами, используйте LANGUAGE sql. Нужна логика (условия, циклы, обработка ошибок) — LANGUAGE plpgsql.

Пример: одна задача — два подхода

LANGUAGE sql

CREATE FUNCTION fn_downtime_summary(
    p_from INT, p_to INT
)
RETURNS TABLE (
    reason VARCHAR, hours NUMERIC
) AS $$
  SELECT dr.reason_name,
    ROUND(SUM(dt.duration_min)
          / 60.0, 1)
  FROM fact_equipment_downtime dt
  JOIN dim_downtime_reason dr
    ON dt.reason_id = dr.reason_id
  WHERE dt.date_id
    BETWEEN p_from AND p_to
  GROUP BY dr.reason_name
  ORDER BY 2 DESC;
$$ LANGUAGE sql;

LANGUAGE plpgsql

CREATE FUNCTION fn_downtime_summary(
    p_from INT, p_to INT
)
RETURNS TABLE (
    reason VARCHAR, hours NUMERIC
) AS $$
BEGIN
  RETURN QUERY
  SELECT dr.reason_name,
    ROUND(SUM(dt.duration_min)
          / 60.0, 1)
  FROM fact_equipment_downtime dt
  JOIN dim_downtime_reason dr
    ON dt.reason_id = dr.reason_id
  WHERE dt.date_id
    BETWEEN p_from AND p_to
  GROUP BY dr.reason_name
  ORDER BY 2 DESC;
END;
$$ LANGUAGE plpgsql;
Результат одинаковый, но SQL-версия будет встроена оптимизатором в вызывающий запрос — потенциально более эффективный план выполнения.

VIEW vs табличная функция

Характеристика VIEW Табличная функция
Параметры Нет (фиксированный запрос) Да (принимает аргументы)
Оптимизация Встраивается в запрос plpgsql — «чёрный ящик»; SQL — inline
Логика Только SELECT IF/ELSE, циклы, переменные
Обновляемость Да (при условиях) Нет
Безопасность GRANT на VIEW GRANT EXECUTE
Материализация MATERIALIZED VIEW Нет (кешируйте результат)

Тема 3. Производные таблицы

Подзапросы в секции FROM

Что такое производная таблица?

  • Подзапрос в FROM, которому присвоен псевдоним
  • Существует только во время выполнения запроса
  • Не сохраняется в базе данных (в отличие от VIEW)
  • Удобна для поэтапных вычислений

Производная таблица: синтаксис

-- Синтаксис
SELECT ...
FROM (
    SELECT ...  -- подзапрос = производная таблица
) AS alias;

-- Пример: Найти шахты, где среднесуточная добыча > 500 тонн
SELECT sub.mine_name, sub.avg_daily_tons
FROM (
    SELECT m.mine_name,
           AVG(daily.total_tons) AS avg_daily_tons
    FROM (
        SELECT mine_id, date_id, SUM(tons_mined) AS total_tons
        FROM fact_production
        GROUP BY mine_id, date_id
    ) daily
    JOIN dim_mine m ON daily.mine_id = m.mine_id
    GROUP BY m.mine_name
) sub
WHERE sub.avg_daily_tons > 500
ORDER BY sub.avg_daily_tons DESC;

Пример: анализ простоев «Руда+»

-- Найти оборудование, у которого доля внеплановых простоев
-- превышает 30% от общего времени простоев
SELECT
    equipment_stats.equipment_name,
    equipment_stats.total_downtime_h,
    equipment_stats.unplanned_downtime_h,
    ROUND(equipment_stats.unplanned_downtime_h * 100.0
          / equipment_stats.total_downtime_h, 1) AS unplanned_pct
FROM (
    SELECT
        e.equipment_name,
        SUM(dt.duration_min) / 60.0 AS total_downtime_h,
        SUM(CASE WHEN dt.is_planned = FALSE
            THEN dt.duration_min ELSE 0 END) / 60.0 AS unplanned_downtime_h
    FROM fact_equipment_downtime dt
    JOIN dim_equipment e ON dt.equipment_id = e.equipment_id
    WHERE dt.date_id BETWEEN 20240101 AND 20240630
    GROUP BY e.equipment_name
) equipment_stats
WHERE equipment_stats.total_downtime_h > 0
  AND (equipment_stats.unplanned_downtime_h / equipment_stats.total_downtime_h) > 0.3
ORDER BY unplanned_pct DESC;

Проблема: глубокая вложенность

-- Трудно читать: 3 уровня вложенности
SELECT final.mine_name, final.efficiency
FROM (
    SELECT sub2.mine_name,
           sub2.produced / sub2.available * 100 AS efficiency
    FROM (
        SELECT sub1.mine_name, sub1.total_tons AS produced,
               sub1.total_hours * 50 AS available
        FROM (
            SELECT m.mine_name,
                   SUM(fp.tons_mined) AS total_tons,
                   SUM(fp.operating_hours) AS total_hours
            FROM fact_production fp
            JOIN dim_mine m ON fp.mine_id = m.mine_id
            GROUP BY m.mine_name
        ) sub1
    ) sub2
) final
WHERE final.efficiency > 70;
Проблема: вложенные производные таблицы быстро становятся нечитаемыми. Решение — CTE (WITH), которые мы изучим далее!

Тема 4. Общие табличные выражения (CTE)

WITH — мощный инструмент структурирования запросов

CTE = Common Table Expression

  • Именованный подзапрос, определённый перед основным SELECT
  • Делает запрос читаемым и модульным
  • Можно определить несколько CTE в одном запросе
  • Поддерживает рекурсию (RECURSIVE)

CTE: базовый синтаксис

-- Синтаксис

-- Комплексный анализ: добыча + простои + качество
WITH daily_production AS (
    SELECT mine_id, date_id,
           SUM(tons_mined) AS tons, SUM(trips_count) AS trips
    FROM fact_production
    GROUP BY mine_id, date_id
),
daily_downtime AS (
    SELECT e.mine_id, dt.date_id,
           SUM(dt.duration_min) AS downtime_min
    FROM fact_equipment_downtime dt
    JOIN dim_equipment e ON dt.equipment_id = e.equipment_id
    GROUP BY e.mine_id, dt.date_id
),
daily_quality AS (
    SELECT mine_id, date_id, AVG(fe_content) AS avg_fe
    FROM fact_ore_quality
    GROUP BY mine_id, date_id
)
SELECT m.mine_name, d.full_date,
       p.tons, p.trips,
       COALESCE(dt.downtime_min, 0) AS downtime_min,
       q.avg_fe
FROM daily_production p
JOIN dim_mine m ON p.mine_id = m.mine_id
JOIN dim_date d ON p.date_id = d.date_id
LEFT JOIN daily_downtime dt
    ON p.mine_id = dt.mine_id AND p.date_id = dt.date_id
LEFT JOIN daily_quality q
    ON p.mine_id = q.mine_id AND p.date_id = q.date_id
ORDER BY d.full_date, m.mine_name;

CTE: MATERIALIZED vs NOT MATERIALIZED

-- PostgreSQL 12+: управление материализацией CTE

-- CTE вычисляется 1 раз и результат кешируется
WITH heavy_query AS MATERIALIZED (
    SELECT equipment_id, AVG(sensor_value) AS avg_val
    FROM fact_equipment_telemetry
    GROUP BY equipment_id
)
SELECT * FROM heavy_query WHERE avg_val > 100
UNION ALL
SELECT * FROM heavy_query WHERE avg_val < 50;

-- CTE встраивается в основной запрос (по умолчанию)
WITH light_filter AS NOT MATERIALIZED (
    SELECT * FROM dim_equipment WHERE status = 'active'
)
SELECT * FROM light_filter WHERE mine_id = 1;
Когда MATERIALIZED: CTE используется > 1 раза или запрос внутри тяжёлый.
Когда NOT MATERIALIZED: CTE используется 1 раз, оптимизатору нужен доступ к фильтрам.

Ограничения CTE MATERIALIZED

ОграничениеСуть
Барьер оптимизации Предикаты из внешнего запроса не проталкиваются внутрь CTE — индексы базовых таблиц могут не использоваться
Нет индексов на результате В отличие от MATERIALIZED VIEW, на результат CTE нельзя создать индекс
Потребление памяти Результат хранится в work_mem, при переполнении — сброс на диск (temp files)
Время жизни — 1 запрос Результат существует только в рамках текущего оператора (нельзя переиспользовать)
Только SELECT Data-modifying CTE (INSERT/UPDATE/DELETE) всегда материализуются — хинт не применим
WITH RECURSIVE Рекурсивные CTE всегда материализуются — NOT MATERIALIZED для них не работает

Пример: барьер оптимизации

-- MATERIALIZED — фильтр НЕ проталкивается внутрь CTE
WITH production AS MATERIALIZED (
    SELECT mine_id, SUM(tons_mined) AS total
    FROM fact_production           -- сканирует ВСЕ строки
    GROUP BY mine_id               -- агрегирует ВСЕ шахты
)
SELECT * FROM production
WHERE mine_id = 1;                 -- фильтр только снаружи

-- NOT MATERIALIZED — оптимизатор встроит CTE и протолкнёт фильтр
WITH production AS NOT MATERIALIZED (
    SELECT mine_id, SUM(tons_mined) AS total
    FROM fact_production           -- оптимизатор добавит mine_id = 1
    GROUP BY mine_id               -- агрегирует только 1 шахту
)
SELECT * FROM production
WHERE mine_id = 1;
EXPLAIN ANALYZE покажет разницу: при MATERIALIZED — Seq Scan по всей таблице, при NOT MATERIALIZED — Index Scan по mine_id = 1.

Когда использовать MATERIALIZED?

MATERIALIZED — полезен

  • CTE используется > 1 раза — избегаем повторного вычисления
  • CTE содержит volatile-функции (random(), now()) — гарантия одного вычисления
  • Нужна стабильность результата — один и тот же snapshot
  • Тяжёлый запрос, результат маленький

MATERIALIZED — вреден

  • Внешний запрос сильно фильтрует результат CTE
  • CTE возвращает большой объём данных
  • Используется один раз — нет смысла кешировать
  • Критична скорость — барьер оптимизации замедлит план
PostgreSQL 12+ по умолчанию: CTE с 1 ссылкой → NOT MATERIALIZED (встраивается), CTE с > 1 ссылкой → MATERIALIZED (кешируется).

Рекурсивные CTE: WITH RECURSIVE

Зачем нужна рекурсия?

  • Обход иерархий (оргструктура, дерево категорий)
  • Генерация последовательностей (даты, числовые ряды)
  • Обход графов (маршруты в шахте)
-- Структура рекурсивного CTE
WITH RECURSIVE cte_name AS (
    -- Якорный (начальный) запрос
    SELECT ...
    UNION ALL
    -- Рекурсивный запрос (ссылается на cte_name)
    SELECT ...
    FROM cte_name
    JOIN ...
)
SELECT * FROM cte_name;

-- Дополнительная таблица для иерархии локаций
CREATE TABLE dim_location_hierarchy (
    location_id     INT PRIMARY KEY,
    parent_id       INT REFERENCES dim_location_hierarchy(location_id),
    location_name   VARCHAR(200),
    location_type   VARCHAR(50),  -- шахта, ствол, горизонт, штрек, забой
    depth_level     INT
);

Рекурсивный обход дерева локаций

-- Все потомки шахты «Северная» (location_id = 1)
WITH RECURSIVE location_tree AS (
    -- Якорь: корневой элемент
    SELECT location_id, parent_id, location_name,
           location_type, depth_level,
           location_name::TEXT AS full_path,
           1 AS tree_depth
    FROM dim_location_hierarchy
    WHERE location_id = 1   -- Шахта «Северная»

    UNION ALL

    -- Рекурсия: дети текущего уровня
    SELECT child.location_id, child.parent_id,
           child.location_name, child.location_type,
           child.depth_level,
           tree.full_path || ' → ' || child.location_name,
           tree.tree_depth + 1
    FROM dim_location_hierarchy child
    JOIN location_tree tree ON child.parent_id = tree.location_id
)
SELECT
    REPEAT('  ', tree_depth - 1) || location_name AS hierarchy,
    location_type,
    full_path
FROM location_tree
ORDER BY full_path;

CTE: накопительная добыча по дням

-- Нарастающий итог добычи за месяц (шахта «Северная»)
WITH daily_tons AS (
    SELECT d.full_date, SUM(fp.tons_mined) AS day_tons
    FROM fact_production fp
    JOIN dim_date d ON fp.date_id = d.date_id
    WHERE fp.mine_id = 1
      AND d.year = 2024 AND d.month = 1
    GROUP BY d.full_date
),
running_total AS (
    SELECT full_date, day_tons,
           SUM(day_tons) OVER (ORDER BY full_date) AS cumulative_tons,
           ROW_NUMBER() OVER (ORDER BY full_date) AS day_num
    FROM daily_tons
)
SELECT full_date, day_tons,
       ROUND(cumulative_tons, 1) AS cumulative_tons,
       ROUND(cumulative_tons / day_num, 1) AS avg_per_day
FROM running_total
ORDER BY full_date;
Здесь CTE используется не рекурсивно, а для поэтапного построения запроса: сначала агрегация по дням, затем нарастающий итог.

Защита от бесконечной рекурсии

-- Ограничение глубины рекурсии
WITH RECURSIVE location_tree AS (
    SELECT location_id, parent_id, location_name,
           1 AS depth
    FROM dim_location_hierarchy
    WHERE parent_id IS NULL

    UNION ALL

    SELECT child.location_id, child.parent_id,
           child.location_name, tree.depth + 1
    FROM dim_location_hierarchy child
    JOIN location_tree tree ON child.parent_id = tree.location_id
    WHERE tree.depth < 10   -- ОГРАНИЧЕНИЕ ГЛУБИНЫ!
)
SELECT * FROM location_tree;

-- Глобальное ограничение PostgreSQL
SET statement_timeout = '30s';  -- Таймаут для любого запроса
Всегда добавляйте ограничение глубины! Циклические ссылки в данных (parent_id указывает на потомка) вызовут бесконечную рекурсию и исчерпание памяти.

PostgreSQL 14+: обнаружение циклов

-- CYCLE clause (PostgreSQL 14+)
WITH RECURSIVE location_tree AS (
    SELECT location_id, parent_id, location_name,
           ARRAY[location_id] AS path
    FROM dim_location_hierarchy
    WHERE parent_id IS NULL

    UNION ALL

    SELECT child.location_id, child.parent_id,
           child.location_name,
           tree.path || child.location_id
    FROM dim_location_hierarchy child
    JOIN location_tree tree ON child.parent_id = tree.location_id
    WHERE child.location_id != ALL(tree.path)  -- Защита от циклов!
)
CYCLE location_id SET is_cycle USING cycle_path;

-- is_cycle = TRUE для строк, создающих цикл
SELECT * FROM location_tree WHERE NOT is_cycle;

Тема 5. Временные таблицы

Промежуточное хранение данных в рамках сессии

Зачем нужны временные таблицы?

  • Промежуточные вычисления — слишком сложно для одного CTE
  • Многократное использование — результат нужен в нескольких запросах
  • Индексы — можно создать индекс на промежуточных данных
  • ETL-процессы — staging-данные при загрузке

CREATE TEMP TABLE: синтаксис

-- Базовый синтаксис
CREATE TEMPORARY TABLE имя_таблицы (
    столбец1 тип,
    столбец2 тип,
    ...
);

-- Сокращённая запись
CREATE TEMP TABLE имя_таблицы (
    столбец1 тип,
    столбец2 тип
);

-- Создание из результата запроса (самый частый способ)
CREATE TEMP TABLE tmp_mine_totals AS
SELECT
    m.mine_name,
    SUM(fp.tons_mined)       AS total_tons,
    SUM(fp.operating_hours)  AS total_hours,
    COUNT(DISTINCT fp.equipment_id) AS equip_count
FROM fact_production fp
JOIN dim_mine m ON fp.mine_id = m.mine_id
WHERE fp.date_id BETWEEN 20240101 AND 20240331
GROUP BY m.mine_name;

Работа с временной таблицей

-- Можно создавать индексы!
CREATE INDEX idx_tmp_mine ON tmp_mine_totals(mine_name);

-- Использовать в нескольких запросах
SELECT mine_name, total_tons
FROM tmp_mine_totals ORDER BY total_tons DESC;

SELECT mine_name,
       ROUND(total_tons / NULLIF(total_hours, 0), 2) AS tons_per_hour
FROM tmp_mine_totals;

-- Можно INSERT/UPDATE/DELETE
INSERT INTO tmp_mine_totals
VALUES ('Шахта Тестовая', 0, 0, 0);

-- Удаление (или исчезнет при завершении сессии)
DROP TABLE tmp_mine_totals;
Главное отличие от CTE: временная таблица живёт всю сессию, на неё можно создать индексы, и она доступна в любом количестве последующих запросов.

ON COMMIT: управление временем жизни

-- По умолчанию: данные живут до конца сессии
CREATE TEMP TABLE tmp_default (id INT)
ON COMMIT PRESERVE ROWS;       -- значение по умолчанию

-- Очистка данных при COMMIT (структура остаётся)
CREATE TEMP TABLE tmp_transaction_only (id INT)
ON COMMIT DELETE ROWS;

-- Удаление таблицы целиком при COMMIT
CREATE TEMP TABLE tmp_drop_on_commit (id INT)
ON COMMIT DROP;
РежимДанные после COMMITТаблица после COMMITКогда использовать
PRESERVE ROWS Сохраняются Существует Многошаговые вычисления в сессии
DELETE ROWS Удаляются Существует Повторные загрузки в каждой транзакции
DROP Удаляются Удаляется Одноразовые промежуточные данные

Область видимости и особенности

Важные свойства временных таблиц в PostgreSQL

  • Видимость: только текущая сессия — другие пользователи не видят
  • Приоритет имён: TEMP TABLE с тем же именем «перекрывает» обычную таблицу
  • Схема: хранятся в специальной схеме pg_temp
  • WAL-логирование: минимальное — быстрее обычных таблиц
  • VACUUM/ANALYZE: autovacuum не обрабатывает TEMP — нужен ручной ANALYZE после массовой загрузки
  • Статистика: планировщик не имеет актуальной статистики — может строить плохие планы
-- После массовой загрузки — обновить статистику вручную
CREATE TEMP TABLE tmp_telemetry AS
SELECT * FROM fact_equipment_telemetry
WHERE date_id BETWEEN 20240101 AND 20240131;

ANALYZE tmp_telemetry;  -- важно для хороших планов запросов!

UNLOGGED TABLE: альтернатива TEMP

-- UNLOGGED — постоянная таблица без WAL-записи
CREATE UNLOGGED TABLE staging_production (
    mine_name   VARCHAR(100),
    shift_date  DATE,
    tons_mined  NUMERIC,
    loaded_at   TIMESTAMP DEFAULT now()
);

-- Быстрая загрузка данных (нет WAL → быстрее INSERT)
INSERT INTO staging_production (mine_name, shift_date, tons_mined)
SELECT m.mine_name, d.full_date, SUM(fp.tons_mined)
FROM fact_production fp
JOIN dim_mine m ON fp.mine_id = m.mine_id
JOIN dim_date d ON fp.date_id = d.date_id
GROUP BY m.mine_name, d.full_date;
Внимание: UNLOGGED таблицы теряют данные при аварийном перезапуске сервера (crash recovery). Используйте только для данных, которые можно восстановить.

Что выбрать для задачи?

Тема 6. SQL vs DAX: табличные выражения

Как те же задачи решаются в DAX?

SQL VIEW ↔ DAX подходы

SQL (PostgreSQL)

-- VIEW = сохранённый запрос
CREATE VIEW v_production AS
SELECT
    m.mine_name,
    d.year_month,
    SUM(fp.tons_mined) AS tons
FROM fact_production fp
JOIN dim_mine m
    ON fp.mine_id = m.mine_id
JOIN dim_date d
    ON fp.date_id = d.date_id
GROUP BY m.mine_name,
         d.year_month;

DAX (Power BI)

// Вычисляемая таблица (Calculated Table)
// Аналог VIEW в Power BI
Production Summary =
ADDCOLUMNS(
    SUMMARIZE(
        fact_production,
        dim_mine[mine_name],
        dim_date[year_month]
    ),
    "Tons",
    CALCULATE(
        SUM(
          fact_production[tons_mined]
        )
    )
)
Различие: VIEW не хранит данные, DAX Calculated Table — материализуется в памяти (аналог MATERIALIZED VIEW).

SQL CTE ↔ DAX VAR

SQL (CTE)

WITH daily AS (
    SELECT date_id,
       SUM(tons_mined) AS tons
    FROM fact_production
    WHERE mine_id = 1
    GROUP BY date_id
),
stats AS (
    SELECT
        AVG(tons) AS avg_tons,
        MAX(tons) AS max_tons
    FROM daily
)
SELECT avg_tons, max_tons,
       max_tons - avg_tons AS delta
FROM stats;

DAX (VAR)

EVALUATE
VAR DailyTons =
    ADDCOLUMNS(
        VALUES(dim_date[date_id]),
        "Tons",
        CALCULATE(
            SUM(
              fact_production[tons_mined]
            ),
            dim_mine[mine_id] = 1
        )
    )
VAR AvgTons = AVERAGEX(DailyTons, [Tons])
VAR MaxTons = MAXX(DailyTons, [Tons])
RETURN
ROW(
    "AvgTons", AvgTons,
    "MaxTons", MaxTons,
    "Delta", MaxTons - AvgTons
)

Рекурсия: SQL vs DAX

SQL (WITH RECURSIVE)

WITH RECURSIVE loc_tree AS (
    SELECT location_id,
           location_name,
           location_name::TEXT
               AS full_path
    FROM dim_location_hierarchy
    WHERE parent_id IS NULL
    UNION ALL
    SELECT c.location_id,
           c.location_name,
           t.full_path || ' → '
               || c.location_name
    FROM dim_location_hierarchy c
    JOIN loc_tree t
        ON c.parent_id = t.location_id
)
SELECT * FROM loc_tree;

DAX (PATH / PATHITEM)

// DAX: встроенные функции
// для иерархий parent-child

FullPath =

    dim_location[location_id],
    dim_location[parent_id]
)
// Результат: "1|3|7|15"

// Извлечение уровня:
Level2Name =

    dim_location[location_name],
    dim_location[location_id],
    PATHITEM(
        dim_location[FullPath],
        2, INTEGER
    )
)

// Глубина:
Depth = PATHLENGTH(
    dim_location[FullPath]
)

Квиз: проверим знания

Ответьте на вопросы по материалу модуля

Вопрос 1

Какое ключевое отличие MATERIALIZED VIEW от обычного VIEW?

A) MATERIALIZED VIEW не поддерживает JOIN
B) MATERIALIZED VIEW физически хранит результат запроса на диске
C) MATERIALIZED VIEW автоматически обновляется при изменении данных
D) MATERIALIZED VIEW может содержать только агрегатные функции
Ответ: B — MATERIALIZED VIEW хранит результат на диске. Обновляется только по команде REFRESH.

Вопрос 2

Что позволяет WITH CHECK OPTION в представлении?

A) Проверить, существует ли представление
B) Гарантировать, что INSERT/UPDATE через VIEW соответствуют условию WHERE
C) Запретить DROP VIEW
D) Включить автоматическую проверку данных
Ответ: B — WITH CHECK OPTION запрещает вставку/обновление строк, которые не удовлетворяют WHERE-условию представления.

Вопрос 3

Какой DAX-аналог SQL CTE (WITH)?

A) CALCULATE
B) SUMMARIZE
C) VAR ... RETURN
D) FILTER
Ответ: C — конструкция VAR/RETURN в DAX позволяет определить промежуточные вычисления, аналогично CTE в SQL.

Вопрос 4

Рекурсивный CTE обязательно содержит:

A) CROSS JOIN и LATERAL
B) Якорный запрос + UNION ALL + рекурсивный запрос
C) Только UNION (без ALL)
D) Подзапрос в WHERE
Ответ: B — рекурсивный CTE = якорь (начальные данные) + UNION ALL + рекурсивная часть (ссылается сама на себя).

Вопрос 5

Главное преимущество табличной функции перед VIEW:

A) Быстрее выполняется
B) Принимает параметры
C) Можно обновлять данные
D) Поддерживает индексы
Ответ: B — табличная функция принимает аргументы, что позволяет параметризовать запрос (VIEW — фиксированный запрос).

Вопрос 6

Какая DAX-функция используется для обхода иерархий parent-child?

A) TREATAS
B) PATH / PATHITEM
C) HIERARCHIZE
D) RECURSIVE
Ответ: B — PATH строит строку с полным путём от корня, PATHITEM извлекает элемент по позиции, PATHLENGTH — глубину.
