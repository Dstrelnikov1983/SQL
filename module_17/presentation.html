1. Блок EXCEPTION

Структура PL/pgSQL блока с обработкой ошибок

BEGIN
    -- Основной код
    -- Если здесь возникает ошибка...
EXCEPTION
    WHEN условие_1 THEN
        -- ...управление передаётся сюда
    WHEN условие_2 THEN
        -- ...или сюда
    WHEN OTHERS THEN
        -- ...или сюда (любая другая ошибка)
END;
Ключевой принцип: При возникновении ошибки в BEGIN-блоке все изменения данных (INSERT, UPDATE, DELETE) в этом блоке автоматически откатываются. Это неявный SAVEPOINT.

Неявный SAVEPOINT — как это работает

Что делает PostgreSQL «под капотом»

BEGIN                -- ← PostgreSQL неявно создаёт SAVEPOINT
    INSERT INTO ...  -- изменение 1
    UPDATE ...       -- изменение 2
    SELECT 1/0;      -- ошибка!
EXCEPTION
    WHEN division_by_zero THEN
        -- ← PostgreSQL делает ROLLBACK TO SAVEPOINT
        -- INSERT и UPDATE уже откачены
        ...
END;
Аналогия с чистым SQL: PL/pgSQL автоматизирует паттерн SAVEPOINT → ROLLBACK TO SAVEPOINT, который иначе пришлось бы писать вручную.

Пошаговый механизм

  1. 1 Вход в BEGIN-блок — PostgreSQL создаёт неявный SAVEPOINT
  2. 2 Выполнение кода — INSERT, UPDATE, DELETE выполняются нормально
  3. 3 Ошибка — выполнение прерывается
  4. 4 Откат — ROLLBACK TO SAVEPOINT, все изменения в блоке отменяются
  5. 5 Обработка — управление переходит в секцию EXCEPTION
Если ошибки не было — SAVEPOINT просто освобождается, все изменения сохраняются.

Пример: откат ВСЕХ изменений в блоке

DO $$
BEGIN
    -- Шаг 1: вставляем запись — она РЕАЛЬНО попадает в таблицу
    INSERT INTO dim_mine (mine_id, mine_name, location, depth_meters, status)
    VALUES (999, 'Тестовая шахта', 'Тест', 100, 'active');

    -- Шаг 2: здесь ошибка — дубликат PK
    INSERT INTO dim_mine (mine_id, mine_name, location, depth_meters, status)
    VALUES (999, 'Дубликат', 'Тест', 100, 'active');  -- unique_violation!

EXCEPTION
    WHEN unique_violation THEN
        -- К этому моменту INSERT из шага 1 УЖЕ ОТКАЧЕН
        RAISE NOTICE 'Обе вставки отменены!';
END;
$$;

-- Проверяем: записи с mine_id = 999 НЕТ в таблице
SELECT * FROM dim_mine WHERE mine_id = 999;  -- пусто!
Важно: откатилась не только 2-я вставка (упавшая), но и 1-я (успешная). SAVEPOINT был создан на входе в блок — откат идёт к этой точке.

Что откатывается, а что нет

Элемент Откатывается? Пояснение
INSERT / UPDATE / DELETE Да Все DML-операции в блоке отменяются
Переменные (DECLARE) Нет Сохраняют значение на момент ошибки
Последовательности (SERIAL) Нет nextval() необратим — будут «дырки» в ID
RAISE NOTICE до ошибки Нет Сообщения уже отправлены клиенту

Переменные vs данные

DO $$
DECLARE
    v_counter INT := 0;
BEGIN
    v_counter := 1;                      -- переменная = 1
    INSERT INTO some_table VALUES (1);   -- строка в таблице
    v_counter := 2;                      -- переменная = 2
    SELECT 1/0;                          -- ошибка!
EXCEPTION WHEN OTHERS THEN
    -- v_counter = 2     ← переменная НЕ откатилась
    -- INSERT отменён    ← данные ОТКАТИЛИСЬ
    RAISE NOTICE 'counter = %', v_counter;  -- выведет 2
END;
$$;
Практический вывод: переменные-счётчики можно использовать для подсчёта ошибок — они сохраняются после отката.

Вложенные блоки — свой SAVEPOINT для каждой итерации

DO $$
DECLARE
    rec RECORD;
    v_ok INT := 0;  v_err INT := 0;
BEGIN
    FOR rec IN SELECT * FROM staging_data
    LOOP
        BEGIN  -- ← свой SAVEPOINT для каждой итерации
            INSERT INTO target_table VALUES (rec.id, rec.value);
            v_ok := v_ok + 1;
        EXCEPTION WHEN OTHERS THEN
            -- ← откатывается ТОЛЬКО эта одна вставка
            v_err := v_err + 1;
            RAISE WARNING 'Ошибка строки %: %', rec.id, SQLERRM;
        END;
        -- остальные итерации продолжаются!
    END LOOP;
    RAISE NOTICE 'Итого: % OK, % ошибок', v_ok, v_err;
END;
$$;
Ключевой паттерн: без вложенного блока ошибка на 5-й записи откатила бы записи 1–4. С вложенным — каждая запись независима.

Накладные расходы SAVEPOINT

Производительность: блок с EXCEPTION работает медленнее, чем без него, даже если ошибка не возникает. PostgreSQL должен создавать и освобождать SAVEPOINT на каждой итерации.

Рекомендации

  • Используйте EXCEPTION когда ошибки ожидаемы (внешние данные, пакетная загрузка)
  • Избегайте EXCEPTION для логики, которую можно проверить через IF
  • Пример: вместо перехвата division_by_zero — проверьте IF v_hours != 0
Вопрос для обсуждения: В каких сценариях на «Руда+» накладные расходы SAVEPOINT будут критичны, а в каких — несущественны?

Пример: безопасное деление

DO $$
DECLARE
    v_tons  NUMERIC := 150;
    v_hours NUMERIC := 0;
    v_rate  NUMERIC;
BEGIN
    v_rate := v_tons / v_hours;  -- Ошибка!
    RAISE NOTICE 'Производительность: % т/ч', v_rate;
EXCEPTION
    WHEN division_by_zero THEN
        RAISE NOTICE 'Ошибка: рабочие часы = 0, деление невозможно';
        RAISE NOTICE 'SQLSTATE: %', SQLSTATE;  -- 22012
        RAISE NOTICE 'Сообщение: %', SQLERRM;
END;
$$;
Встроенные переменные:
  • SQLSTATE — 5-символьный код ошибки (стандарт SQL)
  • SQLERRM — текстовое сообщение об ошибке

Обработка нескольких типов ошибок

DO $$
BEGIN
    -- Попытка вставить дубликат
    INSERT INTO dim_mine (mine_id, mine_name, location, depth_meters, status)
    VALUES (1, 'Тестовая шахта', 'Тест', 100, 'active');

    RAISE NOTICE 'Запись добавлена';
EXCEPTION
    WHEN unique_violation THEN
        RAISE NOTICE 'Дубликат: запись с таким ID уже существует';
    WHEN not_null_violation THEN
        RAISE NOTICE 'Обязательное поле не заполнено';
    WHEN foreign_key_violation THEN
        RAISE NOTICE 'Нарушение ссылочной целостности';
    WHEN OTHERS THEN
        RAISE NOTICE 'Неожиданная ошибка: [%] %', SQLSTATE, SQLERRM;
END;
$$;
Порядок важен! PostgreSQL проверяет условия сверху вниз. WHEN OTHERS всегда должен быть последним.

2. Предопределённые условия ошибок

Класс SQLSTATE Имя в PL/pgSQL Описание
02 02000 no_data_found Запрос не вернул строк
22 22012 division_by_zero Деление на ноль
22 22001 string_data_right_truncation Строка слишком длинная
23 23505 unique_violation Нарушение уникальности
23 23503 foreign_key_violation Нарушение внешнего ключа
23 23502 not_null_violation Нарушение NOT NULL
23 23514 check_violation Нарушение CHECK
40 40P01 deadlock_detected Обнаружена взаимная блокировка
P0 P0001 raise_exception Пользовательское исключение
Полный список: postgresql.org/docs/current/errcodes-appendix.html

SELECT INTO STRICT

DO $$
DECLARE
    v_mine_name VARCHAR;
BEGIN
    -- STRICT гарантирует ровно одну строку
    SELECT mine_name INTO STRICT v_mine_name
    FROM dim_mine
    WHERE mine_id = 999;  -- Не существует

    RAISE NOTICE 'Шахта: %', v_mine_name;
EXCEPTION
    WHEN no_data_found THEN
        RAISE NOTICE 'Шахта не найдена!';
    WHEN too_many_rows THEN
        RAISE NOTICE 'Запрос вернул более одной строки!';
END;
$$;
Без STRICT: SELECT INTO молча вернёт NULL, если строк нет, или первую строку, если их несколько. Со STRICT: генерируется исключение no_data_found или too_many_rows.

3. RAISE EXCEPTION — генерация ошибок

-- Простая ошибка
RAISE EXCEPTION 'Содержание Fe (%) вне диапазона', v_fe_content;

-- С параметрами USING
RAISE EXCEPTION 'Оборудование ID=% не найдено', v_equip_id
    USING HINT    = 'Проверьте справочник dim_equipment',
          DETAIL  = format('Запрос вернул 0 строк для ID=%s', v_equip_id),
          ERRCODE = 'P0002';

-- С пользовательским кодом ошибки
RAISE EXCEPTION 'Вес пробы должен быть положительным'
    USING ERRCODE = 'R0002';
Параметры USING:
  • ERRCODE — код ошибки SQLSTATE (5 символов)
  • HINT — подсказка для исправления
  • DETAIL — дополнительные детали
  • COLUMN, TABLE, SCHEMA — контекст

Пример: валидация данных «Руда+»

CREATE OR REPLACE FUNCTION validate_ore_sample(
    p_fe_content    NUMERIC,
    p_sample_weight NUMERIC,
    p_mine_id       INT
) RETURNS VARCHAR LANGUAGE plpgsql AS $$
DECLARE
    v_mine_exists BOOLEAN;
BEGIN
    -- Проверка 1: диапазон Fe
    IF p_fe_content < 0 OR p_fe_content > 100 THEN
        RAISE EXCEPTION 'Fe содержание (%) вне диапазона [0-100]', p_fe_content
            USING ERRCODE = 'R0001',
                  HINT = 'Проверьте данные лабораторного анализа';
    END IF;

    -- Проверка 2: вес пробы
    IF p_sample_weight <= 0 THEN
        RAISE EXCEPTION 'Вес пробы (%) должен быть положительным', p_sample_weight
            USING ERRCODE = 'R0002';
    END IF;

    -- Проверка 3: существование шахты
    SELECT EXISTS(SELECT 1 FROM dim_mine WHERE mine_id = p_mine_id)
    INTO v_mine_exists;
    IF NOT v_mine_exists THEN
        RAISE EXCEPTION 'Шахта ID=% не найдена', p_mine_id
            USING ERRCODE = 'R0003';
    END IF;

    RETURN 'Валидация пройдена';
END;
$$;

4. GET STACKED DIAGNOSTICS

Детальная информация об ошибке

В блоке EXCEPTION доступны подробные сведения через GET STACKED DIAGNOSTICS.

EXCEPTION WHEN OTHERS THEN
    DECLARE
        v_sqlstate   TEXT;
        v_message    TEXT;
        v_detail     TEXT;
        v_hint       TEXT;
        v_context    TEXT;
        v_schema     TEXT;
        v_table      TEXT;
        v_column     TEXT;
        v_constraint TEXT;
    BEGIN
        GET STACKED DIAGNOSTICS
            v_sqlstate   = RETURNED_SQLSTATE,
            v_message    = MESSAGE_TEXT,
            v_detail     = PG_EXCEPTION_DETAIL,
            v_hint       = PG_EXCEPTION_HINT,
            v_context    = PG_EXCEPTION_CONTEXT,
            v_schema     = SCHEMA_NAME,
            v_table      = TABLE_NAME,
            v_column     = COLUMN_NAME,
            v_constraint = CONSTRAINT_NAME;
    END;

Пример GET STACKED DIAGNOSTICS

DO $$
DECLARE
    v_state TEXT; v_msg TEXT; v_detail TEXT;
    v_hint TEXT; v_context TEXT; v_constraint TEXT;
BEGIN
    INSERT INTO dim_mine (mine_id, mine_name, location, depth_meters, status)
    VALUES (1, 'Дубль', 'Тест', 100, 'active');
EXCEPTION WHEN OTHERS THEN
    GET STACKED DIAGNOSTICS
        v_state      = RETURNED_SQLSTATE,
        v_msg        = MESSAGE_TEXT,
        v_detail     = PG_EXCEPTION_DETAIL,
        v_hint       = PG_EXCEPTION_HINT,
        v_context    = PG_EXCEPTION_CONTEXT,
        v_constraint = CONSTRAINT_NAME;

    RAISE NOTICE '=== Диагностика ===';
    RAISE NOTICE 'SQLSTATE:    %', v_state;
    RAISE NOTICE 'Сообщение:   %', v_msg;
    RAISE NOTICE 'Детали:      %', v_detail;
    RAISE NOTICE 'Подсказка:   %', v_hint;
    RAISE NOTICE 'Контекст:    %', v_context;
    RAISE NOTICE 'Ограничение: %', v_constraint;
END;
$$;
Преимущество перед SQLSTATE/SQLERRM: GET STACKED DIAGNOSTICS даёт имя таблицы, ограничения, полный контекст вызова — незаменимо для отладки.

5. Логирование ошибок в таблицу

-- Таблица для логов
CREATE TABLE error_log (
    log_id      SERIAL PRIMARY KEY,
    log_time    TIMESTAMP DEFAULT NOW(),
    severity    VARCHAR(20),     -- ERROR, WARNING, INFO
    source      VARCHAR(100),    -- Имя функции/процесса
    sqlstate    VARCHAR(5),
    message     TEXT,
    detail      TEXT,
    hint        TEXT,
    context     TEXT,
    username    VARCHAR(100) DEFAULT CURRENT_USER,
    parameters  JSONB            -- Входные параметры (для воспроизведения)
);

-- Функция логирования
CREATE FUNCTION log_error(
    p_severity VARCHAR, p_source VARCHAR,
    p_sqlstate VARCHAR DEFAULT NULL, p_message TEXT DEFAULT NULL,
    p_detail TEXT DEFAULT NULL, p_hint TEXT DEFAULT NULL,
    p_context TEXT DEFAULT NULL, p_parameters JSONB DEFAULT NULL
) RETURNS INT LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE v_id INT;
BEGIN
    INSERT INTO error_log (severity, source, sqlstate, message,
                           detail, hint, context, parameters)
    VALUES (p_severity, p_source, p_sqlstate, p_message,
            p_detail, p_hint, p_context, p_parameters)
    RETURNING log_id INTO v_id;
    RETURN v_id;
END;
$$;

Паттерн: пакетная обработка с логированием

DO $$
DECLARE
    rec RECORD;
    v_ok INT := 0; v_err INT := 0;
    v_state TEXT; v_msg TEXT; v_ctx TEXT;
BEGIN
    FOR rec IN SELECT * FROM dim_equipment LIMIT 10
    LOOP
        BEGIN  -- Вложенный блок для изоляции ошибок
            -- Обработка каждой единицы оборудования
            IF rec.equipment_id % 3 = 0 THEN
                RAISE EXCEPTION 'Тестовая ошибка для ID=%', rec.equipment_id;
            END IF;
            v_ok := v_ok + 1;

        EXCEPTION WHEN OTHERS THEN
            GET STACKED DIAGNOSTICS
                v_state = RETURNED_SQLSTATE,
                v_msg   = MESSAGE_TEXT,
                v_ctx   = PG_EXCEPTION_CONTEXT;

            -- Логируем и продолжаем
            PERFORM log_error('ERROR', 'batch_process',
                v_state, v_msg, NULL, NULL, v_ctx,
                jsonb_build_object('equipment_id', rec.equipment_id));
            v_err := v_err + 1;
        END;
    END LOOP;

    RAISE NOTICE 'Обработано: %, ошибок: %', v_ok, v_err;
END;
$$;
Ключевая идея: Вложенный BEGIN...EXCEPTION...END внутри цикла позволяет обработать ошибку для одного элемента и продолжить с остальными.

6. ASSERT — утверждения

DO $$
DECLARE
    v_mine_count INT;
BEGIN
    SELECT COUNT(*) INTO v_mine_count FROM dim_mine;

    -- Если условие FALSE — генерируется ASSERT_FAILURE
    ASSERT v_mine_count > 0,
        'Справочник шахт пуст! Выполните загрузку данных.';

    RAISE NOTICE 'OK: % шахт в справочнике', v_mine_count;
END;
$$;

ASSERT

  • Для разработки и тестирования
  • Можно отключить параметром plpgsql.check_asserts = off
  • Проверка инвариантов

IF + RAISE EXCEPTION

  • Для production
  • Нельзя отключить
  • Валидация входных данных

7. Best Practices

Рекомендации

  1. Не перехватывайте WHEN OTHERS без логирования — «проглоченные» ошибки — худший враг отладки
  2. Используйте вложенные блоки для изоляции ошибок в циклах
  3. Логируйте входные параметры (JSONB) — это позволит воспроизвести ошибку
  4. Различайте уровни серьёзности: ERROR, WARNING, INFO
  5. Используйте пользовательские коды ошибок (Rxxxx, Sxxxx) для бизнес-ошибок
  6. ASSERT — для разработки, IF + RAISE — для production

Антипаттерны

Плохо: «проглатывание» ошибок

EXCEPTION WHEN OTHERS THEN
    NULL;  -- Ошибка молча игнорируется!

Плохо: слишком широкий перехват

EXCEPTION WHEN OTHERS THEN
    -- Перехватываем ВСЕ, включая out_of_memory,
    -- disk_full, deadlock...

Хорошо: точечный перехват + логирование

EXCEPTION
    WHEN unique_violation THEN
        -- Обработка конкретной ситуации
        PERFORM log_error('WARNING', ...);
    WHEN foreign_key_violation THEN
        PERFORM log_error('ERROR', ...);
        RAISE;  -- Пробрасываем дальше

Квиз: проверка знаний

1. Что происходит с INSERT внутри BEGIN-блока, если возникает ошибка?

A) INSERT остаётся, ошибка только логируется
B) INSERT автоматически откатывается
C) INSERT сохраняется, но помечается как «ошибочный»
D) Зависит от уровня изоляции транзакции
Ответ: B — При входе в блок с EXCEPTION PostgreSQL создаёт неявный SAVEPOINT. При ошибке все изменения в этом блоке откатываются до этого SAVEPOINT.

Квиз (продолжение)

2. Какой SQLSTATE соответствует нарушению уникальности?

A) 22012
B) 23503
C) 23505
D) P0001
Ответ: C — 23505 (unique_violation). 22012 — деление на ноль, 23503 — FK violation, P0001 — raise_exception.

Квиз (продолжение)

3. Для чего нужен USING HINT в RAISE EXCEPTION?

A) Для подсказки оптимизатору запросов
B) Для рекомендации по исправлению ошибки
C) Для указания имени таблицы
D) Для задания уровня серьёзности
Ответ: B — HINT содержит рекомендацию по исправлению ошибки, которая отображается пользователю вместе с сообщением.

Квиз (продолжение)

4. Можно ли отключить ASSERT в production?

A) Нет, ASSERT нельзя отключить
B) Да, через SET plpgsql.check_asserts = off
C) Да, через DROP ASSERT
D) ASSERT не существует в PostgreSQL
Ответ: B — Параметр plpgsql.check_asserts позволяет отключить все ASSERT. Именно поэтому для production-валидации используйте IF + RAISE EXCEPTION.

Кейс для анализа

Сбой ночного ETL на «Руда+»

Инженер написал процедуру импорта, но она падает при первой же ошибке:

CREATE PROCEDURE import_sensor_data(p_data JSONB)
LANGUAGE plpgsql AS $$
DECLARE rec JSONB;
BEGIN
    FOR rec IN SELECT jsonb_array_elements(p_data) LOOP
        INSERT INTO fact_equipment_telemetry (...)
        VALUES ((rec->>'equipment_id')::INT, ...);
    END LOOP;
END;
$$;
Проблемы:
  1. Нет блока EXCEPTION — одна ошибка остановит всё
  2. Нет логирования — непонятно, что пошло не так
  3. Нет валидации — некорректные данные попадут в БД
  4. Нет статистики обработки — непонятно, сколько загружено
