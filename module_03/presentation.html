SQL и DAX: два мира, одна задача

SQL

  • Язык запросов к реляционным БД
  • Декларативный, основан на множествах
  • Работает с таблицами на сервере
  • SELECT → FROM → WHERE → GROUP BY
  • Стандарт ANSI, диалекты (PostgreSQL)

DAX

  • Язык формул для табличных моделей
  • Функциональный, основан на контексте
  • Работает с моделью данных в памяти
  • EVALUATE → функции → модификаторы
  • Microsoft (Power BI, SSAS, Excel)
Ключевое отличие: SQL оперирует таблицами и строками; DAX — контекстами вычислений и связями модели данных.

WHERE vs FILTER / CALCULATETABLE

SQL

Фильтр по равенству

Задача: оборудование шахты «Северная»

SQL

       manufacturer, model, status
FROM dim_equipment
WHERE mine_id = 1;

DAX — CALCULATETABLE

EVALUATE
CALCULATETABLE(
    SELECTCOLUMNS(
        dim_equipment,
        "Название",      dim_equipment[equipment_name],
        "Производитель", dim_equipment[manufacturer],
        "Модель",        dim_equipment[model],

FILTER vs CALCULATETABLE

FILTER (построчная)

EVALUATE
FILTER(
    dim_equipment,
    dim_equipment[mine_id] = 1
)
FILTER перебирает строки одну за одной.
Гибкая, но может быть медленной на больших таблицах.

CALCULATETABLE (контекстная)

EVALUATE
CALCULATETABLE(
    dim_equipment,
    dim_equipment[mine_id] = 1
)
CALCULATETABLE модифицирует контекст фильтра.
Оптимизируется движком лучше.

CALCULATE vs CALCULATETABLE

Два родственных механизма, одно отличие — тип результата

CALCULATE — скалярный результат

-- Одно число: общая добыча на шахте
CALCULATE(
    SUM(fact_production[tons_mined]),
    dim_mine[mine_name] = "Шахта ""Северная"""
)
-- Результат: 128 450
Первый аргумент — скалярное выражение:
SUM(...), COUNTROWS(...), AVERAGE(...)

CALCULATETABLE — таблица

-- Набор строк: оборудование шахты
EVALUATE
CALCULATETABLE(
    dim_equipment,
    dim_mine[mine_name] = "Шахта ""Северная"""
)
-- Результат: таблица (N строк)
Первый аргумент — табличное выражение:
SELECTCOLUMNS(...), имя таблицы, FILTER(...)
Общее: оба модифицируют контекст фильтра одинаковым образом. Фильтры во 2-м и далее аргументах работают идентично.

CALCULATE vs CALCULATETABLE: аналогия с SQL

SQL — скалярный подзапрос

-- Аналог CALCULATE
SELECT SUM(tons_mined)
FROM fact_production p
JOIN dim_mine m ON p.mine_id = m.mine_id
WHERE m.mine_name = 'Шахта "Северная"';
-- → одно число

SQL — табличный подзапрос

-- Аналог CALCULATETABLE
SELECT *
FROM dim_equipment e
JOIN dim_mine m ON e.mine_id = m.mine_id
WHERE m.mine_name = 'Шахта "Северная"';
-- → набор строк
ХарактеристикаCALCULATECALCULATETABLE
ВозвращаетСкаляр (число, строка, дата)Таблица
Первый аргументСкалярное выражениеТабличное выражение
Где используетсяМеры, вычисляемые столбцы, ADDCOLUMNSEVALUATE, внутри табличных функций
Аналог в SQLSELECT SUM(...) WHERE ...SELECT * WHERE ...

Вопрос для обсуждения

Почему внутри ADDCOLUMNS нужен CALCULATE, а не CALCULATETABLE?

Ответ: ADDCOLUMNS добавляет столбцы — каждый новый столбец должен содержать одно значение на строку (скаляр). Поэтому нужен CALCULATE, который возвращает скаляр, а не таблицу.

ADDCOLUMNS(
    SUMMARIZE(fact_production, dim_mine[mine_name]),
    "Всего тонн", CALCULATE(SUM(fact_production[tons_mined])),  -- скаляр ✓
    --          CALCULATETABLE(...)  -- таблица ✗ ошибка!
)

Сравнение и диапазоны

SQL

-- Больше 100 тонн
SELECT * FROM fact_production
WHERE tons_mined > 100;

-- Диапазон дат (BETWEEN)
SELECT * FROM fact_production
WHERE date_id BETWEEN 20240101
                   AND 20240131;

-- Список значений (IN)
SELECT * FROM dim_equipment
WHERE manufacturer
    IN ('Sandvik', 'Caterpillar');

-- Поиск по шаблону (LIKE)
SELECT * FROM dim_equipment
WHERE equipment_name LIKE 'ПДМ%';

DAX

// Больше 100 тонн
EVALUATE
FILTER(fact_production,
    fact_production[tons_mined] > 100)

// Диапазон (аналог BETWEEN)
EVALUATE
CALCULATETABLE(fact_production,
    fact_production[date_id] >= 20240101,
    fact_production[date_id] <= 20240131)

// Список значений (аналог IN)
EVALUATE
CALCULATETABLE(dim_equipment,
    dim_equipment[manufacturer]
        IN {"Sandvik", "Caterpillar"})

// Поиск (аналог LIKE)
EVALUATE
FILTER(dim_equipment,
    CONTAINSSTRING(
        dim_equipment[equipment_name],
        "ПДМ"))

Тема 4. Запросы к нескольким таблицам

JOIN vs RELATED / RELATEDTABLE

SQL: явные JOIN

Указываем связи в каждом запросе через ON

DAX: связи модели

Связи заданы один раз в модели.
RELATED() переходит по связи

Базовое соединение

Задача: оборудование с типом и шахтой

SQL

SELECT e.equipment_name,
       et.type_name AS equipment_type,
       m.mine_name
FROM dim_equipment e
INNER JOIN dim_equipment_type et
    ON e.equipment_type_id
       = et.equipment_type_id
INNER JOIN dim_mine m
    ON e.mine_id = m.mine_id;

DAX

EVALUATE
SELECTCOLUMNS(
    dim_equipment,
    "Название",
        dim_equipment[equipment_name],
    "Тип",
        RELATED(
          dim_equipment_type[type_name]),
    "Шахта",
        RELATED(dim_mine[mine_name])
)
RELATED() «переходит» по связи «многие к одному» (от equipment к mine) — аналог INNER JOIN

Зачем RELATED, если связи уже в модели?

Связи определяют как таблицы соединены. RELATED — механизм доступа к значению по этой связи.

SQL: связь в JOIN

SELECT e.equipment_name,
       t.type_name   -- доступ через алиас
FROM dim_equipment e
JOIN dim_equipment_type t
  ON e.equipment_type_id
     = t.equipment_type_id;
Связь задаётся в JOIN ... ON, доступ — через алиас таблицы

DAX: связь в модели

EVALUATE
SELECTCOLUMNS(
    dim_equipment,
    "Название",
        dim_equipment[equipment_name],
    "Тип",
        RELATED(             -- доступ по связи
          dim_equipment_type[type_name])
)
Связь уже в модели, RELATED — «дотянуться» по ней

Почему столбцы не подтягиваются автоматически?

1. Таблица — замкнутая сущность
Каждая таблица содержит только свои столбцы. dim_equipment[equipment_name] — движок точно знает, откуда взять значение. Без RELATED неясно, к какой таблице относится type_name.
2. Направление связи
RELATED работает только многие → один (от факта к справочнику) и гарантирует одно значение на строку.
В обратную сторону (один → многие) — RELATEDTABLE, возвращает таблицу.
3. Читаемость и контроль
Явный вызов RELATED(dim_mine[mine_name]) сразу показывает: данные из другой таблицы. В SQL это тоже явно — через JOIN + алиас.

Когда RELATED не нужен?

Некоторые функции DAX проходят по связям автоматически

RELATED нужен ✗

-- SELECTCOLUMNS: контекст строки
SELECTCOLUMNS(
    dim_equipment,
    "Тип",
      RELATED(
        dim_equipment_type[type_name])
)

-- ADDCOLUMNS: контекст строки
ADDCOLUMNS(
    dim_equipment,
    "Шахта",
      RELATED(dim_mine[mine_name])
)
Контекст строки — нужно явно указать переход по связи

Автоматически ✓

-- SUMMARIZECOLUMNS: группировка
SUMMARIZECOLUMNS(
    dim_mine[mine_name],
    "Тонн", SUM(
      fact_production[tons_mined])
)

-- CALCULATETABLE: фильтр
CALCULATETABLE(
    dim_equipment,
    dim_mine[mine_name] =
      "Шахта ""Северная"""
)
Контекст фильтра — движок сам «протекает» по связям

RELATED: сводная таблица

СитуацияRELATED нужен?Почему
SELECTCOLUMNS — вывести столбец связанной таблицыДаКонтекст строки, нужен явный переход
ADDCOLUMNS — добавить столбец связанной таблицыДаКонтекст строки
CALCULATETABLE — фильтр по связанной таблицеНетКонтекст фильтра, протекает по связям
SUMMARIZECOLUMNS — группировка по связанной таблицеНетАвтоматический проход по связям
CALCULATE — фильтр по связанной таблицеНетКонтекст фильтра
Правило: связи в модели — это «дороги» между таблицами. RELATED — «поехать по дороге и забрать значение». Функции контекста фильтра «ездят» сами, функции контекста строки — требуют явного вызова.

Вопрос для обсуждения

Почему в CALCULATETABLE фильтр по dim_mine[mine_name] работает без RELATED, а в SELECTCOLUMNS — нет?

Ответ: CALCULATETABLE модифицирует контекст фильтра — фильтр применяется к таблице dim_mine и автоматически «протекает» через связь на dim_equipment.

SELECTCOLUMNS работает в контексте строки — итерирует строки одной таблицы. Чтобы получить значение из другой таблицы, нужно явно «перейти» по связи через RELATED.

RELATEDTABLE — обратная навигация

RELATED идёт многие → один. А если нужно наоборот — один → многие?

SQL — подзапрос / LEFT JOIN

-- Сколько единиц оборудования на каждой шахте
SELECT m.mine_name,
       COUNT(e.equipment_id) AS cnt
FROM dim_mine m
LEFT JOIN dim_equipment e
    ON m.mine_id = e.mine_id
GROUP BY m.mine_name;

-- Шахты, где есть простои > 120 мин
SELECT DISTINCT m.mine_name
FROM dim_mine m
JOIN dim_equipment e
    ON m.mine_id = e.mine_id

Когда нет связи в модели — «ручное» соединение по общим столбцам

Когда использовать NATURALLEFTOUTERJOIN?

Типичные сценарии:
  • Соединение двух вычисленных таблиц (результатов SUMMARIZE)
  • Работа с таблицами, между которыми нет связи в модели
  • Объединение данных из разных источников в одном запросе
Важно:
  • Столбцы для соединения должны иметь одинаковые имена
  • Если связь есть в модели — используйте RELATED / RELATEDTABLE (эффективнее)
  • NATURALLEFTOUTERJOIN = LEFT JOIN (все строки из левой таблицы сохраняются)

GROUP BY vs SUMMARIZE / SUMMARIZECOLUMNS

Добыча по шахтам

SQL

SELECT m.mine_name,
       SUM(fp.tons_mined)  AS total_tons,
       AVG(fp.tons_mined)  AS avg_tons,
       COUNT(*)             AS total_shifts

SUMMARIZECOLUMNS — рекомендуемая функция DAX для группировки. Заменяет GROUP BY + JOIN.

Альтернатива: SUMMARIZE + ADDCOLUMNS

SQL

-- Тот же запрос
SELECT m.mine_name,
       SUM(fp.tons_mined) AS total_tons,
       AVG(fp.tons_mined) AS avg_tons,
       COUNT(*)            AS total_shifts
FROM fact_production fp
JOIN dim_mine m
    ON fp.mine_id = m.mine_id
GROUP BY m.mine_name;

DAX — SUMMARIZE + ADDCOLUMNS

EVALUATE
ADDCOLUMNS(
    SUMMARIZE(
        fact_production,
        dim_mine[mine_name]
    ),
    "Всего тонн",
        CALCULATE(
            SUM(fact_production[tons_mined])),
    "Среднее",
        CALCULATE(
            AVERAGE(
              fact_production[tons_mined])),
    "Кол-во смен",
        CALCULATE(
            COUNTROWS(fact_production))
)
SUMMARIZE создаёт группы, ADDCOLUMNS добавляет вычисления.
Внутри ADDCOLUMNS нужен CALCULATE для переключения контекста!

Зачем CALCULATE внутри ADDCOLUMNS?

В DAX есть два типа контекста вычислений:

Контекст фильтра

Ограничивает, какие строки «видны» для агрегатных функций.
Создаётся: CALCULATETABLE, SUMMARIZECOLUMNS

Контекст строки

Итерирует строки таблицы одну за одной.
Создаётся: ADDCOLUMNS, FILTER, SUMX

Проблема: ADDCOLUMNS создаёт контекст строки, а SUM / COUNTROWS нужен контекст фильтра. Без преобразования они «не понимают друг друга».

Без CALCULATE vs с CALCULATE

Без CALCULATE — неправильно

ADDCOLUMNS(
    SUMMARIZE(
        fact_production,
        dim_mine[mine_name]
    ),
    "Всего тонн",
        SUM(fact_production[tons_mined])
        -- SUM не видит текущую шахту!
        -- Вернёт ОБЩУЮ сумму
        -- для каждой строки
)
SUM работает в контексте строки — он не знает, что нужно считать только для текущей шахты. Результат: одинаковое число для всех строк.

С CALCULATE — правильно

ADDCOLUMNS(
    SUMMARIZE(
        fact_production,
        dim_mine[mine_name]
    ),
    "Всего тонн",
        CALCULATE(
            SUM(fact_production[tons_mined]))
        -- CALCULATE преобразует
        -- контекст строки → фильтра
        -- SUM видит только текущую шахту
)
CALCULATE выполняет context transition — текущая строка (mine_name = «Северная») становится фильтром. SUM считает только для этой шахты.

Context Transition: аналогия с SQL

Без CALCULATE = некоррелированный подзапрос

-- SUM считает по ВСЕЙ таблице
SELECT m.mine_name,
       (SELECT SUM(tons_mined)
        FROM fact_production) AS total
        -- одинаковое число для всех!
FROM dim_mine m;

С CALCULATE = коррелированный подзапрос

-- SUM привязан к текущей строке
SELECT m.mine_name,
       (SELECT SUM(fp.tons_mined)
        FROM fact_production fp
        WHERE fp.mine_id = m.mine_id
       ) AS total
       -- для каждой шахты своё!
FROM dim_mine m;
CALCULATE делает SUM «коррелированным» — привязанным к текущей строке. Это и есть context transition (переход контекста).

Почему в SUMMARIZECOLUMNS без CALCULATE?

SUMMARIZE + ADDCOLUMNS

-- Нужен CALCULATE!
ADDCOLUMNS(
    SUMMARIZE(
        fact_production,
        dim_mine[mine_name]
    ),
    "Тонн",
        CALCULATE(
            SUM(
              fact_production[tons_mined]))
)
Два шага: SUMMARIZE группирует, ADDCOLUMNS итерирует (контекст строки) → нужен CALCULATE

SUMMARIZECOLUMNS

-- Без CALCULATE!
SUMMARIZECOLUMNS(
    dim_mine[mine_name],
    "Тонн",
        SUM(
            fact_production[tons_mined])
)
«Всё в одном»: SUMMARIZECOLUMNS сама создаёт контекст фильтра для каждой группы → CALCULATE не нужен
Рекомендация: используйте SUMMARIZECOLUMNS как основной подход. SUMMARIZE + ADDCOLUMNS — когда нужна дополнительная гибкость (вложенные итерации, сложная логика).

Вопрос 1

Какая функция DAX является аналогом SELECT col1, col2 FROM table?

A) EVALUATE table
B) SELECTCOLUMNS(table, "col1", table[col1], ...)
C) FILTER(table, col1 && col2)
D) SUMMARIZE(table, col1, col2)
Ответ: B — SELECTCOLUMNS() выбирает и переименовывает столбцы. EVALUATE table возвращает все столбцы (аналог SELECT *).

Вопрос 2

Чем CALCULATETABLE отличается от FILTER в DAX?

A) Ничем, это синонимы
B) CALCULATETABLE модифицирует контекст фильтра; FILTER перебирает строки
C) FILTER быстрее, CALCULATETABLE медленнее
D) CALCULATETABLE работает только с мерами
Ответ: B — CALCULATETABLE изменяет контекст фильтра (аналогично WHERE), а FILTER выполняет построчную итерацию.

Вопрос 3

Как в DAX получить данные из связанной таблицы
(аналог INNER JOIN в SQL)?

A) JOIN(table1, table2)
B) MERGE(table1, table2)
C) RELATED(связанная_таблица[столбец])
D) LOOKUP(столбец, таблица)
Ответ: C — RELATED() переходит по связи «многие к одному» в модели данных.

Вопрос 4

Какой SQL-конструкции НЕТ прямого аналога в DAX?

A) GROUP BY
B) HAVING
C) ORDER BY
D) SELECT DISTINCT
Ответ: B — В DAX нет HAVING. Вместо него используют FILTER() поверх агрегированной таблицы.

Вопрос 5

Какая функция DAX рекомендуется для группировки данных?

A) GROUPBY()
B) SUMMARIZECOLUMNS()
C) AGGREGATE()
D) PIVOTTABLE()
Ответ: B — SUMMARIZECOLUMNS() — рекомендуемая функция DAX для группировки и агрегирования. GROUPBY() существует, но используется реже.

Вопрос 6

Как в SQL PostgreSQL ограничить результат 5 записями,
и как сделать то же в DAX?

A) SQL: TOP 5 / DAX: LIMIT(5)
B) SQL: LIMIT 5 / DAX: TOPN(5, ...)
C) SQL: LIMIT 5 / DAX: FIRST(5)
D) SQL: FETCH FIRST 5 / DAX: HEAD(5)
Ответ: B — PostgreSQL использует LIMIT, DAX — функцию TOPN() с указанием столбца сортировки.

Кейс: Анализ эффективности смен

Бизнес-контекст

Руководство «Руда+» хочет понять: какая смена (дневная или ночная) работает эффективнее?

Критерии эффективности:

  • Средний объём добычи за смену
  • Расход топлива на тонну руды
  • Производительность операторов (тонн/час)

Кейс: SQL-решение

-- Сравнение дневной и ночной смены
SELECT sh.shift_name,
       COUNT(DISTINCT fp.date_id)   AS work_days,
       COUNT(*)                     AS total_records,
       SUM(fp.tons_mined)           AS total_tons,
       ROUND(AVG(fp.tons_mined), 2) AS avg_tons,
       SUM(fp.fuel_consumed_l)      AS total_fuel,
       ROUND(
           SUM(fp.fuel_consumed_l) /
           NULLIF(SUM(fp.tons_mined), 0), 3
       ) AS fuel_per_ton
FROM fact_production fp
INNER JOIN dim_shift sh ON fp.shift_id = sh.shift_id
GROUP BY sh.shift_name;

Кейс: DAX-решение

EVALUATE
ADDCOLUMNS(
    SUMMARIZE(fact_production, dim_shift[shift_name]),
    "Рабочих дней",
        CALCULATE(DISTINCTCOUNT(fact_production[date_id])),
    "Всего записей",
        CALCULATE(COUNTROWS(fact_production)),
    "Всего тонн",
        CALCULATE(SUM(fact_production[tons_mined])),
    "Среднее на запись, т",
        CALCULATE(AVERAGE(fact_production[tons_mined])),
    "Всего топлива, л",
        CALCULATE(SUM(fact_production[fuel_consumed_l])),
    "Топливо на тонну, л",
        DIVIDE(
            CALCULATE(SUM(fact_production[fuel_consumed_l])),
            CALCULATE(SUM(fact_production[tons_mined])), 0
        )
)
