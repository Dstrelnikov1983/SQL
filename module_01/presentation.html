Что такое SQL?

SQL (Structured Query Language) — декларативный язык для управления данными в реляционных базах данных.

Ключевые свойства

  • Декларативный — описываем что, а не как
  • Стандартизированный — ANSI/ISO стандарт
  • Универсальный — работает во всех РСУБД
  • Множественный — оперирует наборами строк

Подъязыки SQL

  • DDL CREATE, ALTER, DROP
  • DML SELECT, INSERT, UPDATE, DELETE
  • DCL GRANT, REVOKE
  • TCL BEGIN, COMMIT, ROLLBACK
Спросить аудиторию: «Кто уже работал с SQL? С какими СУБД?» Декларативность — ключевое отличие от процедурных языков (Python, Java).

Один запрос — три диалекта

Получить 5 единиц оборудования с наибольшим временем работы:

PostgreSQL

SELECT equipment_name,
       operating_hours
FROM dim_equipment
ORDER BY operating_hours DESC
LIMIT 5;

T-SQL (MS SQL Server)

SELECT TOP 5
       equipment_name,
       operating_hours
FROM dim_equipment
ORDER BY operating_hours DESC;

PL/SQL (Oracle)

-- Oracle 12c+
SELECT equipment_name,
       operating_hours
FROM dim_equipment
ORDER BY operating_hours DESC
FETCH FIRST 5 ROWS ONLY;

-- Oracle (старый стиль)
SELECT * FROM (
  SELECT equipment_name,
         operating_hours
  FROM dim_equipment
  ORDER BY operating_hours DESC
) WHERE ROWNUM <= 5;
Наглядная демонстрация: задача одна, а синтаксис разный. FETCH FIRST — это стандарт SQL:2008, который поддерживают PostgreSQL и Oracle 12c+. LIMIT — расширение PostgreSQL/MySQL, ставшее де-факто стандартом.

Expression — Выражение

Expression (выражение) — конструкция, которая вычисляется и возвращает одно значение определённого типа данных. 
-- Литеральные выражения
42                              -- целое число
'Шахта Северная'                -- строка
TRUE                            -- логическое значение
CURRENT_DATE                    -- текущая дата

-- Арифметические выражения
tons_mined * 1.05               -- увеличение на 5%
operating_hours / 12.0 * 100    -- процент загрузки смены

-- Выражения с функциями
ROUND(fe_content, 1)            -- округление содержания Fe
COALESCE(end_time, NOW())       -- подстановка текущего времени

-- Условные выражения
CASE
    WHEN fe_content >= 60 THEN 'Высший сорт'
    WHEN fe_content >= 45 THEN 'Первый сорт'
    ELSE 'Требует обогащения'
END
Выражение — «атом» языка SQL. Может использоваться в SELECT, WHERE, HAVING, ORDER BY и т. д.

Clause — Предложение (секция)

Clause (предложение) — именованная секция SQL-инструкции, начинающаяся с ключевого слова. Каждое предложение выполняет свою функцию. 
SELECT                               -- clause: список столбцов
    m.mine_name,
    COUNT(e.equipment_id) AS cnt
FROM dim_mine m                      -- clause: источник данных
JOIN dim_equipment e                 -- clause: соединение
    ON m.mine_id = e.mine_id
WHERE m.status = 'active'            -- clause: фильтрация строк
GROUP BY m.mine_name                 -- clause: группировка
HAVING COUNT(e.equipment_id) > 5     -- clause: фильтрация групп
ORDER BY cnt DESC                    -- clause: сортировка
LIMIT 10;                            -- clause: ограничение
Порядок секций фиксирован: SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT

Statement и Command

Statement (инструкция)

Полная синтаксическая конструкция, которую СУБД может выполнить. Состоит из одного или нескольких предложений (clause).

-- Один statement (инструкция SELECT)
SELECT mine_name, max_depth_m
FROM dim_mine
WHERE status = 'active'
ORDER BY max_depth_m DESC;

Command (команда)

Тип действия, определяемый ключевым словом в начале инструкции. Определяет к какому подъязыку относится инструкция.

-- Команды DDL
CREATE TABLE ...
ALTER TABLE ...
DROP TABLE ...

-- Команды DML
SELECT ... FROM ...
INSERT INTO ... VALUES ...
UPDATE ... SET ...
DELETE FROM ...
Соотношение: Command определяет тип действия; Statement — конкретная инструкция с заполненными параметрами.

Batch — Пакет инструкций

Batch (пакет) — набор SQL-инструкций, отправляемых серверу как единое целое для последовательного выполнения. 
-- Пакет из нескольких инструкций
-- Каждая инструкция завершается точкой с запятой (;)

BEGIN;  -- Начало транзакции

INSERT INTO dim_mine (mine_name, mine_code, region, city, max_depth_m, status)
VALUES ('Шахта "Восточная"', 'MINE_E', 'Курская область',
        'г. Курск', 380.00, 'active');

INSERT INTO dim_shaft (mine_id, shaft_name, shaft_code, shaft_type, depth_m)
VALUES (currval('dim_mine_mine_id_seq'),
        'Ствол №1', 'E_MAIN', 'вертикальный', 380.00);

COMMIT;  -- Фиксация транзакции
В PostgreSQL: разделитель инструкций — точка с запятой (;). В отличие от MS SQL Server, в PostgreSQL нет оператора GO для разделения пакетов.

Иерархия элементов SQL

Batch — пакет инструкций
Statement — инструкция
Clause — предложение (SELECT, FROM, WHERE...)
Expression — выражение (значения, функции, операции) 
-- Batch (пакет)
  -- Statement (инструкция)
    SELECT mine_name,                -- Clause: SELECT
           max_depth_m * 3.281 AS depth_ft  -- Expression: арифметика
    FROM dim_mine                    -- Clause: FROM
    WHERE status = 'active';         -- Clause: WHERE, Expression: сравнение
  -- Statement (инструкция)
    SELECT COUNT(*) FROM dim_equipment;

Что можно комбинировать в пакете?

PostgreSQL — либеральный подход: в одном пакете можно свободно смешивать DDL, DML, DCL инструкции без ограничений. 
-- Всё это — один пакет, и всё работает
CREATE TABLE test_ore (id SERIAL, grade NUMERIC);
INSERT INTO test_ore (grade) VALUES (62.5), (58.3), (71.0);
SELECT AVG(grade) FROM test_ore;
ALTER TABLE test_ore ADD COLUMN sample_date DATE;
DROP TABLE test_ore;
Каждая следующая инструкция «видит» результат предыдущей — INSERT работает с только что созданной таблицей.

Ограничения транзакционного контекста

В PostgreSQL ограничения пакетов связаны не с комбинацией инструкций, а с транзакционным контекстом.

Следующие инструкции нельзя выполнять внутри блока BEGIN ... COMMIT:
ИнструкцияПричина
CREATE DATABASE / DROP DATABASE Операции с базой данных не транзакционны
VACUUM, CLUSTER Требуют эксклюзивного доступа вне транзакции
CREATE INDEX CONCURRENTLY Использует особую многофазную схему блокировок
 
-- Ошибка: CREATE DATABASE не может выполняться внутри транзакции
BEGIN;
CREATE DATABASE analytics_db;  -- ERROR!
COMMIT;

-- Без транзакции — работает
CREATE DATABASE analytics_db;  -- OK

Два мира PostgreSQL: SQL и PL/pgSQL

В PostgreSQL код делится на декларативные SQL-команды и процедурные блоки. Это две разные парадигмы.

Декларативный SQL

Описываем что нужно получить.
Оптимизатор решает как.

-- Нет переменных, нет циклов, нет IF
SELECT mine_name, max_depth_m
FROM dim_mine
WHERE status = 'active'
ORDER BY max_depth_m DESC;

UPDATE dim_equipment
SET status = 'decommissioned'
WHERE install_date < '2010-01-01';

CREATE INDEX idx_mine_status
ON dim_mine (status);

Процедурный PL/pgSQL

Описываем как выполнять —
шаг за шагом.

DO $$
DECLARE
    rec RECORD;
    total NUMERIC := 0;
BEGIN
    FOR rec IN
        SELECT max_depth_m
        FROM dim_mine
    LOOP
        total := total + rec.max_depth_m;
        IF rec.max_depth_m > 400 THEN
            RAISE NOTICE 'Глубокая!';
        END IF;
    END LOOP;
END;
$$;

Важно: процедурный блок не заменяет SQL — он его оборачивает. Внутри блока вы всё равно пишете обычные SQL-команды.

SQL внутри процедурного блока

DO $$
DECLARE
    v_mine_id INT;               -- процедурное: переменная
BEGIN
    -- декларативное: обычный SQL-запрос
    SELECT mine_id INTO v_mine_id
    FROM dim_mine
    WHERE mine_name = 'Северная';

    -- процедурное: ветвление
    IF v_mine_id IS NULL THEN
        -- декларативное: INSERT
        INSERT INTO dim_mine (mine_name, status)
        VALUES ('Северная', 'active')
        RETURNING mine_id INTO v_mine_id;

        RAISE NOTICE 'Создана шахта, id = %', v_mine_id;
    ELSE
        RAISE NOTICE 'Шахта уже существует, id = %', v_mine_id;
    END IF;
END;
$$;
Синим выделена процедурная обёртка (DECLARE, IF, RAISE), а SQL-команды (SELECT, INSERT) остаются декларативными.

Переменные в пакете: блоки PL/pgSQL

В PostgreSQL нет сессионных переменных на уровне SQL. Переменные существуют только внутри блоков DO и функций PL/pgSQL.

Блок DO — анонимный PL/pgSQL

DO $$
DECLARE
    avg_depth NUMERIC;
BEGIN
    SELECT AVG(max_depth_m)
      INTO avg_depth
      FROM dim_mine
     WHERE status = 'active';

    RAISE NOTICE
      'Средняя глубина: % м',
      avg_depth;
END;
$$;

Обходные пути

-- 1. Параметры сессии (GUC)
SET my.mine = 'Северная';
SELECT current_setting('my.mine');

-- 2. Временная таблица
CREATE TEMP TABLE vars AS
SELECT 42 AS mine_id;

SELECT m.mine_name
FROM dim_mine m
JOIN vars v ON m.mine_id = v.mine_id;

-- 3. CTE (внутри одного statement)
WITH params AS (SELECT 42 AS id)
SELECT * FROM dim_mine
WHERE mine_id = (SELECT id FROM params);

Анонимный блок DO

Выполняется один раз и не сохраняется на сервере. Удобен для разовых скриптов, миграций, административных задач.

DO $$
DECLARE
    mine_count INT;
    active_count INT;
BEGIN
    SELECT COUNT(*) INTO mine_count FROM dim_mine;
    SELECT COUNT(*) INTO active_count
      FROM dim_mine WHERE status = 'active';

    RAISE NOTICE 'Всего шахт: %, активных: %',
                  mine_count, active_count;

    IF active_count = 0 THEN
        RAISE WARNING 'Нет активных шахт!';
    END IF;
END;
$$;
Ограничение: блок DO не может возвращать результат в виде таблицы. Для вывода используется RAISE NOTICE или запись во временную таблицу.

Функция (FUNCTION)

Сохраняется на сервере, возвращает значение. Можно вызывать в SELECT, WHERE, JOIN — как часть SQL-выражения.

-- Скалярная функция: возвращает одно значение
CREATE FUNCTION get_mine_depth(p_mine_id INT)
RETURNS NUMERIC AS $$
BEGIN
    RETURN (SELECT max_depth_m FROM dim_mine
            WHERE mine_id = p_mine_id);
END;
$$ LANGUAGE plpgsql;

-- Табличная функция: возвращает набор строк
CREATE FUNCTION get_equipment_by_mine(p_mine_id INT)
RETURNS TABLE(equip_name TEXT, equip_type TEXT) AS $$
BEGIN
    RETURN QUERY
    SELECT e.equipment_name::TEXT, et.type_name::TEXT
    FROM dim_equipment e
    JOIN dim_equipment_type et
         ON e.equipment_type_id = et.equipment_type_id
    WHERE e.mine_id = p_mine_id;
END;
$$ LANGUAGE plpgsql;
-- Вызов скалярной
SELECT mine_name, get_mine_depth(mine_id) FROM dim_mine;
-- Вызов табличной
SELECT * FROM get_equipment_by_mine(1);

Процедура (PROCEDURE) — PostgreSQL 11+

Главное отличие от функции: процедура может управлять транзакциями (COMMIT / ROLLBACK внутри). Функция — нет. 
CREATE PROCEDURE deactivate_old_equipment(p_years INT)
LANGUAGE plpgsql AS $$
DECLARE
    affected INT;
BEGIN
    UPDATE dim_equipment
    SET status = 'decommissioned'
    WHERE install_date < CURRENT_DATE - (p_years * INTERVAL '1 year')
      AND status = 'active';

    GET DIAGNOSTICS affected = ROW_COUNT;
    RAISE NOTICE 'Списано единиц: %', affected;

    COMMIT;  -- фиксация внутри процедуры
END;
$$;

-- Вызов: через CALL, не через SELECT!
CALL deactivate_old_equipment(15);
Когда FUNCTION: нужно вернуть значение, использовать в SELECT/WHERE/JOIN
Когда PROCEDURE: нужно выполнить действие, управлять транзакциями, не нужен возврат в SQL

Триггерная функция

Вызывается автоматически при INSERT / UPDATE / DELETE. Имеет доступ к специальным переменным OLD и NEW.

-- 1. Создаём функцию с возвратом TRIGGER
CREATE FUNCTION audit_mine_changes()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO mine_audit_log(
        mine_id, action, old_status, new_status, changed_at
    ) VALUES (
        NEW.mine_id, TG_OP, OLD.status, NEW.status, NOW()
    );
    RETURN NEW;  -- обязательно для BEFORE-триггеров
END;
$$ LANGUAGE plpgsql;

-- 2. Привязываем к таблице
CREATE TRIGGER trg_mine_audit
    AFTER UPDATE ON dim_mine
    FOR EACH ROW
    WHEN (OLD.status IS DISTINCT FROM NEW.status)
    EXECUTE FUNCTION audit_mine_changes();

Когда срабатываетДо изменения данныхПосле изменения данных
Может изменить данныеДа (через RETURN NEW)Нет (данные уже записаны)
Типичное применениеВалидация, автозаполнениеАудит, логирование, уведомления

Вложенные блоки и обработка исключений

Блоки можно вкладывать. Каждый вложенный блок имеет свою область видимости и может перехватывать исключения, не прерывая родительский блок.

DO $$
DECLARE
    total INT;
BEGIN
    SELECT COUNT(*) INTO total FROM dim_mine;

    -- Вложенный блок: своя область видимости + свой EXCEPTION
    DECLARE
        active INT;
    BEGIN
        SELECT COUNT(*) INTO active
          FROM dim_mine WHERE status = 'active';
        RAISE NOTICE 'Активных: % из %', active, total;
    EXCEPTION
        WHEN OTHERS THEN
            RAISE NOTICE 'Ошибка во внутреннем блоке: %', SQLERRM;
    END;

    -- active здесь уже НЕ видна, total — видна
    RAISE NOTICE 'Всего шахт: %', total;
END;
$$;
Зачем вложенные блоки? Изолировать ошибку: если EXCEPTION перехватил ошибку во внутреннем блоке, внешний блок продолжает выполнение.

Метаданные: данные о данных

Метаданные — это информация о структуре базы данных: таблицы, столбцы, типы данных, ограничения, индексы, права доступа.

Зачем работать с метаданными?

  • Изучить структуру незнакомой БД
  • Найти нужные таблицы и столбцы
  • Проверить типы данных и ограничения
  • Документирование и аудит
  • Автоматическая генерация кода

Два способа доступа

  • information_schema — стандарт SQL (портируемый)
  • pg_catalog — системный каталог PostgreSQL (полный)
Для повседневных задач обычно хватает information_schema. Для специфики PostgreSQL нужен pg_catalog.

information_schema — стандарт SQL

Набор представлений (views), определённых стандартом SQL. Работает одинаково в PostgreSQL, MySQL, SQL Server.

-- Список всех таблиц в схеме public
SELECT table_name, table_type
FROM information_schema.tables
WHERE table_schema = 'public'
ORDER BY table_name;

-- Столбцы конкретной таблицы с типами данных
SELECT column_name,
       data_type,
       is_nullable,
       column_default
FROM information_schema.columns
WHERE table_schema = 'public'
  AND table_name = 'dim_mine'
ORDER BY ordinal_position;

-- Все ограничения (constraints) таблицы
SELECT constraint_name, constraint_type
FROM information_schema.table_constraints
WHERE table_name = 'fact_production';

pg_catalog — системный каталог PostgreSQL

Внутренние системные таблицы PostgreSQL. Содержат полную информацию, но не переносимы между СУБД. 
-- Список таблиц через pg_catalog
SELECT relname AS table_name,
       pg_size_pretty(pg_total_relation_size(oid)) AS total_size,
       reltuples::BIGINT AS approx_rows
FROM pg_catalog.pg_class
WHERE relkind = 'r'  -- 'r' = обычная таблица
  AND relnamespace = (SELECT oid FROM pg_namespace
                      WHERE nspname = 'public')
ORDER BY pg_total_relation_size(oid) DESC;

-- Описание столбцов с комментариями
SELECT a.attname AS column_name,
       pg_catalog.format_type(a.atttypid, a.atttypmod) AS data_type,
       col_description(a.attrelid, a.attnum) AS comment
FROM pg_catalog.pg_attribute a
JOIN pg_catalog.pg_class c ON a.attrelid = c.oid
WHERE c.relname = 'dim_mine'
  AND a.attnum > 0
  AND NOT a.attisdropped;

WHERE table_schema = 'public'
  AND table_name LIKE 'fact_%';

-- 2. Связи (FK) таблицы fact_production
SELECT kcu.column_name AS fk_column,
       ccu.table_name  AS referenced_table,
       ccu.column_name AS referenced_column
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
     ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.constraint_column_usage ccu
     ON tc.constraint_name = ccu.constraint_name
WHERE tc.table_name = 'fact_production'
  AND tc.constraint_type = 'FOREIGN KEY';

SELECT relname,
       pg_size_pretty(pg_total_relation_size(oid)) AS size
FROM pg_class
WHERE relkind = 'r'
  AND relnamespace = 'public'::regnamespace
ORDER BY pg_total_relation_size(oid) DESC;

  • Код «размазан» между БД и приложением
  • Отладка сложнее, чем в приложении
  • Миграция между средами (dev → staging → prod) требует дисциплины

Вариант 2: SQL в коде приложения

Примеры

# Python + psycopg2 — прямой SQL
cursor.execute("""
    SELECT mine_name, max_depth_m
    FROM dim_mine
    WHERE status = %s
""", ('active',))

# Python + SQLAlchemy (ORM)
session.query(Mine)\
    .filter(Mine.status == 'active')\
    .all()

// C# + Entity Framework (ORM)
var mines = context.Mines
    .Where(m => m.Status == "active")
    .ToList();

// Dapper — прямой SQL
var mines = connection.Query<Mine>(
    "SELECT * FROM dim_mine WHERE status = @s",
    new { s = "active" });

Плюсы

  • Код в одном месте — легко искать и рефакторить
  • Полноценное версионирование (Git)
  • Удобное тестирование (unit, integration)
  • ORM генерирует SQL автоматически

Минусы

  • SQL «размазан» по коду — трудно анализировать отдельно
  • ORM может генерировать неоптимальные запросы
  • Каждый вызов — сетевое обращение
  • Дублирование логики, если несколько приложений обращаются к одной БД

Рекомендуемый подход на практике

Комбинированный подход — используйте сильные стороны каждого варианта:
ЧтоГде хранитьПочему
Схема БД (CREATE TABLE, ALTER) Файлы миграций + Git Воспроизводимость, история
Сложная бизнес-логика Хранимые функции (тоже в Git!) Производительность, переиспользование
Запросы приложения Код приложения Связанность с бизнес-логикой
Аналитические отчёты SQL-файлы + Git или BI Совместная работа аналитиков
Тестовые данные (seeds) SQL/CSV файлы + Git Воспроизводимость окружения
Антипаттерн: хранить SQL только на сервере без резервного копирования определений. Один DROP FUNCTION — и код потерян навсегда.
Подчеркнуть студентам: даже если функция живёт на сервере, её определение ДОЛЖНО быть в Git. Сервер — это runtime, Git — источник истины.

Мини-квиз: проверим понимание

1. К какому подъязыку SQL относится команда SELECT?

Ответ: DML (Data Manipulation Language)

2. Как называется элемент SQL, который всегда возвращает одно значение?

Ответ: Expression (выражение)

3. В каком порядке идут секции в SELECT-запросе:
   WHERE, GROUP BY, FROM или FROM, WHERE, GROUP BY?

Ответ: FROM → WHERE → GROUP BY

4. Чем схема «снежинка» отличается от «звезды»?

Ответ: В «снежинке» таблицы измерений нормализованы (имеют подтаблицы), а в «звезде» — денормализованы.
