Тема 1. UNION и UNION ALL

Объединение результатов нескольких запросов в один набор

A B

UNION = A + B (все строки из обоих наборов)

UNION vs UNION ALL

UNION

  • Объединяет результаты
  • Удаляет дубликаты
  • Выполняет неявную сортировку
  • Медленнее (из-за DISTINCT)

UNION ALL

  • Объединяет результаты
  • Сохраняет все строки
  • Не сортирует
  • Быстрее (простая конкатенация)
Правило: если дубликаты допустимы или невозможны — всегда используйте UNION ALL!

Синтаксис UNION

-- UNION — без дубликатов
SELECT столбец1, столбец2 FROM таблица_A
UNION
SELECT столбец1, столбец2 FROM таблица_B;

-- UNION ALL — с дубликатами
SELECT столбец1, столбец2 FROM таблица_A
UNION ALL
SELECT столбец1, столбец2 FROM таблица_B;
Требования:
  • Одинаковое количество столбцов в каждом SELECT
  • Совместимые типы данных в соответствующих позициях
  • Имена столбцов берутся из первого SELECT

Пример «Руда+»: объединение данных шахт

Собрать единый отчёт по добыче из всех шахт

-- Добыча шахты «Северная» за январь 2025
SELECT e.equipment_name, SUM(p.tons_mined) AS total_tons,
       'Северная' AS mine_source
FROM fact_production p
JOIN dim_equipment e ON p.equipment_id = e.equipment_id
JOIN dim_mine m ON p.mine_id = m.mine_id
WHERE m.mine_code = 'MINE-N' AND p.date_id BETWEEN 20250101 AND 20250131
GROUP BY e.equipment_name

UNION ALL

-- Добыча шахты «Южная» за январь 2025
SELECT e.equipment_name, SUM(p.tons_mined), 'Южная'
FROM fact_production p
JOIN dim_equipment e ON p.equipment_id = e.equipment_id
JOIN dim_mine m ON p.mine_id = m.mine_id
WHERE m.mine_code = 'MINE-S' AND p.date_id BETWEEN 20250101 AND 20250131
GROUP BY e.equipment_name
ORDER BY total_tons DESC;

UNION ALL: сводные строки

Часто UNION ALL используется для добавления итоговых строк

-- Добыча по сменам + итого
SELECT s.shift_name, SUM(p.tons_mined) AS tons
FROM fact_production p
JOIN dim_shift s ON p.shift_id = s.shift_id
WHERE p.date_id = 20250115
GROUP BY s.shift_name

UNION ALL

SELECT 'ИТОГО', SUM(tons_mined)
FROM fact_production
WHERE date_id = 20250115
ORDER BY CASE WHEN shift_name = 'ИТОГО' THEN 1 ELSE 0 END,
         shift_name;
Совет: UNION ALL для итоговых строк — классический приём в отчётах

СитуацияОператорПочему
Два разных источника, дубликаты невозможныUNION ALLБыстрее, дубликатов нет

Тема 2. INTERSECT — пересечение

Возвращает только строки, присутствующие в обоих наборах

A B A ∩ B 
SELECT столбцы FROM таблица_A
INTERSECT
SELECT столбцы FROM таблица_B;

Пример: оборудование, работавшее в обе смены

Найти ПДМ, которые работали и в дневную, и в ночную смену

-- Оборудование дневной смены (shift_id = 1)
SELECT p.equipment_id, e.equipment_name
FROM fact_production p
JOIN dim_equipment e ON p.equipment_id = e.equipment_id
WHERE p.date_id = 20250115 AND p.shift_id = 1

INTERSECT

-- Оборудование ночной смены (shift_id = 2)
SELECT p.equipment_id, e.equipment_name
FROM fact_production p
JOIN dim_equipment e ON p.equipment_id = e.equipment_id
WHERE p.date_id = 20250115 AND p.shift_id = 2;
Бизнес-смысл: оборудование, работавшее в обе смены — кандидат на плановое ТО

INTERSECT vs INNER JOIN

INTERSECT

  • Сравнивает все столбцы
  • Возвращает уникальные строки
  • Простой синтаксис
  • Нет доступа к столбцам обоих наборов

INNER JOIN

  • Сравнивает указанные столбцы
  • Может возвращать дубликаты
  • Гибкие условия соединения
  • Доступ к столбцам обеих таблиц

Тема 3. EXCEPT — разность

Возвращает строки из первого набора, отсутствующие во втором

A \ B B 
SELECT столбцы FROM таблица_A
EXCEPT
SELECT столбцы FROM таблица_B;
В Oracle: аналог называется MINUS, в PostgreSQL используется EXCEPT

Пример: оборудование без простоев

Найти оборудование, которое работало, но не имело простоев за день

-- Всё оборудование, работавшее 15.01.2025
SELECT DISTINCT p.equipment_id
FROM fact_production p
WHERE p.date_id = 20250115

EXCEPT

-- Оборудование, имевшее простои в этот день
SELECT DISTINCT d.equipment_id
FROM fact_equipment_downtime d
WHERE d.date_id = 20250115;
Результат: ID оборудования, работавшего без единого простоя

Пример: пропущенные анализы качества

Какие смены за январь не имеют записей о качестве руды?

-- Все комбинации дата+смена, когда была добыча
SELECT DISTINCT p.date_id, p.shift_id
FROM fact_production p
WHERE p.date_id BETWEEN 20250101 AND 20250131

EXCEPT

-- Комбинации дата+смена, где был анализ качества
SELECT DISTINCT q.date_id, q.shift_id
FROM fact_ore_quality q
WHERE q.date_id BETWEEN 20250101 AND 20250131

ORDER BY date_id, shift_id;
Бизнес-смысл: контроль полноты данных MES — все смены должны иметь анализ качества

EXCEPT vs NOT EXISTS / NOT IN

-- EXCEPT (просто и читаемо)
SELECT DISTINCT equipment_id FROM fact_production WHERE date_id = 20250115
EXCEPT
SELECT DISTINCT equipment_id FROM fact_equipment_downtime WHERE date_id = 20250115;

-- NOT EXISTS (более гибко)
SELECT DISTINCT p.equipment_id
FROM fact_production p
WHERE p.date_id = 20250115
  AND NOT EXISTS (
      SELECT 1 FROM fact_equipment_downtime d
      WHERE d.equipment_id = p.equipment_id
        AND d.date_id = 20250115
  );

-- NOT IN (осторожно с NULL!)
SELECT DISTINCT equipment_id
FROM fact_production
WHERE date_id = 20250115
  AND equipment_id NOT IN (
      SELECT equipment_id FROM fact_equipment_downtime
      WHERE date_id = 20250115
  );

Внимание: NOT IN и NULL

Ловушка: если подзапрос в NOT IN возвращает хотя бы один NULL, результат всегда будет пустым! 
-- ОПАСНО: если в подзапросе есть NULL
SELECT 1 WHERE 5 NOT IN (1, 2, NULL);
-- Результат: пустой набор!

-- БЕЗОПАСНО: используйте EXCEPT или NOT EXISTS
SELECT DISTINCT equipment_id FROM fact_production
EXCEPT
SELECT equipment_id FROM fact_equipment_downtime;
Рекомендация: EXCEPT автоматически обрабатывает NULL корректно (сравнивая NULL = NULL)

Тема 4. Правила операторов набора

ПравилоОписание
Количество столбцовДолжно быть одинаковым во всех SELECT
Типы данныхДолжны быть совместимы (неявное приведение)
Имена столбцовБерутся из первого SELECT
ORDER BYДопустим только в самом конце (после последнего оператора)
ПриоритетINTERSECT имеет более высокий приоритет, чем UNION/EXCEPT
NULLДва NULL считаются равными при сравнении

ORDER BY — только в конце!

-- ПРАВИЛЬНО: ORDER BY в конце
SELECT equipment_name, tons_mined FROM view_mine_north
UNION ALL
SELECT equipment_name, tons_mined FROM view_mine_south
ORDER BY tons_mined DESC;

-- НЕПРАВИЛЬНО: ORDER BY посередине
SELECT equipment_name, tons_mined FROM view_mine_north
ORDER BY tons_mined DESC    -- ОШИБКА!
UNION ALL
SELECT equipment_name, tons_mined FROM view_mine_south;

-- Если нужна сортировка до объединения — используйте подзапрос:
SELECT * FROM (
    SELECT equipment_name, tons_mined FROM view_mine_north
    ORDER BY tons_mined DESC LIMIT 5
) top_north
UNION ALL
SELECT * FROM (
    SELECT equipment_name, tons_mined FROM view_mine_south
    ORDER BY tons_mined DESC LIMIT 5
) top_south;

Приоритет операторов

-- INTERSECT имеет приоритет выше, чем UNION
-- Следующий запрос:
SELECT * FROM A
UNION
SELECT * FROM B
INTERSECT
SELECT * FROM C;

-- Эквивалентен:
SELECT * FROM A
UNION
(SELECT * FROM B INTERSECT SELECT * FROM C);

-- Если нужен другой порядок — используйте скобки:
(SELECT * FROM A UNION SELECT * FROM B)
INTERSECT
SELECT * FROM C;
Совет: всегда используйте скобки для явного указания порядка выполнения

Оборудование шахты «Северная», которое работало в январе, но не имело внеплановых простоев

Тема 5. LATERAL JOIN

Аналог APPLY в SQL Server — для PostgreSQL

Основная таблица
→
Подзапрос / Функция
(ссылается на каждую строку)
LATERAL позволяет подзапросу ссылаться на столбцы из предыдущих элементов FROM. Подзапрос выполняется для каждой строки основной таблицы.

SQL Server APPLY vs PostgreSQL LATERAL

SQL ServerPostgreSQLОписание
CROSS APPLYCROSS JOIN LATERAL
или JOIN LATERAL ... ON TRUE		Только строки, где подзапрос вернул результат
OUTER APPLYLEFT JOIN LATERAL ... ON TRUEВсе строки + NULL, если подзапрос пуст
Важно: в PostgreSQL нет ключевых слов CROSS APPLY / OUTER APPLY — используйте LATERAL

Синтаксис LATERAL JOIN

-- CROSS JOIN LATERAL (аналог CROSS APPLY)
SELECT o.*, sub.*
FROM основная_таблица o
CROSS JOIN LATERAL (
    SELECT столбцы
    FROM подчинённая_таблица s
    WHERE s.ключ = o.ключ     -- ссылка на внешнюю таблицу!
    ORDER BY ... LIMIT N
) sub;

-- LEFT JOIN LATERAL (аналог OUTER APPLY)
SELECT o.*, sub.*
FROM основная_таблица o
LEFT JOIN LATERAL (
    SELECT столбцы
    FROM подчинённая_таблица s
    WHERE s.ключ = o.ключ
    ORDER BY ... LIMIT N
) sub ON TRUE;  -- условие ON TRUE обязательно!

Top-N per group с LATERAL

Классическая задача: для каждой шахты найти 3 самые продуктивные единицы оборудования

SELECT m.mine_name,
       top3.equipment_name,
       top3.total_tons
FROM dim_mine m
CROSS JOIN LATERAL (
    SELECT e.equipment_name,
           SUM(p.tons_mined) AS total_tons
    FROM fact_production p
    JOIN dim_equipment e ON p.equipment_id = e.equipment_id
    WHERE p.mine_id = m.mine_id                -- ссылка на внешний запрос
      AND p.date_id BETWEEN 20250101 AND 20250131
    GROUP BY e.equipment_name
    ORDER BY total_tons DESC
    LIMIT 3
) top3
ORDER BY m.mine_name, top3.total_tons DESC;
Преимущество: LATERAL + LIMIT эффективнее, чем ROW_NUMBER() OVER, особенно при больших объёмах

LATERAL vs оконные функции

-- LATERAL: Top-3 за шахту
SELECT m.mine_name, t.*
FROM dim_mine m
CROSS JOIN LATERAL (
    SELECT e.equipment_name,
           SUM(p.tons_mined) AS tons
    FROM fact_production p
    JOIN dim_equipment e
      ON p.equipment_id = e.equipment_id
    WHERE p.mine_id = m.mine_id
    GROUP BY e.equipment_name
    ORDER BY tons DESC
    LIMIT 3
) t;
-- ROW_NUMBER: Top-3 за шахту
SELECT mine_name, equipment_name, tons
FROM (
    SELECT m.mine_name,
           e.equipment_name,
           SUM(p.tons_mined) AS tons,
           ROW_NUMBER() OVER (
               PARTITION BY m.mine_id
               ORDER BY SUM(p.tons_mined) DESC
           ) AS rn
    FROM fact_production p
    JOIN dim_equipment e
      ON p.equipment_id = e.equipment_id
    JOIN dim_mine m
      ON p.mine_id = m.mine_id
    GROUP BY m.mine_id, m.mine_name,
             e.equipment_name
) ranked
WHERE rn <= 3;

LEFT JOIN LATERAL: последний простой

Для каждого оборудования — информация о последнем простое (даже если простоев не было)

SELECT e.equipment_name,
       et.type_name AS equipment_type,
       last_dt.start_time AS last_downtime_start,
       last_dt.duration_min,
       dr.reason_name
FROM dim_equipment e
JOIN dim_equipment_type et ON e.equipment_type_id = et.equipment_type_id
LEFT JOIN LATERAL (
    SELECT d.start_time, d.duration_min, d.reason_id
    FROM fact_equipment_downtime d
    WHERE d.equipment_id = e.equipment_id
    ORDER BY d.start_time DESC
    LIMIT 1
) last_dt ON TRUE
LEFT JOIN dim_downtime_reason dr ON last_dt.reason_id = dr.reason_id
WHERE e.status = 'active'
ORDER BY last_dt.start_time DESC NULLS LAST;

LATERAL с функциями

LATERAL также работает с табличными функциями (SRF)

-- Пример: разбить диапазон смены на часовые интервалы
SELECT s.shift_name, intervals.hour_start
FROM dim_shift s
CROSS JOIN LATERAL generate_series(
    s.start_time::timestamp,
    (s.end_time - interval '1 hour')::timestamp,
    interval '1 hour'
) AS intervals(hour_start);

-- Пример: unnest массива с сохранением связи
SELECT e.equipment_name, tag.value AS tag
FROM dim_equipment e
CROSS JOIN LATERAL unnest(
    ARRAY['GPS', 'video', 'telemetry']
) AS tag(value)
WHERE e.has_navigation = TRUE;

Вопрос 5

Что делает DAX-функция UNION()?

A) Работает как SQL UNION (удаляет дубликаты)
B) Работает как SQL UNION ALL (сохраняет дубликаты)
C) Объединяет столбцы таблиц
D) Создаёт связь между таблицами

Ответ: B — DAX UNION всегда сохраняет дубликаты (аналог UNION ALL)

Групповое задание

Контекст: Руководство «Руда+» планирует внеплановую проверку оборудования. Нужно найти:

  1. Группа 1: составьте запрос с UNION ALL для объединения данных о добыче и простоях в единый журнал событий (дата, оборудование, тип события, значение)
  2. Группа 2: с помощью EXCEPT найдите оборудование, которое числится в справочнике, но не имеет ни одной записи о добыче за январь 2025
  3. Группа 3: используя LATERAL JOIN, для каждой шахты получите 5 последних внеплановых простоев с причинами

Время: 7-10 минут. Обсуждение решений вместе.

Кейс для анализа

Ситуация: MES-система «Руда+» получает данные телеметрии от двух подсистем: навигационной системы (координаты) и системы датчиков (температура, вибрация). Данные хранятся в разных таблицах с одинаковой структурой. Аналитику нужно:

  1. Объединить данные в единый поток для мониторинга
  2. Найти моменты времени, когда навигация была, а датчик — нет (пропуски)
  3. Для каждого оборудования показать последние 3 записи из каждой подсистемы

Вопросы для обсуждения:

  • Какие операторы набора нужны для каждого пункта?
  • Где LATERAL будет эффективнее оконных функций?
  • Как обеспечить полноту данных в реальном времени?

Итоги модуля 12

  • UNION / UNION ALL — объединение наборов; UNION ALL быстрее, если дубликаты не важны
  • INTERSECT — пересечение; аналог INNER JOIN по всем столбцам
  • EXCEPT — разность; безопаснее, чем NOT IN при наличии NULL
  • ORDER BY — только в конце составного запроса
  • LATERAL JOIN — мощный инструмент для Top-N per group и коррелированных подзапросов
  • DAX: UNION (= UNION ALL), EXCEPT, INTERSECT, GENERATE (= LATERAL)
