Ключевой модуль курса: оконные функции — один из самых мощных инструментов аналитического SQL. Они позволяют выполнять вычисления над группами строк без потери детализации.

Часть 1: Основы оконных функций

Зачем нужны и как устроены

Проблема: агрегация vs. детализация

GROUP BY

-- Сколько добыто за смену?
SELECT shift_id,
       SUM(tons_mined) AS total
FROM fact_production
WHERE date_id = 20240115
GROUP BY shift_id;

Результат: 2 строки
Детали по оборудованию потеряны!

Оконная функция

-- Добыча каждой единицы + итог смены
SELECT equipment_id,
       tons_mined,
       SUM(tons_mined) OVER (
           PARTITION BY shift_id
       ) AS shift_total
FROM fact_production
WHERE date_id = 20240115;

Результат: все строки + агрегат по смене в каждой

Что такое оконная функция?

Оконная функция (window function) — функция, которая выполняет вычисление над набором строк (окном), связанных с текущей строкой. В отличие от GROUP BY, оконные функции не сворачивают строки.
  • Появились в стандарте SQL:2003
  • PostgreSQL поддерживает все оконные функции с версии 8.4 (2009)
  • Выполняются после WHERE, GROUP BY, HAVING
  • Но до ORDER BY и LIMIT

Порядок выполнения запроса


  1. FROM / JOIN         -- источники данных
  2. WHERE               -- фильтрация строк
  3. GROUP BY            -- группировка
  4. HAVING              -- фильтрация групп
  ─────────────────────────────────────────
  5. SELECT (оконные функции)  ◀ ЗДЕСЬ!
  ─────────────────────────────────────────
  6. DISTINCT
  7. ORDER BY
  8. LIMIT / OFFSET
Оконные функции нельзя использовать в WHERE и HAVING! Для фильтрации по результату оконной функции используйте подзапрос или CTE.

Синтаксис: предложение OVER()

функция(аргументы) OVER (
  [ PARTITION BY выражение [, ...] ]
  [ ORDER BY выражение [ ASC|DESC] [, ...] ]
  [ спецификация_рамки ]
)
  • PARTITION BY — определяет "окно" (аналог GROUP BY, но без схлопывания)
  • ORDER BY — порядок строк внутри окна
  • Рамка (frame) — уточняет, какие строки окна участвуют в вычислении

PARTITION BY: определение окон

-- Добыча по каждой единице оборудования с итогом по шахте
SELECT
    e.equipment_name,
    m.mine_name,
    fp.tons_mined,
    SUM(fp.tons_mined) OVER (PARTITION BY fp.mine_id)
        AS mine_total,
    ROUND(
        fp.tons_mined * 100.0
        / SUM(fp.tons_mined) OVER (PARTITION BY fp.mine_id),
        1
    ) AS pct_of_mine
FROM fact_production fp
JOIN dim_equipment e ON e.equipment_id = fp.equipment_id
JOIN dim_mine m ON m.mine_id = fp.mine_id
WHERE fp.date_id = 20240115 AND fp.shift_id = 1;
Каждая строка сохраняется, но обогащается агрегатом по своему разделу (шахте).

ORDER BY внутри окна

-- Нарастающий итог добычи за январь 2024 (шахта "Северная")
SELECT
    d.full_date,
    SUM(fp.tons_mined)
        AS daily_tons,
    SUM(SUM(fp.tons_mined)) OVER (
        ORDER BY d.full_date
    ) AS running_total
FROM fact_production fp
JOIN dim_date d ON d.date_id = fp.date_id
WHERE fp.mine_id = 1
  AND d.year = 2024 AND d.month = 1
GROUP BY d.full_date
ORDER BY d.full_date;
Когда указан ORDER BY без явной рамки, по умолчанию используется
RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — нарастающий итог!

Спецификация рамки окна (Frame)

{ ROWS | RANGE | GROUPS } BETWEEN
  начало_рамки AND конец_рамки

Границы рамки:
  UNBOUNDED PRECEDING — от начала раздела
  N PRECEDING — N строк/значений назад
  CURRENT ROW — текущая строка
  N FOLLOWING — N строк/значений вперёд
  UNBOUNDED FOLLOWING — до конца раздела

ROWS vs RANGE vs GROUPS

Тип Что считает Когда использовать
ROWS Физические строки Скользящее среднее по N строкам
RANGE Логический диапазон значений Диапазон по значению (дата, число).
Дубликаты считаются вместе
GROUPS Группы одинаковых значений Скользящее окно по группам peer-строк
(PostgreSQL 11+)
По умолчанию (если указан ORDER BY):
RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
Это означает нарастающий итог, но дубликаты обрабатываются как одна группа!

ROWS vs RANGE vs GROUPS: ключевые условия

Чтобы увидеть разницу всех трёх режимов, нужны данные с двумя свойствами:

1. Дубликаты в ORDER BY

Несколько строк с одинаковым значением сортировки

Пример: 2 единицы одного типа оборудования

ROWS отличается от RANGE и GROUPS

2. Пропуски в значениях

Не все значения между min и max присутствуют

Пример: типы оборудования 1, 3, 7 (нет 2, 4, 5, 6)

RANGE отличается от GROUPS

Важно: RANGE с N PRECEDING требует ровно одно числовое/дата поле в ORDER BY. С несколькими полями ORDER BY поддерживаются только UNBOUNDED и CURRENT ROW.

ROWS vs RANGE vs GROUPS: пример

Типы оборудования: 1 (ПДМ), 3 (Самосвал), 7 (Подъёмник) — дубликаты + пропуски

-- CTE для наглядности: 6 строк, 3 группы, пропуски между группами
WITH data AS (
    SELECT * FROM (VALUES
        (1, 'ПДМ-01',       100),
        (1, 'ПДМ-02',       120),
        (3, 'Самосвал-01',  200),
        (3, 'Самосвал-02',  180),
        (7, 'Подъёмник-01', 250),
        (7, 'Подъёмник-02', 230)
    ) AS t(type_id, equip_name, tons)
)
SELECT
    type_id, equip_name, tons,
    SUM(tons) OVER (ORDER BY type_id
        ROWS   BETWEEN 1 PRECEDING AND CURRENT ROW) AS sum_rows,
    SUM(tons) OVER (ORDER BY type_id
        RANGE  BETWEEN 1 PRECEDING AND CURRENT ROW) AS sum_range,
    SUM(tons) OVER (ORDER BY type_id
        GROUPS BETWEEN 1 PRECEDING AND CURRENT ROW) AS sum_groups
FROM data;

Одинаковая рамка 1 PRECEDING AND CURRENT ROW для всех трёх — но результаты разные.

Итог: когда что использовать

ROWS RANGE GROUPS
Единица отсчёта Физическая строка Значение ORDER BY Группа peer-строк
1 PRECEDING 1 строка назад Значение минус 1 1 группа назад
Дубликаты ORDER BY Обрабатывает по одной Все дубликаты вместе Все дубликаты вместе
Пропуски в значениях Не замечает Теряет данные через пропуск Перешагивает через пропуск
Когда использовать Скользящее среднее по N строкам Диапазон по значению (дата ± N дней) Окно по N логических групп
Практический совет: в производственных данных (типы оборудования, категории качества, горизонты шахты) значения часто идут с пропусками. GROUPS «перешагивает» пропуски и берёт соседнюю группу, а RANGE — нет.

Пример: скользящее среднее за 7 дней

-- Скользящее среднее добычи за 7 дней по шахте "Северная"
SELECT
    d.full_date,
    SUM(fp.tons_mined) AS daily_tons,
    ROUND(
        AVG(SUM(fp.tons_mined)) OVER (
            ORDER BY d.full_date
            ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
        ), 2
    ) AS moving_avg_7d,
    ROUND(
        AVG(SUM(fp.tons_mined)) OVER (
            ORDER BY d.full_date
            ROWS BETWEEN 14 PRECEDING AND CURRENT ROW
        ), 2
    ) AS moving_avg_15d
FROM fact_production fp
JOIN dim_date d ON d.date_id = fp.date_id
WHERE fp.mine_id = 1 AND d.year = 2024 AND d.month <= 3
GROUP BY d.full_date
ORDER BY d.full_date;

ROWS BETWEEN 6 PRECEDING AND CURRENT ROW — текущая строка + 6 предыдущих = окно из 7 строк

Рамка по умолчанию в PostgreSQL

С ORDER BY (указан)

RANGE BETWEEN
  UNBOUNDED PRECEDING
  AND CURRENT ROW

Нарастающий итог от начала раздела до текущей строки

Без ORDER BY

RANGE BETWEEN
  UNBOUNDED PRECEDING
  AND UNBOUNDED FOLLOWING

Весь раздел целиком — одно общее значение

⚠ Ловушка LAST_VALUE: без явной рамки LAST_VALUE() вернёт текущую строку, а не последнюю в разделе! 
-- ✅ Правильно — нужна явная рамка
LAST_VALUE(tons) OVER (
    PARTITION BY mine_id ORDER BY full_date
    ROWS BETWEEN UNBOUNDED PRECEDING
             AND UNBOUNDED FOLLOWING
)

Агрегатные функции + OVER()

Все стандартные агрегатные функции могут использоваться как оконные. Вместо сворачивания результата они добавляют агрегат к каждой строке.

Нарастающий итог добычи

-- Нарастающий итог добычи по месяцам с разбивкой по шахтам
SELECT
    d.year_month,
    m.mine_name,
    SUM(fp.tons_mined)                           AS month_tons,
    SUM(SUM(fp.tons_mined)) OVER (
        PARTITION BY fp.mine_id
        ORDER BY d.year_month
    )                                             AS cumulative_tons,
    COUNT(DISTINCT fp.equipment_id) OVER (
        PARTITION BY fp.mine_id
    )                                             AS total_equipment
FROM fact_production fp
JOIN dim_date d ON d.date_id = fp.date_id
JOIN dim_mine m ON m.mine_id = fp.mine_id
WHERE d.year = 2024
GROUP BY d.year_month, fp.mine_id, m.mine_name
ORDER BY m.mine_name, d.year_month;

Вложенные операторы в агрегатных оконных функциях

-- Доля добычи каждого оператора с накопительным процентом
SELECT
    o.last_name || ' ' || LEFT(o.first_name, 1) || '.' AS operator,
    SUM(fp.tons_mined)                                    AS total_tons,
    ROUND(
        SUM(fp.tons_mined) * 100.0
        / SUM(SUM(fp.tons_mined)) OVER (),
        1
    )                                                     AS pct_total,
    ROUND(
        SUM(SUM(fp.tons_mined)) OVER (
            ORDER BY SUM(fp.tons_mined) DESC
        ) * 100.0
        / SUM(SUM(fp.tons_mined)) OVER (),
        1
    )                                                     AS cumulative_pct
FROM fact_production fp
JOIN dim_operator o ON o.operator_id = fp.operator_id
WHERE fp.date_id BETWEEN 20240101 AND 20240131
GROUP BY o.operator_id, o.last_name, o.first_name
ORDER BY total_tons DESC;
Вложенный SUM(SUM(...)) — внутренний SUM является агрегатом GROUP BY, внешний — оконной функцией.

Скользящее среднее качества руды

-- Скользящее среднее содержания Fe за 5 проб
SELECT
    fq.sample_number,
    d.full_date,
    fq.fe_content,
    ROUND(
        AVG(fq.fe_content) OVER (
            PARTITION BY fq.mine_id
            ORDER BY fq.quality_id
            ROWS BETWEEN 4 PRECEDING AND CURRENT ROW
        ), 2
    ) AS moving_avg_fe,
    MIN(fq.fe_content) OVER (
        PARTITION BY fq.mine_id
        ORDER BY fq.quality_id
        ROWS BETWEEN 4 PRECEDING AND CURRENT ROW
    ) AS min_fe_window,
    MAX(fq.fe_content) OVER (
        PARTITION BY fq.mine_id
        ORDER BY fq.quality_id
        ROWS BETWEEN 4 PRECEDING AND CURRENT ROW
    ) AS max_fe_window
FROM fact_ore_quality fq
JOIN dim_date d ON d.date_id = fq.date_id
WHERE fq.mine_id = 1 AND d.year = 2024 AND d.month = 1
ORDER BY fq.quality_id
LIMIT 20;

Обнаружение аномалий в телеметрии

-- Обнаружение аномальных показаний температуры двигателя
-- (отклонение от скользящего среднего более чем на 2 сигмы)
SELECT
    d.full_date,
    t.hour_minute,
    ft.sensor_value AS temperature,
    ROUND(AVG(ft.sensor_value) OVER w, 2) AS avg_temp,
    ROUND(STDDEV(ft.sensor_value) OVER w, 2) AS stddev_temp,
    CASE
        WHEN ft.sensor_value >
             AVG(ft.sensor_value) OVER w
             + 2 * STDDEV(ft.sensor_value) OVER w
        THEN 'АНОМАЛИЯ: перегрев'
        WHEN ft.sensor_value <
             AVG(ft.sensor_value) OVER w
             - 2 * STDDEV(ft.sensor_value) OVER w
        THEN 'АНОМАЛИЯ: переохлаждение'
        ELSE 'Норма'
    END AS status
FROM fact_equipment_telemetry ft
JOIN dim_date d ON d.date_id = ft.date_id
JOIN dim_time t ON t.time_id = ft.time_id
WHERE ft.sensor_id = 1  -- датчик температуры ПДМ-001
  AND ft.date_id BETWEEN 20240101 AND 20240107
WINDOW w AS (ORDER BY ft.date_id, ft.time_id
             ROWS BETWEEN 10 PRECEDING AND 10 FOLLOWING)
ORDER BY d.full_date, t.hour_minute;
Используется именованное окно (WINDOW) для повторного использования определения окна.

Квиз: Агрегатные оконные функции

Вопрос 1: Что вернёт SUM(tons_mined) OVER () (пустые скобки)?

  1. Ошибку синтаксиса
  2. Сумму по всем строкам результата
  3. NULL
  4. Сумму по текущей строке

Ответ: B — пустое OVER() означает "окно = весь набор строк"

Квиз: Агрегатные оконные функции

Вопрос 2: В чём разница между результатами?

-- Запрос A:
SUM(tons) OVER (ORDER BY date_id)

-- Запрос B:
SUM(tons) OVER (ORDER BY date_id
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
  1. Результаты всегда одинаковы
  2. A использует RANGE (по умолчанию), B использует ROWS
  3. A не сортирует, B сортирует
  4. B работает только в PostgreSQL

Ответ: B — при одинаковых date_id запрос A (RANGE) присвоит одинаковую сумму всем дубликатам, B (ROWS) — последовательно

Часть 3: Функции ранжирования

ROW_NUMBER, RANK, DENSE_RANK, NTILE

Обзор функций ранжирования

Функция Описание При дубликатах Пример ряда
ROW_NUMBER() Уникальный порядковый номер Уникальные номера всегда 1, 2, 3, 4, 5
RANK() Ранг с пропусками Одинаковый ранг, потом пропуск 1, 2, 2, 4, 5
DENSE_RANK() Ранг без пропусков Одинаковый ранг, продолжение 1, 2, 2, 3, 4
NTILE(n) Разбиение на n групп Равномерное распределение 1, 1, 2, 2, 3
Все функции ранжирования требуют ORDER BY внутри OVER()!

Пример: наглядное сравнение

-- Ранжирование операторов по добыче за январь 2024
SELECT
    o.last_name,
    SUM(fp.tons_mined) AS total_tons,
    ROW_NUMBER() OVER (ORDER BY SUM(fp.tons_mined) DESC)  AS row_num,
    RANK()       OVER (ORDER BY SUM(fp.tons_mined) DESC)  AS rank_val,
    DENSE_RANK() OVER (ORDER BY SUM(fp.tons_mined) DESC)  AS dense_rank_val,
    NTILE(3)     OVER (ORDER BY SUM(fp.tons_mined) DESC)  AS tercile
FROM fact_production fp
JOIN dim_operator o ON o.operator_id = fp.operator_id
WHERE fp.date_id BETWEEN 20240101 AND 20240131
GROUP BY o.operator_id, o.last_name
ORDER BY total_tons DESC;

-- ТОП-3 самых продуктивных дня для каждой единицы оборудования
WITH daily AS (
    SELECT
        fp.equipment_id,
        e.equipment_name,
        d.full_date,
        SUM(fp.tons_mined) AS daily_tons,
        ROW_NUMBER() OVER (
            PARTITION BY fp.equipment_id
            ORDER BY SUM(fp.tons_mined) DESC
        ) AS rn
    FROM fact_production fp
    JOIN dim_equipment e ON e.equipment_id = fp.equipment_id
    JOIN dim_date d ON d.date_id = fp.date_id
    WHERE d.year = 2024
    GROUP BY fp.equipment_id, e.equipment_name, d.full_date
)
SELECT equipment_name, full_date, daily_tons
FROM daily
WHERE rn <= 3
ORDER BY equipment_name, rn;
Классический паттерн: CTE + ROW_NUMBER + фильтрация — один из самых частых приёмов в аналитике!

LAG: сравнение с предыдущим периодом

-- Сравнение добычи с предыдущим днём и расчёт динамики
SELECT
    d.full_date,
    SUM(fp.tons_mined) AS today_tons,
    LAG(SUM(fp.tons_mined), 1) OVER (ORDER BY d.full_date)
        AS yesterday_tons,
    LAG(SUM(fp.tons_mined), 7) OVER (ORDER BY d.full_date)
        AS week_ago_tons,
    ROUND(
        (SUM(fp.tons_mined)
         - LAG(SUM(fp.tons_mined), 1) OVER (ORDER BY d.full_date))
        * 100.0
        / NULLIF(LAG(SUM(fp.tons_mined), 1) OVER (ORDER BY d.full_date), 0),
        1
    ) AS day_over_day_pct
FROM fact_production fp
JOIN dim_date d ON d.date_id = fp.date_id
WHERE fp.mine_id = 1
  AND d.year = 2024 AND d.month = 1
GROUP BY d.full_date
ORDER BY d.full_date;

LAG с значением по умолчанию

-- LAG с третьим аргументом - значение по умолчанию
SELECT
    d.full_date,
    SUM(fp.tons_mined) AS today_tons,
    LAG(SUM(fp.tons_mined), 1, 0) OVER (ORDER BY d.full_date)
        AS yesterday_tons_or_zero
FROM fact_production fp

WHERE fp.mine_id = 1 AND d.year = 2024 AND d.month = 1

Синтаксис: LAG(выражение, смещение, значение_по_умолчанию)
смещение по умолчанию = 1. значение_по_умолчанию по умолчанию = NULL.

FIRST_VALUE и LAST_VALUE

-- Сравнение текущего дня с первым и лучшим днём месяца
SELECT
    d.full_date,
    SUM(fp.tons_mined) AS daily_tons,
    FIRST_VALUE(SUM(fp.tons_mined)) OVER (
        PARTITION BY d.month
        ORDER BY d.full_date
    ) AS first_day_tons,
    FIRST_VALUE(SUM(fp.tons_mined)) OVER (
        PARTITION BY d.month
        ORDER BY SUM(fp.tons_mined) DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS best_day_tons,
    LAST_VALUE(SUM(fp.tons_mined)) OVER (
        PARTITION BY d.month
        ORDER BY d.full_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS last_day_tons
FROM fact_production fp
JOIN dim_date d ON d.date_id = fp.date_id
WHERE fp.mine_id = 1 AND d.year = 2024 AND d.month <= 3
GROUP BY d.full_date, d.month
ORDER BY d.full_date;
Внимание! LAST_VALUE по умолчанию использует рамку до текущей строки. Для получения последнего значения в разделе нужно явно указать ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING!

NTH_VALUE

-- Второе по величине значение добычи в каждом месяце
SELECT DISTINCT
    d.year_month,
    FIRST_VALUE(daily_tons) OVER w AS best_day,
    NTH_VALUE(daily_tons, 2) OVER w AS second_best_day,
    NTH_VALUE(daily_tons, 3) OVER w AS third_best_day
FROM (
    SELECT
        fp.date_id,
        SUM(fp.tons_mined) AS daily_tons
    FROM fact_production fp
    WHERE fp.mine_id = 1
    GROUP BY fp.date_id
) sub
JOIN dim_date d ON d.date_id = sub.date_id
WHERE d.year = 2024
WINDOW w AS (
    PARTITION BY d.year_month
    ORDER BY daily_tons DESC
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
)
ORDER BY d.year_month;
NTH_VALUE(col, N) возвращает N-е значение в окне. Появилась в PostgreSQL 8.4.

PERCENT_RANK и CUME_DIST

-- Процентильное ранжирование операторов по эффективности
SELECT
    o.last_name,
    et.type_name,
    ROUND(AVG(fp.tons_mined / NULLIF(fp.operating_hours, 0)), 2)
        AS tons_per_hour,
    ROUND(PERCENT_RANK() OVER (
        PARTITION BY e.equipment_type_id
        ORDER BY AVG(fp.tons_mined / NULLIF(fp.operating_hours, 0))
    )::NUMERIC, 3) AS pct_rank,
    ROUND(CUME_DIST() OVER (
        PARTITION BY e.equipment_type_id
        ORDER BY AVG(fp.tons_mined / NULLIF(fp.operating_hours, 0))
    )::NUMERIC, 3) AS cume_dist
FROM fact_production fp
JOIN dim_operator o ON o.operator_id = fp.operator_id
JOIN dim_equipment e ON e.equipment_id = fp.equipment_id
JOIN dim_equipment_type et
    ON et.equipment_type_id = e.equipment_type_id
WHERE fp.date_id BETWEEN 20240101 AND 20240630
GROUP BY o.operator_id, o.last_name,
         e.equipment_type_id, et.type_name
ORDER BY et.type_name, pct_rank;
PERCENT_RANK = (rank - 1) / (total_rows - 1)
CUME_DIST = количество строк <= текущей / общее количество

Детекция выбросов (IQR-метод)

-- Обнаружение выбросов в качестве руды методом IQR
WITH quartiles AS (
    SELECT
        mine_id,
        PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY fe_content) AS q1,
        PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY fe_content) AS q3
    FROM fact_ore_quality
    WHERE date_id BETWEEN 20240101 AND 20240630
    GROUP BY mine_id
)
SELECT
    fq.sample_number,
    d.full_date,
    m.mine_name,
    fq.fe_content,
    ROUND(q.q1::NUMERIC, 2) AS q1,
    ROUND(q.q3::NUMERIC, 2) AS q3,
    ROUND((q.q3 - q.q1)::NUMERIC, 2) AS iqr,
    CASE
        WHEN fq.fe_content < q.q1 - 1.5 * (q.q3 - q.q1) THEN 'Выброс (низ)'
        WHEN fq.fe_content > q.q3 + 1.5 * (q.q3 - q.q1) THEN 'Выброс (верх)'
        ELSE 'Норма'
    END AS outlier_status
FROM fact_ore_quality fq
JOIN quartiles q ON q.mine_id = fq.mine_id
JOIN dim_date d ON d.date_id = fq.date_id
JOIN dim_mine m ON m.mine_id = fq.mine_id
WHERE fq.fe_content < q.q1 - 1.5 * (q.q3 - q.q1)
   OR fq.fe_content > q.q3 + 1.5 * (q.q3 - q.q1)
ORDER BY d.full_date;

Именованные окна (WINDOW)

-- Использование WINDOW для повторного определения окна
SELECT
    d.full_date,
    SUM(fp.tons_mined) AS daily_tons,
    AVG(SUM(fp.tons_mined)) OVER w7   AS avg_7d,
    MIN(SUM(fp.tons_mined)) OVER w7   AS min_7d,
    MAX(SUM(fp.tons_mined)) OVER w7   AS max_7d,
    AVG(SUM(fp.tons_mined)) OVER w30  AS avg_30d,
    SUM(SUM(fp.tons_mined)) OVER w_cum AS running_total
FROM fact_production fp
JOIN dim_date d ON d.date_id = fp.date_id
WHERE fp.mine_id = 1 AND d.year = 2024 AND d.quarter = 1
GROUP BY d.full_date
WINDOW
    w7    AS (ORDER BY d.full_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW),
    w30   AS (ORDER BY d.full_date ROWS BETWEEN 29 PRECEDING AND CURRENT ROW),
    w_cum AS (ORDER BY d.full_date)
ORDER BY d.full_date;
Предложение WINDOW избавляет от дублирования определений окон и делает запрос читабельнее. Размещается после HAVING, до ORDER BY.

Наследование окон

-- Базовое окно с наследованием
SELECT
    d.full_date,
    fp.equipment_id,
    SUM(fp.tons_mined) AS daily_tons,
    -- Окна наследуют от base_w, добавляя рамку
    SUM(SUM(fp.tons_mined)) OVER (
        base_w ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total,
    AVG(SUM(fp.tons_mined)) OVER (
        base_w ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS avg_7d
FROM fact_production fp

WHERE fp.mine_id = 1 AND d.year = 2024 AND d.month = 1
GROUP BY d.full_date, fp.equipment_id
WINDOW base_w AS (
    PARTITION BY fp.equipment_id
    ORDER BY d.full_date
)
ORDER BY fp.equipment_id, d.full_date;

Вопрос 5: Что вернёт LAST_VALUE(tons) OVER (ORDER BY date_id) для каждой строки?

  1. Последнее значение во всём разделе
  2. Значение текущей строки (т.к. рамка по умолчанию до текущей строки)

Вопрос 6: В чём преимущество оконных функций перед самосоединениями (self-join)?

  1. Оконные функции всегда быстрее
  2. Оконные функции проще читать и менее подвержены ошибкам
  3. Self-join не поддерживается в PostgreSQL
  4. Оконные функции возвращают меньше строк

Ответ: B — оконные функции более декларативны, понятнее и часто эффективнее, хотя не всегда быстрее.

Часть 6: SQL vs DAX

Оконные функции в двух мирах

Ключевое отличие: SQL vs DAX

SQL

  • Оконные функции — встроенная конструкция языка
  • OVER() с PARTITION BY, ORDER BY, рамкой
  • Декларативный подход
  • Одна строка кода — один расчёт

DAX

  • Нет прямых оконных функций
  • Аналоги через CALCULATE + FILTER
  • RANKX, EARLIER, контексты фильтра
  • OFFSET (DAX 2022+) — аналог LAG/LEAD
  • Более императивный подход
В DAX нет единого аналога SQL OVER(). Каждая задача решается своей комбинацией функций.

Нарастающий итог: SQL vs DAX

SQL

SELECT
  full_date,
  daily_tons,
  SUM(daily_tons) OVER (
    ORDER BY full_date
  ) AS running_total
FROM daily_production;

DAX

Running Total =
CALCULATE(
  SUM(fact_production[tons_mined]),
  FILTER(
    ALL(dim_date),
    dim_date[full_date]
      <= MAX(dim_date[full_date])
  )
)
В DAX нарастающий итог реализуется через CALCULATE + FILTER + ALL — мы "расширяем" контекст фильтра до всех дат, не превышающих текущую.

Ранжирование: SQL vs DAX

SQL

SELECT
  equipment_name,
  total_tons,
  RANK() OVER (
    ORDER BY total_tons DESC
  ) AS equip_rank
FROM equipment_summary;

DAX

Equipment Rank =
RANKX(
  ALL(dim_equipment),
  CALCULATE(
    SUM(fact_production[tons_mined])
  ),
  ,
  DESC,
  Dense
)
RANKX принимает таблицу, выражение для ранжирования, порядок и тип (Skip/Dense). Skip = RANK(), Dense = DENSE_RANK().

Сравнение с предыдущим периодом: SQL vs DAX

SQL (LAG)

SELECT
  full_date,
  daily_tons,
  LAG(daily_tons) OVER (
    ORDER BY full_date
  ) AS prev_day_tons
FROM daily_production;

DAX (OFFSET)

-- DAX 2022+ (Power BI)
Prev Day Tons =
CALCULATE(
  SUM(fact_production[tons_mined]),
  OFFSET(
    -1,
    ALLSELECTED(dim_date[full_date]),
    ORDERBY(dim_date[full_date], ASC)
  )
)

-- Классический DAX:
Prev Day Tons Classic =
CALCULATE(
  SUM(fact_production[tons_mined]),
  PREVIOUSDAY(dim_date[full_date])
)

Скользящее среднее: SQL vs DAX

SQL

AVG(daily_tons) OVER (
  ORDER BY full_date
  ROWS BETWEEN 6 PRECEDING
    AND CURRENT ROW
) AS ma_7d

DAX

MA 7d =
VAR CurrentDate =
  MAX(dim_date[full_date])
RETURN
AVERAGEX(
  FILTER(
    ALL(dim_date),
    dim_date[full_date]
      >= CurrentDate - 6
    && dim_date[full_date]

      fact_production[tons_mined]
    )
  )
)

Работа в группах (15 минут)

Задание: Разделитесь на 3 группы. Каждая группа разрабатывает SQL-запрос для своего сценария MES "Руда+":

  1. Группа "Добыча": Построить ежедневный отчёт добычи с нарастающим итогом, скользящим средним за 7 дней, рангом дня в месяце и процентом от плана (план = 500 т/день)
  2. Группа "Качество": Для каждой пробы руды рассчитать отклонение от скользящего среднего Fe (5 проб), флаг аномалии (>2 стандартных отклонений), ранг пробы по качеству за день
  3. Группа "Оборудование": Для каждого простоя определить время до следующего простоя (LAG/LEAD), сравнить длительность с предыдущим простоем того же типа, рассчитать нарастающее время простоя за месяц

Кейс: Предиктивное обслуживание ПДМ

Ситуация: На шахте "Северная" ПДМ-001 (Sandvik LH514) за последние 3 месяца трижды выходил на аварийный ремонт. Начальник участка просит аналитиков разработать систему раннего предупреждения.

Задача: Используя оконные функции, разработайте запрос, который:

  1. Рассчитывает скользящее среднее температуры и вибрации за последние 2 часа
  2. Определяет скорость роста температуры (LAG)
  3. Находит отклонения от нормы (PERCENT_RANK)
  4. Присваивает уровень риска: Норма / Внимание / Опасность

Решение кейса: SQL-запрос

WITH telemetry_enriched AS (
    SELECT
        ft.telemetry_id,
        d.full_date, t.hour_minute,
        st.type_code,
        ft.sensor_value,
        -- Скользящее среднее за 2 часа (8 точек по 15 мин)
        AVG(ft.sensor_value) OVER w8 AS avg_2h,
        STDDEV(ft.sensor_value) OVER w8 AS stddev_2h,
        -- Скорость изменения (разница с предыдущим показанием)
        ft.sensor_value - LAG(ft.sensor_value) OVER w_seq AS delta,
        -- Процентильный ранг
        PERCENT_RANK() OVER (
            PARTITION BY ft.sensor_id
            ORDER BY ft.sensor_value
        ) AS pct_rank
    FROM fact_equipment_telemetry ft
    JOIN dim_date d ON d.date_id = ft.date_id
    JOIN dim_time t ON t.time_id = ft.time_id
    JOIN dim_sensor s ON s.sensor_id = ft.sensor_id
    JOIN dim_sensor_type st ON st.sensor_type_id = s.sensor_type_id
    WHERE ft.equipment_id = 1
      AND st.type_code IN ('TEMP_ENGINE', 'VIBRATION')
      AND ft.date_id BETWEEN 20240101 AND 20240107
    WINDOW
        w8 AS (PARTITION BY ft.sensor_id
               ORDER BY ft.date_id, ft.time_id
               ROWS BETWEEN 7 PRECEDING AND CURRENT ROW),
        w_seq AS (PARTITION BY ft.sensor_id
                  ORDER BY ft.date_id, ft.time_id)
)
SELECT *, CASE
    WHEN pct_rank > 0.95 OR delta > avg_2h * 0.1
        THEN 'ОПАСНОСТЬ'
    WHEN pct_rank > 0.85 OR delta > avg_2h * 0.05
        THEN 'ВНИМАНИЕ'
    ELSE 'Норма'
END AS risk_level
FROM telemetry_enriched
WHERE pct_rank > 0.80
ORDER BY full_date, hour_minute;

Типичные ошибки

Ошибка 1

Забыть рамку для LAST_VALUE

-- Неправильно:
LAST_VALUE(x) OVER (
  ORDER BY date
)
-- Вернёт текущую строку!

-- Правильно:
LAST_VALUE(x) OVER (
  ORDER BY date
  ROWS BETWEEN
    UNBOUNDED PRECEDING
    AND UNBOUNDED FOLLOWING
)

Ошибка 2

Оконная функция в WHERE

-- Неправильно:
SELECT *
FROM t
WHERE ROW_NUMBER()
  OVER(ORDER BY x) <= 5;

-- Правильно:
SELECT * FROM (
  SELECT *,
    ROW_NUMBER()
      OVER(ORDER BY x) AS rn
  FROM t
) sub
WHERE rn <= 5;

Ошибка 3

RANGE vs ROWS при дубликатах

-- Если date имеет дубликаты:
SUM(x) OVER (
  ORDER BY date
)
-- RANGE: дубликаты
-- получат одно значение

SUM(x) OVER (
  ORDER BY date
  ROWS UNBOUNDED PRECEDING
)
-- ROWS: каждая строка
-- по отдельности

Лучшие практики

  • Используйте WINDOW для повторяющихся определений окон — улучшает читаемость
  • Явно указывайте рамку при использовании ORDER BY — не полагайтесь на значение по умолчанию
  • Индексируйте столбцы в PARTITION BY и ORDER BY для больших таблиц
  • Используйте CTE для фильтрации по результатам оконных функций
  • ROWS предпочтительнее RANGE для числовых вычислений — более предсказуемое поведение
  • Комбинируйте несколько оконных функций в одном запросе — PostgreSQL оптимизирует выполнение
