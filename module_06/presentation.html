Часть 1. Математические функции

Функции для числовых вычислений в аналитических запросах

Зачем? Расчёт KPI, нормализация данных, контроль допусков, округление результатов анализов качества руды.

ROUND, CEIL, FLOOR

ФункцияОписаниеПримерРезультат
ROUND(x, n)Округление до n знаковROUND(56.789, 1)56.8
CEIL(x)Округление вверхCEIL(56.1)57
FLOOR(x)Округление внизFLOOR(56.9)56
TRUNC(x, n)Отсечение до n знаковTRUNC(56.789, 1)56.7
Разница ROUND vs TRUNC: ROUND(2.55, 1) = 2.6, но TRUNC(2.55, 1) = 2.5

Пример: Округление содержания Fe

SQL (PostgreSQL)

SELECT
    sample_number,
    fe_content,
    ROUND(fe_content, 1)  AS fe_round_1,
    CEIL(fe_content)      AS fe_ceil,
    FLOOR(fe_content)     AS fe_floor,
    TRUNC(fe_content, 0)  AS fe_trunc
FROM fact_ore_quality
WHERE date_id = 20240315
ORDER BY fe_content DESC
LIMIT 5;

DAX

EVALUATE
TOPN(5,
  ADDCOLUMNS(
    FILTER(fact_ore_quality,
      fact_ore_quality[date_id] = 20240315),
    "Fe округл.",
      ROUND(fact_ore_quality[fe_content], 1),
    "Fe вверх",
      CEILING(fact_ore_quality[fe_content], 1),
    "Fe вниз",
      FLOOR(fact_ore_quality[fe_content], 1)

DAX: CEILING и FLOOR в DAX принимают второй параметр — значимость (significance). Подробнее на следующем слайде →

Параметр significance в CEILING / FLOOR

Параметр significance задаёт кратность округления — шаг сетки, к которой «прилипает» число.

Выражение DAXРезультатПояснение
CEILING(2.3, 1)3Вверх до целого (кратно 1)
CEILING(2.3, 0.5)2.5Вверх до ближайшего 0.5
CEILING(2.3, 5)5Вверх до кратного 5
FLOOR(7.8, 1)7Вниз до целого
FLOOR(7.8, 0.25)7.75Вниз до ближайшего 0.25
FLOOR(7.8, 5)5Вниз до кратного 5
В PostgreSQL у CEIL() / FLOOR() параметра significance нет — они всегда округляют до целого.
Аналог с произвольным шагом: 
-- Аналог CEILING(x, 0.5)
SELECT CEIL(x / 0.5) * 0.5;

-- Аналог FLOOR(x, 5)
SELECT FLOOR(x / 5) * 5;

Пример: Отклонение содержания Fe от нормы

-- Абсолютное отклонение от целевого содержания Fe (60%)
SELECT
    sample_number,
    fe_content,
    ABS(fe_content - 60.0)        AS abs_deviation,
    POWER(fe_content - 60.0, 2)   AS squared_deviation,
    SQRT(POWER(fe_content - 60.0, 2)) AS rmse_component
FROM fact_ore_quality
WHERE date_id BETWEEN 20240301 AND 20240331
ORDER BY abs_deviation DESC
LIMIT 10;
Бизнес-смысл: Целевое содержание Fe = 60%. Отклонение влияет на сортность руды и стоимость продукции.

Пример: Сводка добычи за март 2024

SQL (PostgreSQL)

SELECT
    COUNT(*)             AS total_records,
    COUNT(DISTINCT equipment_id)
                         AS unique_equipment,
    SUM(tons_mined)      AS total_tons,
    AVG(tons_mined)      AS avg_tons,
    MIN(tons_mined)      AS min_tons,
    MAX(tons_mined)      AS max_tons,
    ROUND(AVG(operating_hours), 2)
                         AS avg_hours
FROM fact_production
WHERE date_id BETWEEN 20240301
                   AND 20240331;

EVALUATE

STRING_AGG — конкатенация строк

SQL (PostgreSQL)

-- Список оборудования каждой шахты одной строкой
SELECT
    m.mine_name,
    STRING_AGG(
        e.equipment_name,
        ', '
        ORDER BY e.equipment_name
    ) AS equipment_list,
    COUNT(*) AS total
FROM dim_equipment e
JOIN dim_mine m ON e.mine_id = m.mine_id
WHERE e.status = 'active'
GROUP BY m.mine_name;

DAX

// CONCATENATEX — аналог STRING_AGG
EVALUATE
ADDCOLUMNS(
  VALUES(dim_mine[mine_name]),
  "Оборудование",
    CONCATENATEX(
      RELATEDTABLE(dim_equipment),
      dim_equipment[equipment_name],
      ", ",
      dim_equipment[equipment_name], ASC
    ),
  "Всего",
    COUNTROWS(
      RELATEDTABLE(dim_equipment))
)
STRING_AGG (SQL) = CONCATENATEX (DAX). Оба поддерживают сортировку.

ARRAY_AGG — специфика PostgreSQL

-- Массив уникальных дат добычи по каждому оператору
SELECT
    o.last_name || ' ' || o.first_name AS operator_name,
    ARRAY_AGG(DISTINCT fp.date_id ORDER BY fp.date_id) AS work_dates,
    ARRAY_LENGTH(
        ARRAY_AGG(DISTINCT fp.date_id), 1
    ) AS days_worked
FROM fact_production fp
JOIN dim_operator o ON fp.operator_id = o.operator_id
WHERE fp.date_id BETWEEN 20240301 AND 20240307
GROUP BY o.last_name, o.first_name
ORDER BY days_worked DESC;
Только PostgreSQL! ARRAY_AGG возвращает массив. В DAX прямого аналога нет — используйте CONCATENATEX или промежуточные таблицы.

Агрегаты с FILTER (PostgreSQL)

-- Условная агрегация: добыча по сменам в одном запросе
SELECT
    d.full_date,
    COUNT(*) FILTER (WHERE fp.shift_id = 1) AS records_shift_1,
    COUNT(*) FILTER (WHERE fp.shift_id = 2) AS records_shift_2,
    SUM(fp.tons_mined) FILTER (WHERE fp.shift_id = 1) AS tons_shift_1,
    SUM(fp.tons_mined) FILTER (WHERE fp.shift_id = 2) AS tons_shift_2,
    ROUND(
        AVG(fp.tons_mined) FILTER (WHERE fp.shift_id = 1), 2
    ) AS avg_tons_shift_1
FROM fact_production fp
JOIN dim_date d ON fp.date_id = d.date_id
WHERE d.year = 2024 AND d.month = 3
GROUP BY d.full_date
ORDER BY d.full_date;
FILTER-клаузула в агрегатах PostgreSQL — мощная альтернатива CASE WHEN внутри SUM/COUNT.

Мини-квиз: Агрегатные функции

Вопрос 1: Какой результат вернёт запрос?

SELECT COUNT(sio2_content), COUNT(*), COUNT(DISTINCT ore_grade_id)
FROM fact_ore_quality
-- Допустим: 100 строк, в 15 строках sio2_content = NULL, 4 уникальных сорта
A) 100, 100, 4
B) 85, 100, 4
C) 85, 85, 4
D) 100, 85, 4
Ответ: B) COUNT(col) пропускает NULL, COUNT(*) считает все строки, COUNT(DISTINCT col) — уникальные не-NULL.

Мини-квиз: Агрегатные функции (2)

Вопрос 2: Чем отличаются эти два запроса?

-- Запрос A
SELECT AVG(tons_mined) FROM fact_production;

-- Запрос B
SELECT SUM(tons_mined) / COUNT(*) FROM fact_production;
A) Результат всегда одинаковый
B) Различие при наличии NULL в tons_mined
C) Различие в типе результата
Ответ: B) AVG пропускает NULL (делит на кол-во не-NULL). SUM/COUNT(*) делит сумму на все строки, включая NULL. Если tons_mined NOT NULL — результат одинаков.

Часть 3. Функции преобразования типов

Явное и неявное приведение типов данных

Зачем? Объединение данных из разных источников, форматирование отчётов, корректные вычисления с разнотипными данными.

CAST и оператор :: (PostgreSQL)

-- Стандартный синтаксис CAST
SELECT CAST(date_id AS VARCHAR)     AS date_str,
       CAST('123.45' AS NUMERIC)    AS num_value,
       CAST(tons_mined AS INTEGER)  AS tons_int
FROM fact_production
LIMIT 3;

-- Краткий синтаксис PostgreSQL ::
SELECT date_id::VARCHAR              AS date_str,
       '123.45'::NUMERIC             AS num_value,
       tons_mined::INTEGER           AS tons_int
FROM fact_production
LIMIT 3;
:: — краткий оператор приведения типа, специфичный для PostgreSQL. CAST — стандарт SQL (работает везде).

TO_NUMBER, TO_DATE, TO_TIMESTAMP

-- TO_DATE: преобразование строки в дату по шаблону
SELECT TO_DATE('15.03.2024', 'DD.MM.YYYY') AS parsed_date;

-- TO_TIMESTAMP: преобразование строки во временну́ю метку

SELECT TO_NUMBER('1 234,56', '9G999D99') AS parsed_num;

       TO_CHAR(12345.6, 'FM999G999D00')         AS formatted_num;
Шаблоны: DD — день, MM — месяц, YYYY — год, HH24 — час (24ч), MI — минуты, 9 — цифра, D — разделитель дробной части, G — разделитель тысяч.

Неявное vs явное преобразование

Явное преобразование (рекомендуется)

Программист указывает тип: CAST(x AS INTEGER) или x::INTEGER

Неявное преобразование (осторожно!)

СУБД автоматически приводит типы при сравнении / операциях

-- PostgreSQL: неявное приведение при конкатенации
SELECT 'Добыто: ' || tons_mined || ' тонн' -- NUMERIC -> TEXT автоматически
FROM fact_production LIMIT 1;

-- Ошибка! PostgreSQL строже MySQL:
-- SELECT '10' + 5;  -- Ошибка в PostgreSQL!
SELECT '10'::INTEGER + 5; -- Правильно: 15

Приоритет неявных преобразований (PostgreSQL)

При смешивании типов PostgreSQL приводит «младший» тип к «старшему» по иерархии:

ПриоритетТипПояснение
1 (высший)NUMERIC / DECIMALТочная арифметика — «побеждает» в вычислениях
2DOUBLE PRECISIONПриближённое число с плавающей точкой
3REAL4-байтовое число с плавающей точкой
4BIGINT8-байтовое целое
5INTEGER4-байтовое целое
6 (низший)SMALLINT2-байтовое целое
Правило: при операции INTEGER + NUMERIC → результат NUMERIC.
При INTEGER + DOUBLE PRECISION → результат DOUBLE PRECISION.

Примеры неявных преобразований (PostgreSQL)

-- 1. INTEGER + NUMERIC → NUMERIC
SELECT 10 + 2.5;              -- результат: 12.5 (NUMERIC)

-- 2. INTEGER / INTEGER → INTEGER (потеря точности!)
SELECT 7 / 2;                 -- результат: 3, а не 3.5!
SELECT 7 / 2.0;               -- результат: 3.5 (NUMERIC)

-- 3. Конкатенация: число → текст автоматически
SELECT 'Смена #' || shift_id   -- shift_id (INT) → TEXT
FROM fact_production LIMIT 1;

-- 4. Сравнение DATE и TIMESTAMP → TIMESTAMP
SELECT * FROM fact_production
WHERE production_date > '2024-03-15';
-- строка '2024-03-15' → DATE неявно
Ловушка: целочисленное деление 7 / 2 = 3. Чтобы получить дробь, хотя бы один операнд должен быть NUMERIC или FLOAT.

Неявные преобразования в DAX

DAX выполняет неявные преобразования гораздо агрессивнее, чем PostgreSQL:

ОперацияDAX-поведениеPostgreSQL
"10" + 515 (текст → число автоматически)Ошибка!
"3.14" * 26.28 (текст → число)Ошибка!
TRUE + TRUE2 (TRUE = 1, FALSE = 0)2 (аналогично)
1 & " шт.""1 шт." (число → текст для &)Используется ||, аналогично
"2024-03-15" + 1Дата + 1 день (текст → дата → число)Ошибка без CAST
Осторожно: «Удобство» DAX может маскировать ошибки в данных. Строка "N/A" вместо числа вызовет ошибку только в момент вычисления, а не при загрузке.

Приоритет типов в DAX

При смешивании типов DAX приводит оба операнда к типу с наивысшим приоритетом:

ПриоритетТип DAXАналог SQL
1 (высший)Currency (фикс. точка)NUMERIC(19,4)
2Decimal Number (Double)DOUBLE PRECISION
3Whole Number (Int64)BIGINT
4DateTimeTIMESTAMP

Пример: Whole Number + Decimal Number → результат Decimal Number.
Но для оператора & (конкатенация) оба операнда всегда приводятся к Text.

Неявные преобразования: SQL vs DAX — выводы

PostgreSQL

  • Строгий подход — ошибка при несовместимых типах
  • Числовые типы приводятся автоматически (INT → NUMERIC)
  • Текст → число: только явно через CAST
  • Конкатенация ||: число → текст автоматически
  • Целочисленное деление сохраняется

DAX

  • Гибкий подход — пытается преобразовать автоматически
  • Текст «10» + число 5 = 15 (без ошибки)
  • Boolean → число: TRUE = 1, FALSE = 0
  • Конкатенация &: всё → текст
  • Нет целочисленного деления: 7 / 2 = 3.5
Рекомендация: Используйте явное преобразование в обоих языках — это делает код читаемым и предсказуемым.

CASE WHEN, COALESCE, NULLIF, GREATEST, LEAST

Суть: Принятие решений внутри запроса — классификация, подстановка значений, защита от ошибок.

CASE WHEN — простая форма

-- Классификация руды по содержанию Fe
SELECT
    sample_number,
    fe_content,
    CASE
        WHEN fe_content >= 65 THEN 'Богатая руда'
        WHEN fe_content >= 55 THEN 'Средняя руда'
        WHEN fe_content >= 45 THEN 'Бедная руда'
        ELSE 'Забалансовая'
    END AS ore_category,
    CASE
        WHEN fe_content >= 60 THEN 'Соответствует'
        ELSE 'Не соответствует'
    END AS meets_target
FROM fact_ore_quality
WHERE date_id = 20240315
ORDER BY fe_content DESC;
Порядок важен! Условия проверяются сверху вниз. Первое истинное срабатывает. Всегда добавляйте ELSE.

Две формы CASE

Простая форма

-- Сравнение с одним выражением
SELECT
    shift_id,
    CASE shift_id
        WHEN 1 THEN 'Утренняя'
        WHEN 2 THEN 'Дневная'
        WHEN 3 THEN 'Ночная'
        ELSE 'Неизвестно'
    END AS shift_name
FROM fact_production;

Поисковая форма

-- Произвольные условия
SELECT
    equipment_id,
    tons_mined,
    CASE
        WHEN tons_mined > 150 THEN 'Высокая'
        WHEN tons_mined > 100 THEN 'Средняя'
        WHEN tons_mined > 50  THEN 'Низкая'
        ELSE 'Критически низкая'
    END AS productivity
FROM fact_production;
Простая форма — когда сравниваем одно поле с конкретными значениями. Поисковая — когда нужны диапазоны или разные столбцы.

CASE внутри агрегатных функций

-- Кросс-табуляция: добыча по категориям качества
SELECT
    d.full_date,
    SUM(CASE WHEN oq.fe_content >= 60 THEN 1 ELSE 0 END) AS good_samples,
    SUM(CASE WHEN oq.fe_content < 60  THEN 1 ELSE 0 END) AS poor_samples,
    ROUND(
        100.0 * SUM(CASE WHEN oq.fe_content >= 60 THEN 1 ELSE 0 END)
        / NULLIF(COUNT(*), 0),
    1) AS good_pct
FROM fact_ore_quality oq
JOIN dim_date d ON oq.date_id = d.date_id
WHERE d.year = 2024 AND d.month = 3
GROUP BY d.full_date
ORDER BY d.full_date;
Паттерн: SUM(CASE WHEN ... THEN 1 ELSE 0 END) — подсчёт строк по условию. Альтернатива: COUNT(*) FILTER (WHERE ...) в PostgreSQL.

SQL CASE vs DAX IF / SWITCH

SQL: CASE WHEN

SELECT
    fe_content,
    CASE
        WHEN fe_content >= 65 THEN 'Богатая'
        WHEN fe_content >= 55 THEN 'Средняя'
        WHEN fe_content >= 45 THEN 'Бедная'
        ELSE 'Забалансовая'
    END AS category
FROM fact_ore_quality;

DAX: IF / SWITCH

// IF (вложенный)
EVALUATE
ADDCOLUMNS(fact_ore_quality,
  "Категория",
    IF(fact_ore_quality[fe_content] >= 65,
      "Богатая",
      IF(fact_ore_quality[fe_content] >= 55,
        "Средняя",
        IF(fact_ore_quality[fe_content] >= 45,
          "Бедная",
          "Забалансовая")))
)

// SWITCH(TRUE(), ...) — более читаемо
"Категория" =
SWITCH(TRUE(),
  fact_ore_quality[fe_content] >= 65, "Богатая",
  fact_ore_quality[fe_content] >= 55, "Средняя",
  fact_ore_quality[fe_content] >= 45, "Бедная",
  "Забалансовая"
)
SWITCH(TRUE(), ...) в DAX — самый близкий аналог поискового CASE WHEN из SQL.

COALESCE — первое не-NULL значение

SQL (PostgreSQL)

-- Подстановка значения по умолчанию
SELECT
    sample_number,
    fe_content,
    COALESCE(sio2_content, 0) AS sio2_safe,
    COALESCE(al2o3_content, 0) AS al2o3_safe,
    COALESCE(moisture, 0)     AS moisture_safe,
    COALESCE(
        comment,

    ) AS comment_text

WHERE date_id = 20240315;

-- Цепочка: первое непустое
SELECT COALESCE(
    sio2_content,
    al2o3_content,
    0
) AS first_non_null;

DAX

// COALESCE (DAX 2021+)
EVALUATE
ADDCOLUMNS(
  FILTER(fact_ore_quality,

  "SiO2 безоп.",

NULLIF — защита от деления на ноль

-- NULLIF(a, b): возвращает NULL, если a = b, иначе a

-- Расчёт эффективности транспортировки
SELECT
    equipment_id,
    tons_mined,
    tons_transported,
    trips_count,
    -- Без защиты: ошибка при trips_count = 0
    -- tons_transported / trips_count,

    -- С защитой: NULLIF превращает 0 в NULL, деление на NULL = NULL
    ROUND(
        tons_transported / NULLIF(trips_count, 0), 2
    ) AS tons_per_trip
FROM fact_production
WHERE date_id = 20240315;
Паттерн: x / NULLIF(y, 0) — безопасное деление. Если y = 0, результат NULL вместо ошибки.
DAX: Используйте DIVIDE(a, b, alt) — встроенная безопасная дивизия: DIVIDE(tons_transported, trips_count, 0)

Безопасное деление: SQL vs DAX

SQL (PostgreSQL)

-- Вариант 1: NULLIF
SELECT
    tons_transported
    / NULLIF(trips_count, 0) AS v1,

-- Вариант 2: CASE WHEN
    CASE
        WHEN trips_count = 0 THEN NULL
        ELSE tons_transported
             / trips_count
    END AS v2,

-- Вариант 3: COALESCE + NULLIF
    COALESCE(
        tons_transported
        / NULLIF(trips_count, 0),
        0
    ) AS v3
FROM fact_production;

DAX

// DIVIDE — одна функция решает всё!
EVALUATE
ADDCOLUMNS(
  fact_production,
  "Тонн/рейс",
    DIVIDE(
      fact_production[tons_transported],
      fact_production[trips_count],
      0  -- альтернативный результат при /0
    )
)

// DIVIDE(числитель, знаменатель [, альт])
// Если знаменатель = 0 или BLANK,
// возвращает альт (по умолчанию BLANK)
Преимущество DAX: DIVIDE() — чище и безопаснее. В SQL приходится комбинировать NULLIF + COALESCE.

GREATEST / LEAST

-- GREATEST: наибольшее из списка значений
-- LEAST: наименьшее из списка значений

SELECT
    sample_number,
    fe_content,
    sio2_content,
    al2o3_content,
    -- Максимальное содержание примеси
    GREATEST(
        COALESCE(sio2_content, 0),
        COALESCE(al2o3_content, 0)
    ) AS max_impurity,
    -- Ограничение значения в диапазоне (clamp)
    GREATEST(LEAST(fe_content, 70.0), 40.0) AS fe_clamped
    -- Результат будет в диапазоне [40, 70]
FROM fact_ore_quality
WHERE date_id = 20240315;
DAX: Прямых аналогов GREATEST/LEAST нет. Используйте MAX(a, b) / MIN(a, b) в вычисляемых столбцах или вложенные IF.

Часть 5. Работа с NULL

NULL — не значение, а отсутствие значения

Три правила NULL

  1. NULL = NULL — не TRUE и не FALSE, а UNKNOWN
  2. NULL в арифметике: любая операция с NULL = NULL
  3. NULL в агрегатах: SUM, AVG, COUNT(col) игнорируют NULL

IS NULL / IS NOT NULL

-- Найти незавершённые простои (end_time не заполнен)
SELECT
    d.downtime_id,
    e.equipment_name,
    d.start_time,
    d.end_time,       -- NULL = простой продолжается
    d.duration_min     -- NULL = длительность не рассчитана
FROM fact_equipment_downtime d
JOIN dim_equipment e ON d.equipment_id = e.equipment_id
WHERE d.end_time IS NULL;

-- Пробы, где SiO2 не определён
SELECT COUNT(*) AS total,
       COUNT(sio2_content) AS with_sio2,
       COUNT(*) - COUNT(sio2_content) AS without_sio2
FROM fact_ore_quality;
НИКОГДА: WHERE sio2_content = NULL — всегда возвращает 0 строк! Используйте IS NULL.

NULL в DAX: BLANK()

SQL: NULL

-- Проверка на NULL
SELECT *
FROM fact_ore_quality
WHERE sio2_content IS NULL;

-- Подстановка
SELECT COALESCE(sio2_content, 0)
FROM fact_ore_quality;

-- Установка NULL
UPDATE fact_ore_quality
SET sio2_content = NULL
WHERE quality_id = 5;

DAX: BLANK

// Проверка: ISBLANK
EVALUATE
FILTER(fact_ore_quality,
  ISBLANK(fact_ore_quality[sio2_content])
)

// Подстановка: COALESCE или IF
EVALUATE
ADDCOLUMNS(fact_ore_quality,
  "SiO2 безоп.",
    COALESCE(
      fact_ore_quality[sio2_content], 0)
)

// Создание BLANK
"Результат" =
IF(condition, value, BLANK())
SQL NULL = DAX BLANK(). Проверка: IS NULL = ISBLANK(). Важно: в DAX BLANK() + 5 = 5, а в SQL NULL + 5 = NULL!

Подводные камни NULL

-- 1. NOT IN с NULL — ОПАСНО!
-- Если подзапрос возвращает NULL, NOT IN вернёт пустой результат
SELECT * FROM dim_equipment
WHERE equipment_id NOT IN (1, 2, NULL);
-- Вернёт 0 строк! Используйте NOT EXISTS или добавьте фильтр

-- 2. UNION и NULL
SELECT NULL UNION SELECT NULL;
-- Вернёт одну строку (UNION считает NULL = NULL для дедупликации)

-- 3. ORDER BY и NULL
SELECT equipment_name, sio2_content
FROM fact_ore_quality
ORDER BY sio2_content NULLS LAST; -- или NULLS FIRST

-- 4. COALESCE в WHERE
SELECT * FROM fact_ore_quality
WHERE COALESCE(sio2_content, 0) > 5; -- Работает, но не использует индекс!
Правило: NULLS FIRST / NULLS LAST в ORDER BY — специфика PostgreSQL. По умолчанию NULL идут последними при ASC.

Мини-квиз: NULL

Вопрос: Что вернёт этот запрос?

SELECT
    CASE WHEN NULL = NULL      THEN 'Да' ELSE 'Нет' END AS test1,
    CASE WHEN NULL IS NULL     THEN 'Да' ELSE 'Нет' END AS test2,
    CASE WHEN NOT (NULL = 1)   THEN 'Да' ELSE 'Нет' END AS test3,
    COALESCE(NULL, NULL, 'OK') AS test4,
    NULLIF(5, 5)               AS test5;
A) 'Да', 'Да', 'Да', 'OK', NULL
B) 'Нет', 'Да', 'Нет', 'OK', NULL
C) 'Нет', 'Да', 'Да', NULL, 5
D) NULL, 'Да', 'Нет', 'OK', 0
Ответ: B) NULL=NULL → UNKNOWN → ELSE. NULL IS NULL → TRUE. NOT(UNKNOWN) → UNKNOWN → ELSE. COALESCE пропускает NULL. NULLIF(5,5) → NULL.

Часть 6. Кейс: KPI эффективности оборудования

Бизнес-задача: Руководство «Руда+» запросило ежемесячный отчёт по эффективности оборудования с расчётом следующих KPI:
  • Коэффициент использования = отработано часов / плановое время
  • Производительность = тонн / отработано часов
  • Категория эффективности: высокая / средняя / низкая
  • Процент качественной руды (Fe ≥ 60%)

Кейс: SQL решение

SELECT
    e.equipment_name,
    et.type_name,
    COUNT(fp.production_id)             AS total_shifts,
    ROUND(SUM(fp.tons_mined), 1)        AS total_tons,
    ROUND(SUM(fp.operating_hours), 1)   AS total_hours,
    -- KPI: производительность (тонн/час)
    ROUND(
        SUM(fp.tons_mined)
        / NULLIF(SUM(fp.operating_hours), 0), 2
    ) AS tons_per_hour,
    -- KPI: коэффициент использования (%)
    ROUND(
        100.0 * SUM(fp.operating_hours)
        / NULLIF(COUNT(fp.production_id) * 8.0, 0), 1
    ) AS utilization_pct,
    -- Категория эффективности
    CASE
        WHEN SUM(fp.tons_mined) / NULLIF(SUM(fp.operating_hours), 0) > 20
            THEN 'Высокая'
        WHEN SUM(fp.tons_mined) / NULLIF(SUM(fp.operating_hours), 0) > 12
            THEN 'Средняя'
        ELSE 'Низкая'
    END AS efficiency_category
FROM fact_production fp
JOIN dim_equipment e ON fp.equipment_id = e.equipment_id
JOIN dim_equipment_type et ON e.equipment_type_id = et.equipment_type_id
WHERE fp.date_id BETWEEN 20240301 AND 20240331
GROUP BY e.equipment_name, et.type_name
ORDER BY tons_per_hour DESC;

Кейс: DAX решение

EVALUATE
VAR _march =
  FILTER(fact_production,
    fact_production[date_id] >= 20240301
    && fact_production[date_id] <= 20240331)
RETURN
ADDCOLUMNS(
  SUMMARIZE(_march,
    dim_equipment[equipment_name],
    dim_equipment_type[type_name]),
  "Всего смен",    CALCULATE(COUNTROWS(fact_production), _march),
  "Всего тонн",    CALCULATE(SUM(fact_production[tons_mined]), _march),
  "Всего часов",   CALCULATE(SUM(fact_production[operating_hours]), _march),
  "Тонн/час",
    DIVIDE(
      CALCULATE(SUM(fact_production[tons_mined]), _march),
      CALCULATE(SUM(fact_production[operating_hours]), _march),
      0),
  "Использование %",
    DIVIDE(
      CALCULATE(SUM(fact_production[operating_hours]), _march),
      CALCULATE(COUNTROWS(fact_production), _march) * 8,
      0) * 100,
  "Категория",
    SWITCH(TRUE(),
      DIVIDE(
        CALCULATE(SUM(fact_production[tons_mined]), _march),
        CALCULATE(SUM(fact_production[operating_hours]), _march), 0) > 20,
          "Высокая",
      DIVIDE(
        CALCULATE(SUM(fact_production[tons_mined]), _march),
        CALCULATE(SUM(fact_production[operating_hours]), _march), 0) > 12,
          "Средняя",
      "Низкая")
)
