Транзакция — логическая единица работы, которая переводит базу данных из одного согласованного состояния в другое.

ACID на примере «Руда+»

Свойство Пример
Atomicity
Всё или ничего		 Запись данных о смене: добыча + качество + простои. Если ошибка при записи простоев — откатываются ВСЕ три вставки.

Durability
Надёжность		 После COMMIT данные сохраняются даже при сбое питания — PostgreSQL использует WAL (Write-Ahead Log).

2. BEGIN / COMMIT / ROLLBACK

-- Успешная транзакция
BEGIN;
    INSERT INTO fact_production (production_id, date_id, shift_id,
        equipment_id, mine_id, operator_id, tons_mined, trips_count, operating_hours)
    VALUES (800001, 20250201, 1, 1, 1, 1, 120.5, 10, 7.5);

    INSERT INTO fact_production (production_id, date_id, shift_id,
        equipment_id, mine_id, operator_id, tons_mined, trips_count, operating_hours)
    VALUES (800002, 20250201, 1, 2, 1, 2, 98.3, 8, 6.5);
COMMIT;

-- Отмена транзакции
BEGIN;
    INSERT INTO fact_production (...)
    VALUES (800003, 20250201, 2, 1, 1, 1, 200.0, 15, 8.0);
    -- Обнаружена ошибка в данных
ROLLBACK;  -- Запись 800003 НЕ сохранена

Явные и неявные транзакции

Явная (Explicit) Неявная (Implicit / Autocommit)
Синтаксис Программист пишет BEGIN … COMMIT / ROLLBACK Каждая отдельная SQL-команда автоматически оборачивается в транзакцию
Когда фиксируется Только по команде COMMIT Сразу после успешного выполнения команды
Откат ROLLBACK отменяет все изменения с момента BEGIN Откатить нельзя — изменения уже зафиксированы
При ошибке Транзакция переходит в состояние aborted. Все предыдущие изменения откатываются при ROLLBACK. Данные остаются согласованными. Ошибочная команда не выполняется, но все предыдущие команды уже зафиксированы и откатить их нельзя. Данные могут быть несогласованы.
Пример BEGIN; INSERT ...; UPDATE ...; COMMIT; INSERT INTO fact_production VALUES (...);
PostgreSQL работает в режиме autocommit по умолчанию. Если вы не написали BEGIN, каждая команда — это отдельная неявная транзакция, которая автоматически фиксируется.

Явные vs неявные: пример «Руда+»

✅ Явная транзакция

-- Атомарная запись данных смены
BEGIN;
    INSERT INTO fact_production
    VALUES (900001, 20250301, 1, 1, 1, 1,
            150.0, 12, 8.0);

    INSERT INTO fact_ore_quality
    VALUES (500001, 20250301, 1, 1,
            62.5, 4.2, 0.08, 'A');

    INSERT INTO fact_downtime
    VALUES (700001, 20250301, 1, 1,
            3, 45, 'Замена ковша');
COMMIT;
-- Все 3 записи сохранены вместе

⚠️ Неявные транзакции

-- Каждая команда — отдельная транзакция
INSERT INTO fact_production
VALUES (900001, 20250301, 1, 1, 1, 1,
        150.0, 12, 8.0);
-- ✅ Зафиксировано

INSERT INTO fact_ore_quality
VALUES (500001, 20250301, 1, 1,
        62.5, 4.2, 0.08, 'A');
-- ✅ Зафиксировано

INSERT INTO fact_downtime
VALUES (700001, 20250301, 1, 1,
        3, 45, 'Замена ковша');
-- ❌ ОШИБКА! Но предыдущие
-- записи УЖЕ сохранены —
-- данные несогласованы!
Вывод: Когда несколько операций должны выполняться как единое целое — всегда используйте явную транзакцию (BEGIN ... COMMIT). Autocommit подходит только для одиночных независимых операций.

🗣️ Вопрос для обсуждения

Диспетчер MES-системы «Руда+» выполняет INSERT для записи данных о добыче, но забывает написать BEGIN.

Что произойдёт с данными?

Сможет ли он откатить изменения?

Ответ: Данные будут сразу зафиксированы (autocommit). Откатить командой ROLLBACK уже нельзя — придётся вручную удалять или обновлять записи.

Ошибка внутри транзакции

BEGIN;
    -- Первый INSERT — OK
    INSERT INTO fact_production (production_id, ...)
    VALUES (800004, ...);

    -- Второй INSERT — ОШИБКА (FK violation)
    INSERT INTO fact_production (production_id, ..., equipment_id, ...)
    VALUES (800005, ..., 99999, ...);
    -- ERROR: insert or update on table "fact_production" violates
    --        foreign key constraint

    -- Любой последующий запрос:
    SELECT 1;
    -- ERROR: current transaction is aborted

    -- Единственный выход:
ROLLBACK;  -- Откатываются ОБА INSERT
В PostgreSQL после ошибки внутри транзакции она переходит в состояние «aborted». Единственное, что можно сделать — ROLLBACK. Это отличие от некоторых других СУБД.

3. SAVEPOINT — точки сохранения

Зачем нужен SAVEPOINT?

Позволяет откатить часть транзакции, не теряя остальные изменения.

BEGIN;
    -- Часть 1: данные о добыче (критичные)
    INSERT INTO fact_production (...) VALUES (...);

    SAVEPOINT sp_quality;  -- Точка сохранения

    -- Часть 2: данные о качестве (могут быть ошибки)
    INSERT INTO fact_ore_quality (...) VALUES (...);

    -- Если ошибка:
    ROLLBACK TO sp_quality;  -- Откатить ТОЛЬКО quality

    -- production — сохранена!
    RELEASE SAVEPOINT sp_quality;  -- Освободить SAVEPOINT
COMMIT;

SAVEPOINT в PL/pgSQL

Важно: В PL/pgSQL нельзя использовать SAVEPOINT/ROLLBACK TO напрямую. Вместо этого используются вложенные BEGIN...EXCEPTION...END блоки, которые автоматически создают неявный SAVEPOINT. 
DO $$
DECLARE
    v_ok INT := 0; v_err INT := 0;
BEGIN
    FOR i IN 1..10 LOOP
        -- Каждая итерация — неявный SAVEPOINT
        BEGIN
            INSERT INTO fact_production (...) VALUES (...);
            v_ok := v_ok + 1;
        EXCEPTION WHEN OTHERS THEN
            -- Откат ТОЛЬКО этой итерации
            v_err := v_err + 1;
        END;
    END LOOP;
    -- Все успешные INSERT сохранены!
    RAISE NOTICE 'OK: %, ошибок: %', v_ok, v_err;
END;
$$;

Вложенные транзакции (Nested Transactions)

PostgreSQL НЕ поддерживает вложенные транзакции!

BEGIN;  -- Транзакция 1
    INSERT INTO fact_production (...) VALUES (...);

    BEGIN;  -- ⚠️ WARNING: there is already a transaction in progress
        INSERT INTO fact_ore_quality (...) VALUES (...);
    COMMIT;  -- Фиксирует ВСЮ транзакцию 1, а не «внутреннюю»

    INSERT INTO fact_downtime (...) VALUES (...);
    -- ERROR: эта команда выполнится ВНЕ транзакции!
Второй BEGIN игнорируется с предупреждением. Первый COMMIT закрывает всю транзакцию. Вложенности нет — это частая ошибка.

Альтернативы вложенным транзакциям

Механизм Что делает Ограничения
SAVEPOINT
(SQL)		 Точка внутри транзакции для частичного отката — по сути «подтранзакция» Нельзя зафиксировать часть отдельно — итоговый COMMIT фиксирует всё
BEGIN...EXCEPTION...END
(PL/pgSQL)		 Вложенный блок с неявным SAVEPOINT — откат только своего блока при ошибке Работает только в PL/pgSQL, не в обычном SQL
Автономные транзакции Независимая транзакция внутри другой — фиксируется отдельно Не поддерживаются в PostgreSQL (есть в Oracle). Обходной путь — dblink

SAVEPOINT как замена вложенных транзакций

❌ Вложенные (не работает)

BEGIN;  -- Внешняя
    INSERT INTO fact_production
    VALUES (...);

    BEGIN;  -- «Внутренняя» — нет!
        INSERT INTO fact_ore_quality
        VALUES (...);
        -- Ошибка — хотим откатить
        -- только quality
    ROLLBACK;  -- Откатит ВСЁ!

    -- production тоже потеряна
COMMIT;

✅ SAVEPOINT (работает)

BEGIN;
    INSERT INTO fact_production
    VALUES (...);

    SAVEPOINT sp_quality;

        INSERT INTO fact_ore_quality
        VALUES (...);
        -- Ошибка — откатываем
        -- только quality
    ROLLBACK TO sp_quality;

    -- production сохранена!
COMMIT;
Правило: Вместо вложенных BEGIN всегда используйте SAVEPOINT. Это единственный способ частичного отката внутри транзакции в PostgreSQL.

Откат внешнего блока отменяет ВСЕ вложенные

DO $$
DECLARE
    v_result TEXT := '';
BEGIN
    -- Вложенный блок 1 — успех
    BEGIN
        INSERT INTO fact_production (production_id, date_id, shift_id,
            equipment_id, mine_id, operator_id, tons_mined, trips_count, operating_hours)
        VALUES (900020, 20250301, 1, 1, 1, 1, 120.0, 10, 7.5);
        v_result := v_result || 'Блок 1: OK | ';
    EXCEPTION WHEN OTHERS THEN
        v_result := v_result || 'Блок 1: ОШИБКА | ';
    END;

    -- Вложенный блок 2 — успех
    BEGIN
        INSERT INTO fact_ore_quality (quality_id, date_id, shift_id,
            mine_id, fe_content, moisture, impurity_pct, grade)
        VALUES (500020, 20250301, 1, 1, 63.0, 4.0, 0.07, 'A');
        v_result := v_result || 'Блок 2: OK | ';
    EXCEPTION WHEN OTHERS THEN
        v_result := v_result || 'Блок 2: ОШИБКА | ';
    END;

    RAISE NOTICE '%', v_result;  -- «Блок 1: OK | Блок 2: OK |»

    -- Ошибка во внешнем блоке!
    RAISE EXCEPTION 'Критическая ошибка во внешнем блоке!';

END;
$$;
-- ERROR: Критическая ошибка во внешнем блоке!
-- Блоки 1 и 2 выполнились успешно, НО:
-- ❌ production  — ОТКАЧЕНА
-- ❌ ore_quality — ОТКАЧЕНА

Почему так происходит?

1. Вложенный блок завершился успешно
→ Неявный SAVEPOINT освобождается (RELEASE)
→ Изменения вливаются в родительскую транзакцию
→ Но НЕ фиксируются — это не COMMIT!
2. Вложенный блок завершился с ошибкой
→ Выполняется ROLLBACK TO неявного SAVEPOINT
→ Откатываются только изменения этого блока
→ Родительская транзакция продолжает работу
3. Внешний блок завершился с ошибкой / ROLLBACK
→ Откатывается ВСЯ транзакция целиком
→ Все изменения всех вложенных блоков отменяются
→ Даже если внутренние блоки завершились успешно
Запомните: RELEASE SAVEPOINT ≠ COMMIT. Успешный вложенный блок лишь говорит: «мои изменения готовы». Окончательная судьба данных решается на уровне внешней транзакции.

Подтранзакция ≠ Транзакция

Функция поддерживает транзакции? — Нет!

Наличие BEGIN...EXCEPTION...END в функции — это подтранзакции (subtransactions), а не управление транзакциями.

Возможность FUNCTION PROCEDURE / DO
BEGIN...EXCEPTION...END
(неявный SAVEPOINT)		 ✅ Да ✅ Да
COMMIT / ROLLBACK ❌ Нет ✅ Да
Начать новую транзакцию ❌ Нет ✅ Да
Свой уровень изоляции ❌ Нет ✅ Да (после COMMIT)
Контекст выполнения Внутри транзакции вызывающего Может управлять своими транзакциями

Что может и чего не может подтранзакция

Подтранзакция (FUNCTION)

  • ✅ Перехватить ошибку и продолжить работу
  • ✅ Откатить часть изменений
  • ❌ Зафиксировать изменения независимо
  • ❌ Пережить откат внешней транзакции
  • ❌ Установить свой уровень изоляции

Транзакция (PROCEDURE)

  • ✅ Перехватить ошибку и продолжить работу
  • ✅ Откатить часть изменений
  • ✅ Зафиксировать изменения (COMMIT)
  • ✅ Данные после COMMIT сохранены навсегда
  • ✅ Начать новую транзакцию с другим уровнем изоляции
Корректная формулировка: Функция поддерживает обработку ошибок с частичным откатом (через неявные SAVEPOINT), но не управление транзакциями. Для управления транзакциями используйте PROCEDURE.

Транзакции: SQL vs PL/pgSQL

Аспект SQL (интерактивный) PL/pgSQL
BEGIN Начинает транзакцию Начинает блок кода (не транзакцию!)
COMMIT / ROLLBACK Завершает транзакцию Доступно только в PROCEDURE.
В FUNCTION — запрещено!
SAVEPOINT SAVEPOINT sp1;
ROLLBACK TO sp1;		 Нельзя напрямую. Заменяется вложенным
BEGIN ... EXCEPTION ... END
Контекст транзакции Программист управляет сам Функция выполняется внутри транзакции вызывающего
Обработка ошибок Транзакция переходит в состояние «aborted», нужен ROLLBACK Блок EXCEPTION перехватывает ошибку и откатывает только свой блок

SQL vs PL/pgSQL: примеры

SQL

-- Явное управление транзакцией
BEGIN;
    INSERT INTO fact_production (...)
    VALUES (...);

    SAVEPOINT sp1;

    INSERT INTO fact_ore_quality (...)
    VALUES (...);
    -- При ошибке:
    ROLLBACK TO sp1;

COMMIT;

PL/pgSQL (функция)

CREATE FUNCTION load_shift_data()
RETURNS void AS $$
BEGIN  -- это блок кода, НЕ транзакция
    INSERT INTO fact_production (...)
    VALUES (...);

    BEGIN  -- вложенный блок = неявный SAVEPOINT
        INSERT INTO fact_ore_quality (...)
        VALUES (...);
    EXCEPTION WHEN OTHERS THEN
        RAISE NOTICE 'Ошибка quality: %',
              SQLERRM;
    END;
    -- production сохранена!
END;
$$ LANGUAGE plpgsql;

PL/pgSQL: PROCEDURE vs FUNCTION

✅ PROCEDURE (PostgreSQL 11+)

CREATE PROCEDURE close_shift(
    p_shift_id INT
) AS $$
BEGIN
    UPDATE fact_production
    SET status = 'closed'
    WHERE shift_id = p_shift_id;

    COMMIT;  -- ✅ Разрешено!

    INSERT INTO shift_log (shift_id, action)
    VALUES (p_shift_id, 'closed');

    COMMIT;  -- Вторая транзакция
END;
$$ LANGUAGE plpgsql;

-- Вызов:
CALL close_shift(42);

❌ FUNCTION

CREATE FUNCTION close_shift(
    p_shift_id INT
) RETURNS void AS $$
BEGIN
    UPDATE fact_production
    SET status = 'closed'
    WHERE shift_id = p_shift_id;

    COMMIT;  -- ❌ ОШИБКА!
    -- ERROR: invalid transaction
    -- termination

    -- Функция ВСЕГДА работает
    -- внутри транзакции
    -- вызывающего кода
END;
$$ LANGUAGE plpgsql;
Правило: Если логика требует управления транзакциями (COMMIT/ROLLBACK) — используйте PROCEDURE. Если нужно вернуть значение или использовать в SELECT — используйте FUNCTION.

✅ PostgreSQL 11+

DO $$

    COMMIT;  -- ✅ Разрешено!

    VALUES (...);

    COMMIT;  -- ❌ ОШИБКА!

    -- DO-блок ведёт себя как
    -- FUNCTION: работает внутри
    -- транзакции вызывающего
END;

PROCEDURE vs DO-блок vs FUNCTION

Конструкция COMMIT / ROLLBACK Возврат значений Вызов
PROCEDURE ✅ Да Только через OUT-параметры CALL proc_name()
DO-блок ✅ Да (PG 11+) Нет (анонимный) Выполняется напрямую
FUNCTION ❌ Нет RETURNS / SETOF / TABLE SELECT func_name()
DO-блок — это анонимная процедура. Начиная с PostgreSQL 11, он поддерживает COMMIT / ROLLBACK так же, как PROCEDURE. Удобен для одноразовых скриптов миграции и загрузки данных.

4. Два подхода к изоляции транзакций

Как СУБД обеспечивает изоляцию?

Существуют два принципиально разных механизма. Выбор механизма определяет поведение СУБД при конкурентном доступе.

🔒 Блокировки (Locking)

  • Транзакция блокирует данные, которые читает или изменяет
  • Другие транзакции ждут снятия блокировки
  • Пессимистичный подход: «заблокирую заранее, чтобы никто не помешал»

СУБД: MS SQL Server, MySQL (InnoDB — гибрид), DB2

📸 Версионность (MVCC)

  • Каждая транзакция видит свой снимок (snapshot) данных
  • Читатели не блокируют писателей и наоборот
  • Оптимистичный подход: «работаю со своей версией, конфликт проверю при COMMIT»

СУБД: PostgreSQL, Oracle, MySQL (InnoDB — гибрид)

Блокировки vs MVCC: сравнение

Критерий Блокировки (Locking) Версионность (MVCC)
Чтение + Запись Читатель ждёт, пока писатель завершит транзакцию Читатель видит старую версию строки — не ждёт
Чтение + Чтение Могут блокировать друг друга (shared locks) Никогда не блокируют друг друга
Запись + Запись Блокировка до завершения первой транзакции Блокировка тоже есть — MVCC не устраняет write-write конфликты
Накладные расходы Lock manager, возможны deadlocks Хранение нескольких версий строк, нужен VACUUM
Пропускная способность Ниже при интенсивном чтении (читатели ждут) Выше — читатели и писатели работают параллельно
PostgreSQL использует MVCC — именно поэтому SELECT никогда не блокируется обычным UPDATE. Это ключевое преимущество для систем с интенсивным чтением, таких как MES-аналитика «Руда+».

Пример: «Руда+» — два оператора одновременно

🔒 Модель на блокировках (MS SQL)

-- Сессия 1: аналитик строит отчёт
BEGIN TRANSACTION;
SELECT SUM(tons_mined)
FROM fact_production
WHERE date_id = 20250301;
-- Накладывает shared lock на строки

-- Сессия 2: оператор вносит данные
BEGIN TRANSACTION;
UPDATE fact_production
SET tons_mined = 155.0
WHERE production_id = 900001;
-- ⏳ ЖДЁТ! Shared lock мешает
-- Оператор заблокирован, пока
-- аналитик не завершит SELECT

📸 Модель MVCC (PostgreSQL)

-- Сессия 1: аналитик строит отчёт
BEGIN;
SELECT SUM(tons_mined)
FROM fact_production
WHERE date_id = 20250301;
-- Читает из своего снимка (snapshot)

-- Сессия 2: оператор вносит данные
BEGIN;
UPDATE fact_production
SET tons_mined = 155.0
WHERE production_id = 900001;
-- ✅ Выполняется СРАЗУ!
-- Создаётся новая версия строки
-- Аналитик продолжает видеть
-- старое значение из снимка
Результат: В MVCC аналитик и оператор работают одновременно без ожидания. В модели на блокировках оператор вынужден ждать завершения аналитического запроса.

Уровни изоляции транзакций

Уровень Dirty Read Non-Repeatable Read Phantom Read Serialization Anomaly
READ UNCOMMITTED Нет* Да Да Да
READ COMMITTED Нет Да Да Да
REPEATABLE READ Нет Нет Нет* Да
SERIALIZABLE Нет Нет Нет Нет
* В PostgreSQL: READ UNCOMMITTED работает как READ COMMITTED (dirty read невозможен). REPEATABLE READ защищает и от phantom reads (благодаря MVCC).

Установка уровня изоляции

-- Для текущей транзакции
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- ...
COMMIT;

-- Для всей сессии
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- Проверка текущего уровня
SHOW transaction_isolation;
-- По умолчанию: read committed
Рекомендация: В большинстве случаев READ COMMITTED (по умолчанию) достаточен. Используйте REPEATABLE READ для аналитических отчётов, SERIALIZABLE — для критичных финансовых операций.

Аномалии конкурентного доступа

Dirty Read (грязное чтение)

Сессия A

BEGIN;
UPDATE fact_production
SET tons_mined = 999
WHERE production_id = 1;
-- НЕ COMMIT!

Сессия B

SELECT tons_mined
FROM fact_production
WHERE production_id = 1;
-- Dirty Read: видит 999?
PostgreSQL: Dirty Read невозможен ни на каком уровне изоляции. Даже при READ UNCOMMITTED фактически работает READ COMMITTED.

Non-Repeatable Read

Сессия A (READ COMMITTED)

BEGIN;
SELECT SUM(tons_mined)
FROM fact_production
WHERE date_id = 20250115;
-- Результат: 5000

-- ... время проходит ...

SELECT SUM(tons_mined)
FROM fact_production
WHERE date_id = 20250115;
-- Результат: 5150 (!)
COMMIT;

Сессия B

-- Между двумя SELECT Сессии A:
INSERT INTO fact_production
    (production_id, date_id, ...,
     tons_mined, ...)
VALUES
    (800020, 20250115, ...,
     150.0, ...);
-- Автокоммит
Проблема: Один и тот же запрос возвращает разные результаты внутри одной транзакции. Решение: Используйте REPEATABLE READ.

Phantom Read (фантомное чтение)

Сессия A

BEGIN;
SELECT COUNT(*)
FROM fact_production
WHERE date_id = 20250115;
-- Результат: 100

-- ... время ...

SELECT COUNT(*)
FROM fact_production
WHERE date_id = 20250115;
-- Результат: 101 (!)
-- Появилась «фантомная» строка
COMMIT;

Сессия B

INSERT INTO fact_production
    (production_id,
     date_id, ...)
VALUES
    (800021, 20250115, ...);
-- Новая строка попадает
-- в диапазон запроса A
В PostgreSQL REPEATABLE READ защищает от phantom reads (в отличие от стандарта SQL, где для этого нужен SERIALIZABLE).

Serialization Anomaly (аномалия сериализации)

Что это?

Результат параллельного выполнения транзакций отличается от любого возможного последовательного выполнения тех же транзакций. Даже если каждая транзакция корректна сама по себе — вместе они дают логически невозможный результат.

Отличие от других аномалий: Dirty Read, Non-Repeatable Read, Phantom Read — это проблемы чтения. Serialization Anomaly — это проблема логической согласованности при взаимозависимых записях.

Serialization Anomaly: пример «Руда+»

Сессия A — Диспетчер шахты 1

BEGIN ISOLATION LEVEL REPEATABLE READ;

-- Читает итог по шахте 2
SELECT SUM(tons_mined)
FROM fact_production
WHERE mine_id = 2
  AND date_id = 20250301;
-- Результат: 500 тонн

-- Записывает бонус для шахты 1
-- на основе данных шахты 2
INSERT INTO shift_bonuses
    (mine_id, date_id, bonus)
VALUES (1, 20250301,
        500 * 0.1);  -- 50

COMMIT;  -- ✅ Успех

Сессия B — Диспетчер шахты 2

BEGIN ISOLATION LEVEL REPEATABLE READ;

-- Читает итог по шахте 1
SELECT SUM(tons_mined)
FROM fact_production
WHERE mine_id = 1
  AND date_id = 20250301;
-- Результат: 800 тонн

-- Записывает бонус для шахты 2
-- на основе данных шахты 1
INSERT INTO shift_bonuses
    (mine_id, date_id, bonus)
VALUES (2, 20250301,
        800 * 0.1);  -- 80

COMMIT;  -- ✅ Успех

Serialization Anomaly: почему это проблема?

Обе транзакции успешно завершились. Но результат невозможен!

Порядок Бонус шахты 1 Бонус шахты 2
Сначала A, потом B 50 (видит 500 т шахты 2) 80 (видит 800 т шахты 1 + бонус A)
Сначала B, потом A 50 (видит 500 т шахты 2 + бонус B) 80 (видит 800 т шахты 1)
Параллельно (аномалия!) 50 (не видит бонус B) 80 (не видит бонус A)
Каждая транзакция работала со старым снимком — ни одна не увидела запись другой. Результат не соответствует ни одному последовательному порядку выполнения.

Решение: SERIALIZABLE

-- Те же операции, но с уровнем SERIALIZABLE
BEGIN ISOLATION LEVEL SERIALIZABLE;

SELECT SUM(tons_mined)
FROM fact_production
WHERE mine_id = 2 AND date_id = 20250301;

INSERT INTO shift_bonuses (mine_id, date_id, bonus)
VALUES (1, 20250301, 500 * 0.1);

COMMIT;
-- ERROR: could not serialize access due to read/write
--        dependencies among transactions
-- HINT: The transaction might succeed if retried.
PostgreSQL обнаруживает зависимость между транзакциями и отменяет одну из них с ошибкой serialization_failure. Приложение должно повторить отменённую транзакцию.
Цена SERIALIZABLE: возможны откаты транзакций → приложение должно уметь повторять. Используйте только когда логическая согласованность критична (финансы, расчёт бонусов, распределение ресурсов).

5. MVCC — Multi-Version Concurrency Control

Ключевая идея

PostgreSQL не блокирует строки для чтения. Вместо этого каждая транзакция видит свой снимок (snapshot) данных.

  • При UPDATE PostgreSQL не изменяет старую строку, а создаёт новую версию
  • Старая версия остаётся для транзакций, которые ещё её видят
  • VACUUM очищает «мёртвые» версии, когда они больше не нужны
-- Системные столбцы MVCC
SELECT
    ctid,   -- Физическое расположение (page, offset)
    xmin,   -- ID транзакции, создавшей строку
    xmax,   -- ID транзакции, удалившей/обновившей (0 = актуальна)
    mine_name
FROM dim_mine LIMIT 3;

MVCC: как работает UPDATE

Шаг Действие ctid xmin xmax Данные
1 INSERT (TX 100) (0,1) 100 0 «Северная»
2 UPDATE (TX 105) (0,1) 100 105 «Северная» (мёртвая)
(0,2) 105 0 «Северная-2» (новая)
3 VACUUM Строка (0,1) удалена, пространство освобождено
Следствия MVCC:
  • Чтение НЕ блокирует запись, запись НЕ блокирует чтение
  • UPDATE создаёт новые кортежи — таблица «раздувается» (bloat)
  • VACUUM необходим для очистки — autovacuum делает это автоматически

Снимки (Snapshots) транзакций

-- READ COMMITTED: новый снимок для КАЖДОГО запроса
BEGIN;
SELECT * FROM fact_production;  -- Снимок 1
SELECT * FROM fact_production;  -- Снимок 2 (может отличаться!)
COMMIT;

-- REPEATABLE READ: один снимок на всю транзакцию
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT * FROM fact_production;  -- Снимок 1
SELECT * FROM fact_production;  -- Тот же снимок 1!
COMMIT;
Практическая польза: Для длительных аналитических запросов REPEATABLE READ гарантирует согласованность данных на протяжении всего отчёта.

7. Иерархия блокировок в PostgreSQL

Блокировки действуют на разных уровнях

PostgreSQL использует многоуровневую систему блокировок. Каждый уровень защищает свой тип объекта.

Таблица

Table-level locks

Защищают структуру таблицы. Берутся автоматически при любой операции.

LOCK TABLE,
DDL-команды

Строка

Row-level locks

Защищают конкретные строки. Берутся при UPDATE, DELETE, SELECT FOR UPDATE.

FOR UPDATE,
FOR SHARE

Рекомендательная

Advisory locks

Определяются приложением. PostgreSQL не знает, что они защищают.

pg_advisory_lock()

Блокировки на уровне таблицы (Table-level)

Режим блокировки Когда берётся Что защищает Строгость
ACCESS SHARE SELECT Запрещает удаление/перестроение таблицы во время чтения Самая мягкая
ROW SHARE SELECT FOR UPDATE/SHARE Сигнализирует: «я собираюсь блокировать строки» Мягкая
ROW EXCLUSIVE INSERT, UPDATE, DELETE Сигнализирует: «я изменяю данные в таблице» Средняя
SHARE CREATE INDEX Запрещает изменение данных, пока строится индекс Строгая
SHARE ROW EXCLUSIVE CREATE TRIGGER, некоторые ALTER Запрещает изменение данных и параллельные DDL Строгая
EXCLUSIVE REFRESH MATERIALIZED VIEW CONCURRENTLY Разрешает только чтение (ACCESS SHARE) Очень строгая
ACCESS EXCLUSIVE ALTER TABLE, DROP TABLE, VACUUM FULL Полная монопольная блокировка — никто не может даже читать Максимальная
Ключевой принцип: Чем «строже» блокировка, тем больше других операций она блокирует. SELECT (ACCESS SHARE) совместим почти со всем. ALTER TABLE (ACCESS EXCLUSIVE) блокирует всех.

Блокировки на уровне строк (Row-level)

Режим Команда Что разрешает другим Что запрещает другим
FOR KEY SHARE Неявно при проверке FK SELECT, UPDATE (не ключевых столбцов) DELETE, UPDATE ключевых столбцов
FOR SHARE SELECT ... FOR SHARE SELECT, FOR KEY SHARE, FOR SHARE UPDATE, DELETE, FOR UPDATE
FOR NO KEY UPDATE Неявно при UPDATE (не ключевых столбцов) SELECT, FOR KEY SHARE UPDATE, DELETE, FOR SHARE, FOR UPDATE
FOR UPDATE SELECT ... FOR UPDATE, DELETE, UPDATE ключа Только SELECT (без FOR) Любые другие блокировки строки
Важно: Обычный SELECT (без FOR) никогда не берёт блокировку строк — он работает через MVCC и читает снимок данных. Поэтому SELECT не мешает UPDATE и наоборот.

Как уровни блокировок связаны

-- Пример: UPDATE одной строки в fact_production
BEGIN;
UPDATE fact_production
SET tons_mined = 155.0
WHERE production_id = 900001;
-- PostgreSQL автоматически берёт:
-- 1) ROW EXCLUSIVE lock на ТАБЛИЦУ fact_production
--    → другие SELECT, INSERT, UPDATE — ОК
--    → CREATE INDEX, ALTER TABLE — ЖДУТ
-- 2) FOR NO KEY UPDATE lock на СТРОКУ production_id = 900001
--    → SELECT этой строки — ОК (MVCC)
--    → UPDATE/DELETE этой же строки — ЖДУТ
COMMIT;  -- Обе блокировки сняты
Принцип: Table-level блокировка защищает структуру (чтобы никто не сделал ALTER TABLE). Row-level блокировка защищает данные конкретной строки (чтобы два UPDATE не перезаписали друг друга). Они работают вместе.

SELECT FOR UPDATE — явная блокировка строк

-- Блокируем строки для обновления
BEGIN;
SELECT * FROM fact_production
WHERE equipment_id = 1 AND date_id = 20250115
FOR UPDATE;  -- Другие транзакции ждут COMMIT/ROLLBACK

-- Безопасно обновляем
UPDATE fact_production
SET tons_mined = tons_mined * 1.05
WHERE equipment_id = 1 AND date_id = 20250115;
COMMIT;

-- Варианты FOR:
SELECT ... FOR UPDATE;         -- Эксклюзивная блокировка
SELECT ... FOR SHARE;          -- Разделяемая (другие могут читать)
SELECT ... FOR UPDATE NOWAIT;  -- Не ждать, ошибка при конфликте
SELECT ... FOR UPDATE SKIP LOCKED;  -- Пропустить заблокированные

Deadlock — взаимная блокировка

Сессия A

BEGIN;
-- Шаг 1: блокировка ID=1
UPDATE test_deadlock
SET amount = amount - 100
WHERE id = 1;

-- Шаг 3: ждёт ID=2 (!)
UPDATE test_deadlock
SET amount = amount + 100
WHERE id = 2;
-- ОЖИДАНИЕ...

Сессия B

BEGIN;
-- Шаг 2: блокировка ID=2
UPDATE test_deadlock
SET amount = amount - 200
WHERE id = 2;

-- Шаг 4: ждёт ID=1 (!)
UPDATE test_deadlock
SET amount = amount + 200
WHERE id = 1;
-- DEADLOCK DETECTED!
PostgreSQL обнаруживает deadlock (по умолчанию через 1 секунду) и отменяет одну из транзакций. Вторая транзакция продолжает работу.

Предотвращение Deadlock

Правило: единый порядок блокировки

Всегда блокируйте ресурсы в одном и том же порядке (например, по возрастанию ID).

CREATE FUNCTION safe_transfer(p_from INT, p_to INT, p_amount NUMERIC)
RETURNS VARCHAR LANGUAGE plpgsql AS $$
DECLARE
    v_first INT; v_second INT;
BEGIN
    -- Упорядочиваем ID для единого порядка блокировки
    IF p_from < p_to THEN
        v_first := p_from; v_second := p_to;
    ELSE
        v_first := p_to; v_second := p_from;
    END IF;

    -- Блокируем в отсортированном порядке
    PERFORM 1 FROM equipment_balance WHERE equipment_id = v_first FOR UPDATE;
    PERFORM 1 FROM equipment_balance WHERE equipment_id = v_second FOR UPDATE;

    -- Теперь безопасно обновляем
    UPDATE equipment_balance SET balance_tons = balance_tons - p_amount
    WHERE equipment_id = p_from;
    UPDATE equipment_balance SET balance_tons = balance_tons + p_amount
    WHERE equipment_id = p_to;

    RETURN 'OK';
END;
$$;

Обработка deadlock с повтором

CREATE FUNCTION retry_on_deadlock(p_from INT, p_to INT, p_tons NUMERIC)
RETURNS VARCHAR LANGUAGE plpgsql AS $$
DECLARE
    v_attempt INT := 0;
    v_max_retries INT := 3;
BEGIN
    LOOP
        v_attempt := v_attempt + 1;
        BEGIN
            UPDATE fact_production SET tons_mined = tons_mined - p_tons
            WHERE equipment_id = p_from AND date_id = 20250115;
            UPDATE fact_production SET tons_mined = tons_mined + p_tons
            WHERE equipment_id = p_to AND date_id = 20250115;

            RETURN format('OK за %s попыток', v_attempt);
        EXCEPTION
            WHEN deadlock_detected THEN
                IF v_attempt >= v_max_retries THEN
                    RAISE EXCEPTION 'Deadlock: % попыток исчерпано', v_max_retries;
                END IF;
                RAISE NOTICE 'Deadlock, попытка %/%', v_attempt, v_max_retries;
                PERFORM pg_sleep(random());  -- Случайная задержка
        END;
    END LOOP;
END;
$$;

8. Advisory Locks

Рекомендательные блокировки

Пользовательские блокировки, управляемые приложением. PostgreSQL не навязывает семантику — вы решаете, что они означают.

Функция Описание
pg_advisory_lock(id) Блокирующий захват (ждёт)
pg_try_advisory_lock(id) Неблокирующий (TRUE/FALSE)
pg_advisory_unlock(id) Освобождение
pg_advisory_xact_lock(id) Автоматически освобождается при COMMIT/ROLLBACK

Advisory Lock: защита ETL

CREATE FUNCTION run_daily_etl(p_date_id INT)
RETURNS VARCHAR LANGUAGE plpgsql AS $$
DECLARE
    v_acquired BOOLEAN;
BEGIN
    -- Пытаемся захватить блокировку (ID = дата)
    v_acquired := pg_try_advisory_lock(p_date_id::BIGINT);

    IF NOT v_acquired THEN
        RETURN format('ETL для даты %s уже выполняется', p_date_id);
    END IF;

    BEGIN
        -- Выполняем ETL
        INSERT INTO report_daily_production (...)
        SELECT ... FROM fact_production ...
        WHERE date_id = p_date_id ...;

        PERFORM pg_advisory_unlock(p_date_id::BIGINT);
        RETURN 'ETL завершён успешно';
    EXCEPTION WHEN OTHERS THEN
        -- ОБЯЗАТЕЛЬНО освобождаем при ошибке!
        PERFORM pg_advisory_unlock(p_date_id::BIGINT);
        RAISE;
    END;
END;
$$;
Типичные применения: защита от параллельного запуска ETL, распределённая координация, управление очередями задач.

9. Best Practices

Рекомендации по транзакциям

  1. Короткие транзакции: держите блокировки минимальное время
  2. Единый порядок блокировки: всегда по возрастанию ID
  3. Обрабатывайте deadlock: повтор с экспоненциальной задержкой
  4. Используйте SAVEPOINT в пакетных операциях
  5. REPEATABLE READ для аналитических отчётов
  6. Мониторьте длительные транзакции: pg_stat_activity
  7. COMMIT в процедурах для пакетной загрузки (не в функциях!)
  8. Advisory locks для координации процессов

Мониторинг транзакций

-- Длительные транзакции
SELECT pid, usename, state,
    now() - xact_start AS tx_duration,
    LEFT(query, 80) AS query
FROM pg_stat_activity
WHERE xact_start IS NOT NULL AND state != 'idle'
ORDER BY xact_start;

-- Ожидающие блокировки
SELECT blocked.pid AS blocked_pid,
    blocking.pid AS blocking_pid,
    LEFT(blocked.query, 50) AS blocked_query,
    now() - blocked.query_start AS wait_time
FROM pg_stat_activity blocked
JOIN pg_locks bl ON blocked.pid = bl.pid AND NOT bl.granted
JOIN pg_locks bk ON bl.locktype = bk.locktype
    AND bl.relation = bk.relation AND bl.pid != bk.pid AND bk.granted
JOIN pg_stat_activity blocking ON bk.pid = blocking.pid;

Транзакции: SQL vs DAX

Аспект PostgreSQL DAX / Power BI
Транзакции BEGIN / COMMIT / ROLLBACK Нет (только чтение)
Изоляция 4 уровня Snapshot (модель в памяти)
Конкурентность MVCC, блокировки Нет конфликтов (read-only)
Deadlocks Возможны, обнаруживаются Невозможны
Обновление данных INSERT/UPDATE/DELETE Refresh модели (полная/инкрементальная)
Вывод: DAX работает с «замороженным» снимком данных. Все проблемы конкурентности решаются на стороне SQL-источника при ETL.

Ответ: B — SAVEPOINT позволяет откатить часть транзакции через ROLLBACK TO, не теряя остальные изменения.

Квиз (продолжение)

4. Как PostgreSQL обрабатывает deadlock?

A) Ждёт бесконечно
B) Отменяет обе транзакции
C) Обнаруживает и отменяет одну из транзакций
D) Перезапускает сервер
Ответ: C — PostgreSQL обнаруживает циклическую зависимость и отменяет одну из транзакций (с ошибкой deadlock_detected). Вторая продолжает работу.

Квиз (продолжение)

5. Возможен ли Dirty Read в PostgreSQL?

A) Да, при READ UNCOMMITTED
B) Да, при отключении MVCC
C) Нет, PostgreSQL всегда обеспечивает минимум READ COMMITTED
D) Да, при параллельных запросах
Ответ: C — Даже при установке READ UNCOMMITTED PostgreSQL фактически работает на уровне READ COMMITTED. Dirty read невозможен.
