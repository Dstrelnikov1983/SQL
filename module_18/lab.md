## Задание 1. BEGIN / COMMIT / ROLLBACK (простое)

**Бизнес-задача:** Зафиксировать данные о добыче за одну смену. Все данные смены должны быть записаны вместе или не записаны вовсе.

**Требования:**

1. Начните транзакцию
2. Вставьте 5 записей в `fact_production` для одной смены (date_id = 20250310, shift_id = 1, mine_id = 1, различные equipment_id)
3. Проверьте, что записи видны в текущей сессии
4. Зафиксируйте транзакцию
5. Проверьте, что записи сохранились после COMMIT
6. Повторите с другой сменой (shift_id = 2), но выполните ROLLBACK вместо COMMIT
7. Убедитесь, что записи второй смены не сохранились

## Задание 3. ACID на практике (простое)

**Бизнес-задача:** Продемонстрировать атомарность — перевод тонн между двумя единицами оборудования.

**Требования:**

1. Подготовьте: создайте тестовую таблицу `equipment_balance`:
   ```sql
   CREATE TABLE equipment_balance (
       equipment_id INT PRIMARY KEY,
       balance_tons NUMERIC DEFAULT 0,
       CHECK (balance_tons >= 0)
   );
   INSERT INTO equipment_balance VALUES (1, 1000), (2, 500);
   ```
2. Реализуйте «перевод» 200 тонн с оборудования 1 на оборудование 2:
   - Уменьшите баланс оборудования 1 на 200
   - Увеличьте баланс оборудования 2 на 200
   - Оба UPDATE в одной транзакции
3. Проверьте балансы после COMMIT
4. Попробуйте перевести 1500 тонн с оборудования 2 (нарушит CHECK) — что произойдёт?
5. Убедитесь, что после ошибки оба баланса остались неизменными (атомарность)

## Задание 4. Уровни изоляции — сравнение (среднее, в парах)

**Бизнес-задача:** Понять, как уровень изоляции влияет на видимость данных при одновременной работе аналитиков.

**Требования:**

Работайте в двух сессиях (A и B). Для каждого эксперимента заполните таблицу.

1. **Эксперимент 1: READ COMMITTED**
   - A: `BEGIN;` выполните `SELECT COUNT(*) FROM fact_production WHERE date_id = 20250115;`
   - B: вставьте новую запись с date_id = 20250115
   - A: повторите тот же SELECT
   - Запишите: изменился ли результат?

2. **Эксперимент 2: REPEATABLE READ**
   - A: `BEGIN; SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;` тот же SELECT
   - B: вставьте ещё одну запись
   - A: повторите SELECT
   - Запишите: изменился ли результат?

3. **Эксперимент 3: Конфликт обновления при REPEATABLE READ**
   - A: `BEGIN; SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;` прочитайте запись
   - B: обновите ту же запись и COMMIT
   - A: попробуйте обновить ту же запись
   - Запишите: что произошло?

4. Заполните итоговую таблицу:

| Аномалия | READ COMMITTED | REPEATABLE READ | SERIALIZABLE |
|----------|:-:|:-:|:-:|
| Dirty Read | | | |

## Задание 5. Обработка конфликтов блокировок (среднее)

**Бизнес-задача:** Два оператора одновременно корректируют данные о добыче. Нужно обработать конфликт.

   ```sql
   CREATE FUNCTION safe_update_production(
       p_production_id INT,
       p_new_tons NUMERIC,
   ```

3. Протестируйте:
   - В сессии A начните транзакцию и заблокируйте запись (`SELECT ... FOR UPDATE`)

## Задание 6. Предотвращение Deadlock (среднее, в парах)

**Бизнес-задача:** Два процесса обновляют данные по шахтам. Нужно предотвратить взаимную блокировку.

**Требования:**

1. Создайте таблицу:
   ```sql
   CREATE TABLE mine_daily_stats (
       mine_id    INT,
       date_id    INT,
       total_tons NUMERIC DEFAULT 0,
       status     VARCHAR(20) DEFAULT 'pending',
       PRIMARY KEY (mine_id, date_id)
   );
   INSERT INTO mine_daily_stats (mine_id, date_id) VALUES (1, 20250301), (2, 20250301);
   ```
2. **Воспроизведите deadlock:**
   - A: обновите mine_id=1, затем mine_id=2
   - B: обновите mine_id=2, затем mine_id=1
   - Зафиксируйте сообщение об ошибке
3. **Исправьте deadlock:**
   - Создайте функцию `update_mine_stats(p_mine_ids INT[], p_date_id INT, p_tons NUMERIC[])`, которая:
     - Сортирует mine_ids по возрастанию
     - Блокирует строки в отсортированном порядке (`FOR UPDATE`)
     - Обновляет total_tons
   - Протестируйте: вызовите функцию из двух сессий одновременно с разным порядком mine_ids
   - Убедитесь, что deadlock больше не возникает

## Задание 7. Advisory Lock — защита ETL (среднее)

**Бизнес-задача:** Ночной ETL-процесс не должен запускаться параллельно.

**Требования:**

1. Создайте функцию `etl_daily_report(p_date_id INT)`:
   - Использует `pg_try_advisory_lock(p_date_id)` для проверки
   - Если блокировка не захвачена — возвращает 'ETL уже запущен'
   - Если захвачена:
     - Проверяет, не была ли дата уже обработана
     - Выполняет агрегацию данных и вставку в `report_daily_production`
     - Освобождает блокировку через `pg_advisory_unlock()`
   - В случае ошибки — освобождает блокировку и пробрасывает ошибку
2. Протестируйте:
   - Запустите из двух сессий одновременно (используйте `pg_sleep(5)` для имитации длительной работы)
   - Убедитесь, что вторая сессия получает сообщение 'ETL уже запущен'
3. Убедитесь, что блокировка корректно освобождается после ошибки

## Задание 8. MVCC — наблюдение (среднее)

**Бизнес-задача:** Понять, как PostgreSQL хранит версии строк.

1. Создайте тестовую таблицу:
   ```sql
   CREATE TABLE test_mvcc (
       id   INT PRIMARY KEY,
       data VARCHAR(50)
   );
   INSERT INTO test_mvcc VALUES (1, 'версия 1');
   ```
2. Выполните и запишите результаты:
   ```sql
   -- Начальное состояние
   SELECT ctid, xmin, xmax, * FROM test_mvcc;
   ```

   ```sql
   UPDATE test_mvcc SET data = 'версия 2' WHERE id = 1;
   UPDATE test_mvcc SET data = 'версия 3' WHERE id = 1;
   UPDATE test_mvcc SET data = 'версия 4' WHERE id = 1;
   ```
4. После каждого UPDATE фиксируйте: `ctid`, `xmin`, `xmax`
5. Ответьте на вопросы:
   - Как меняется ctid? Почему?
   - Что происходит с xmin после каждого UPDATE?
   - Когда xmax становится ненулевым?

## Задание 9. Процедура с управлением транзакциями (сложное)

**Бизнес-задача:** Пакетная загрузка данных о добыче за месяц с фиксацией по дням.

**Требования:**

1. Создайте процедуру `load_monthly_production(p_year INT, p_month INT)`:
   - Перебирает все дни месяца
   - Для каждого дня:
     - Агрегирует данные из `fact_production`
     - Вставляет в `report_daily_production` (UPSERT)
     - Фиксирует после каждого дня (`COMMIT` в процедуре)
   - При ошибке конкретного дня:
     - Логирует ошибку
     - Продолжает с следующим днём
   - Возвращает (через RAISE NOTICE) статистику: дней обработано, ошибок, строк вставлено
2. Протестируйте: `CALL load_monthly_production(2025, 1);`
3. Проверьте, что данные загружены: `SELECT COUNT(*) FROM report_daily_production WHERE report_date BETWEEN '2025-01-01' AND '2025-01-31';`
4. Проверьте, что при прерывании (Ctrl+C) часть данных уже зафиксирована (в отличие от обычной транзакции)

## Задание 10. Комплексный кейс: параллельная обработка смен (сложное)

**Бизнес-задача:** Три аналитика одновременно корректируют данные по трём сменам одного дня. Нужно обеспечить целостность.

**Требования:**

1. Создайте таблицу итогов:
   ```sql
   CREATE TABLE shift_summary (
       date_id     INT,
       shift_id    INT,
       mine_id     INT,
       total_tons  NUMERIC,
       total_trips INT,
       oee_percent NUMERIC,
       updated_by  VARCHAR(50),
       updated_at  TIMESTAMP DEFAULT NOW(),
       version     INT DEFAULT 1,
       PRIMARY KEY (date_id, shift_id, mine_id)
   );
   ```
2. Создайте функцию `update_shift_summary` с оптимистичной блокировкой:
   - Принимает: `p_date_id, p_shift_id, p_mine_id, p_total_tons, p_version`
   - Обновляет запись только если `version` совпадает с переданным
   - При несовпадении version — возвращает ошибку 'Данные были изменены другим пользователем'
   - При успехе — увеличивает version на 1
3. Протестируйте оптимистичную блокировку:
   - Оба аналитика читают запись (получают version = 1)
   - Первый обновляет (version становится 2)
   - Второй пытается обновить (version = 1 != текущий 2) — получает ошибку
4. Реализуйте функцию `refresh_shift_summary(p_date_id INT)`:
   - Использует `pg_try_advisory_xact_lock` для предотвращения параллельного запуска
   - Пересчитывает все смены для указанной даты из `fact_production`
   - Использует UPSERT
