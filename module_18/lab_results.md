> Модуль 18 ориентирован на практические эксперименты с транзакциями, которые выполняются интерактивно в двух сессиях SQL-клиента. Ниже приведены ожидаемые результаты для каждого задания.

## Задание 1. BEGIN / COMMIT / ROLLBACK

**Исходное состояние (date_id=20250310, shift_id=1, mine_id=1):** 5 записей

**После COMMIT (shift_id=1):**
- Вставлено 5 записей — они сохранены в базе
- `SELECT COUNT(*)` возвращает 10 (5 исходных + 5 новых)

**После ROLLBACK (shift_id=2):**
- Вставлено 5 записей — они отменены
- `SELECT COUNT(*)` для shift_id=2 возвращает прежнее значение

> COMMIT фиксирует изменения навсегда. ROLLBACK полностью отменяет все операции в транзакции.

## Задание 2. SAVEPOINT — частичная загрузка

**Сценарий:**

1. `BEGIN` -- транзакция открыта
2. `INSERT INTO fact_production ...` -- запись добычи
3. `SAVEPOINT sp_after_production`
4. `INSERT INTO fact_ore_quality ...` -- запись качества
5. `SAVEPOINT sp_after_quality`
6. `INSERT INTO fact_equipment_telemetry (sensor_id = 99999)` -- ОШИБКА: FK violation
7. `ROLLBACK TO sp_after_quality` -- откат только телеметрии
8. `COMMIT` -- фиксация production + quality

**Результат:** Записи production и quality сохранены. Запись telemetry отменена.

> SAVEPOINT позволяет откатить часть транзакции без потери уже выполненных операций.

## Задание 3. ACID на практике

**Начальное состояние:**

```
equipment_id | balance_tons
-------------+-------------
1            | 1000
2            |  500
```

**После перевода 200 тонн (1 -> 2):**

```
equipment_id | balance_tons
-------------+-------------
1            |  800
2            |  700
```

**Попытка перевести 1500 тонн с equipment_id=2:**

```
ОШИБКА: new row for relation "equipment_balance" violates check constraint
  Деталь: Failing row contains (2, -800).
```

**Состояние после ошибки (атомарность):**

```
equipment_id | balance_tons
-------------+-------------
1            |  800    (не изменилось)
2            |  700    (не изменилось)
```

> Благодаря атомарности, при нарушении CHECK-ограничения вся транзакция откатывается — ни один UPDATE не применяется.

## Задание 4. Уровни изоляции — сравнение

**Эксперимент 1: READ COMMITTED**
- Сессия A видит изменения, зафиксированные сессией B между двумя SELECT

**Эксперимент 2: REPEATABLE READ**
- Сессия A НЕ видит изменения, зафиксированные сессией B — результат стабилен

**Эксперимент 3: Конфликт обновления при REPEATABLE READ**
- Сессия A получает ошибку: `ERROR: could not serialize access due to concurrent update`

**Итоговая таблица:**

| Аномалия              | READ COMMITTED | REPEATABLE READ | SERIALIZABLE |
|-----------------------|:-:|:-:|:-:|
| Dirty Read            | Нет | Нет | Нет |
| Non-Repeatable Read   | Да  | Нет | Нет |
| Phantom Read          | Да  | Нет | Нет |

> В PostgreSQL REPEATABLE READ реализован через snapshot isolation, что предотвращает и phantom read (в отличие от стандарта SQL, где phantom read допускается на этом уровне).

## Задание 6. Предотвращение Deadlock

**Воспроизведение deadlock:**

```
Сессия A: UPDATE mine_daily_stats SET total_tons = 100 WHERE mine_id = 1;  -- OK
Сессия B: UPDATE mine_daily_stats SET total_tons = 200 WHERE mine_id = 2;  -- OK
Сессия A: UPDATE mine_daily_stats SET total_tons = 300 WHERE mine_id = 2;  -- ОЖИДАНИЕ
Сессия B: UPDATE mine_daily_stats SET total_tons = 400 WHERE mine_id = 1;  -- DEADLOCK!

ERROR: deadlock detected
DETAIL: Process 1234 waits for ShareLock on transaction 5678;
        blocked by process 5679.
        Process 5679 waits for ShareLock on transaction 5677;
        blocked by process 1234.
```

**Решение:** Сортировка mine_ids по возрастанию перед блокировкой. Обе сессии блокируют строки в одном порядке (сначала mine_id=1, потом mine_id=2), что исключает циклическое ожидание.

## Задание 7. Advisory Lock — защита ETL

**Сессия A:**
```sql
SELECT etl_daily_report(20250115);  -- Начинает обработку (pg_sleep(5))
```

**Сессия B (одновременно):**
```sql
SELECT etl_daily_report(20250115);  -- Получает: 'ETL уже запущен'
```

> Advisory Lock гарантирует, что только один экземпляр ETL-процесса обрабатывает данные за конкретную дату.

## Задание 8. MVCC — наблюдение

```
-- Начальное состояние:
ctid=(0,1), xmin=100, xmax=0, data='версия 1'

-- После UPDATE до 'версия 2':
ctid=(0,2), xmin=101, xmax=0, data='версия 2'

-- После UPDATE до 'версия 3':
ctid=(0,3), xmin=102, xmax=0, data='версия 3'

-- После UPDATE до 'версия 4':
ctid=(0,4), xmin=103, xmax=0, data='версия 4'
```

**Ответы на вопросы:**

1. **ctid меняется** при каждом UPDATE — PostgreSQL создаёт новую физическую копию строки (tuple) в другом месте страницы или на новой странице.
2. **xmin увеличивается** — это ID транзакции, создавшей текущую версию строки. Каждый UPDATE — это новая транзакция.
3. **xmax** становится ненулевым у *старой* версии строки (той, которую заменили). В текущей (актуальной) версии xmax = 0.

**После VACUUM:** ctid может измениться (если строка перемещена при уплотнении), xmin и xmax сохраняются.

## Задание 10. Комплексный кейс: параллельная обработка смен

**Оптимистичная блокировка:**

1. Аналитик A читает запись (version = 1)
2. Аналитик B читает ту же запись (version = 1)
3. Аналитик A обновляет: `UPDATE ... SET version = 2 WHERE version = 1` — успех
4. Аналитик B обновляет: `UPDATE ... SET version = 2 WHERE version = 1` — 0 строк обновлено, ошибка: «Данные были изменены другим пользователем»

> Оптимистичная блокировка через version-поле позволяет обнаруживать конфликты без длительных блокировок строк. `pg_try_advisory_xact_lock` используется для предотвращения параллельного пересчёта.
