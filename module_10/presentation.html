Тема 1. Что такое подзапрос?

Подзапрос (Subquery)

Запрос SELECT, вложенный в другой SQL-оператор (SELECT, INSERT, UPDATE, DELETE)

Синонимы: вложенный запрос, внутренний запрос, подчинённый запрос
Ключевая идея: подзапрос возвращает набор данных, который используется внешним запросом

Где можно использовать подзапросы?

СекцияЧто возвращаетПример
SELECT Скалярное значение SELECT (SELECT MAX(tons_mined) FROM fact_production)
WHERE Скаляр, список, таблица WHERE mine_id IN (SELECT mine_id FROM ...)
FROM Таблица (Inline View) FROM (SELECT ... GROUP BY ...) AS sub
HAVING Скалярное значение HAVING COUNT(*) > (SELECT AVG(...) FROM ...)
INSERT Набор строк INSERT INTO ... SELECT ...

Классификация подзапросов

Замкнутый (Self-Contained)

  • Не зависит от внешнего запроса
  • Можно выполнить отдельно
  • Выполняется один раз

Коррелированный (Correlated)

  • Зависит от внешнего запроса
  • Нельзя выполнить отдельно
  • Выполняется для каждой строки
По результату: скалярный (одно значение), однозначный (один столбец, много строк), табличный (много столбцов, много строк)

Порядок выполнения

Замкнутый подзапрос

1. Выполнить подзапрос → 2. Подставить результат → 3. Выполнить внешний запрос

Коррелированный подзапрос

1. Взять строку внешнего запроса → 2. Выполнить подзапрос с параметрами строки → 3. Повторить для каждой строки
Внимание: Коррелированные подзапросы потенциально медленнее — N выполнений подзапроса вместо одного!

Тема 2. Замкнутые подзапросы

Подзапросы, которые не зависят от внешнего запроса

Скалярный

Возвращает одно значение

Используется в SELECT, WHERE, HAVING

Многозначный

Возвращает список значений

Используется с IN, NOT IN, ANY, ALL

Скалярный подзапрос в WHERE

Задача: Найти все смены, где добыча руды выше средней

-- Оборудование «Руда+» с добычей выше средней за смену
SELECT fp.production_id,
       de.equipment_name,
       dm.mine_name,
       fp.tons_mined
FROM   fact_production fp
JOIN   dim_equipment de ON fp.equipment_id = de.equipment_id
JOIN   dim_mine dm      ON fp.mine_id = dm.mine_id
WHERE  fp.tons_mined > (
           SELECT AVG(tons_mined)
           FROM   fact_production
       )
ORDER BY fp.tons_mined DESC;
Как это работает: Подзапрос вычисляет среднее (один раз!), затем результат подставляется в WHERE

Скалярный подзапрос в SELECT

Задача: Для каждой шахты показать добычу и общее среднее

-- Добыча по шахтам с отклонением от среднего
SELECT dm.mine_name,
       SUM(fp.tons_mined)  AS total_tons,
       (SELECT AVG(tons_mined)
        FROM   fact_production) AS overall_avg,
       SUM(fp.tons_mined) -
       (SELECT SUM(tons_mined) / COUNT(DISTINCT mine_id)
        FROM   fact_production) AS deviation
FROM   fact_production fp
JOIN   dim_mine dm ON fp.mine_id = dm.mine_id
GROUP BY dm.mine_name
ORDER BY total_tons DESC;
Скалярный подзапрос в SELECT выполняется один раз и добавляется в каждую строку результата

Многозначный подзапрос: IN

Задача: Найти операторов, работающих на шахтах Курской области

-- Операторы шахт Курской области
SELECT o.tab_number,
       o.last_name || ' ' || o.first_name AS full_name,
       o.position,
       o.qualification
FROM   dim_operator o
WHERE  o.mine_id IN (
           SELECT mine_id
           FROM   dim_mine
           WHERE  region = 'Курская область'
       )
ORDER BY o.last_name;
IN проверяет, входит ли значение в список, возвращённый подзапросом. Эквивалент: = ANY (...)

NOT IN: осторожно с NULL!

Задача: Оборудование, которое ни разу не простаивало

-- ОПАСНО! Если в подзапросе есть NULL — вернёт пустой результат!
SELECT equipment_name
FROM   dim_equipment
WHERE  equipment_id NOT IN (
           SELECT equipment_id  -- может содержать NULL!
           FROM   fact_equipment_downtime
       );

-- БЕЗОПАСНАЯ версия:
SELECT equipment_name
FROM   dim_equipment
WHERE  equipment_id NOT IN (
           SELECT equipment_id
           FROM   fact_equipment_downtime
           WHERE  equipment_id IS NOT NULL
       );
Ловушка NULL: NOT IN со списком, содержащим NULL, всегда возвращает UNKNOWN → 0 строк. Используйте NOT EXISTS вместо NOT IN!

Операторы ANY и ALL

-- ANY: добыча выше ЛЮБОГО (минимального) значения дневной смены
SELECT de.equipment_name, fp.tons_mined
FROM   fact_production fp
JOIN   dim_equipment de ON fp.equipment_id = de.equipment_id
WHERE  fp.tons_mined > ANY (
           SELECT tons_mined
           FROM   fact_production
           WHERE  shift_id = 1  -- дневная смена
       );

-- ALL: добыча выше ВСЕХ значений ночной смены
SELECT de.equipment_name, fp.tons_mined
FROM   fact_production fp
JOIN   dim_equipment de ON fp.equipment_id = de.equipment_id
WHERE  fp.tons_mined > ALL (
           SELECT tons_mined
           FROM   fact_production
           WHERE  shift_id = 3  -- ночная смена
       );
ОператорСмыслЭквивалент
> ANYБольше хотя бы одного> MIN(...)
> ALLБольше всех> MAX(...)
= ANYРавен хотя бы одномуIN (...)

Тема 3. Коррелированные подзапросы

Коррелированный подзапрос

Ссылается на столбцы внешнего запроса. Выполняется заново для каждой строки внешнего запроса.

Строка 1 → подзапрос Строка 2 → подзапрос Строка 3 → подзапрос ... N строк
Производительность: Концептуально O(N × M), но оптимизатор PostgreSQL часто преобразует в JOIN

Оборудование с добычей выше среднего по своей шахте

-- Для каждой единицы оборудования сравниваем со средней по ЕЁ шахте
SELECT de.equipment_name,
       dm.mine_name,
       fp.tons_mined
FROM   fact_production fp
JOIN   dim_equipment de ON fp.equipment_id = de.equipment_id
JOIN   dim_mine dm      ON fp.mine_id = dm.mine_id
WHERE  fp.tons_mined > (
           SELECT AVG(fp2.tons_mined)
           FROM   fact_production fp2
           WHERE  fp2.mine_id = fp.mine_id   -- корреляция!
       )
ORDER BY dm.mine_name, fp.tons_mined DESC;
Корреляция: fp2.mine_id = fp.mine_id — подзапрос ссылается на столбец внешнего запроса

Коррелированный подзапрос в SELECT

Задача: Для каждой шахты показать кол-во внеплановых простоев

SELECT dm.mine_name,
       (SELECT COUNT(*)
        FROM   fact_equipment_downtime fd
        JOIN   dim_equipment de2 ON fd.equipment_id = de2.equipment_id
        WHERE  de2.mine_id = dm.mine_id
        AND    fd.is_planned = FALSE
       ) AS unplanned_downtime_count
FROM   dim_mine dm
WHERE  dm.status = 'active'
ORDER BY unplanned_downtime_count DESC;
Коррелированный подзапрос в SELECT — удобен для «добавления» агрегатной метрики из другой таблицы

Лучший оператор каждой шахты

Задача: Найти оператора с максимальной суммарной добычей в каждой шахте

-- Коррелированный подзапрос: максимум по каждой группе
SELECT dm.mine_name,
       do2.last_name || ' ' || do2.first_name AS best_operator,
       sub.total_tons
FROM   (
    SELECT mine_id, operator_id,
           SUM(tons_mined) AS total_tons
    FROM   fact_production
    GROUP BY mine_id, operator_id
) sub
JOIN dim_mine dm     ON sub.mine_id = dm.mine_id
JOIN dim_operator do2 ON sub.operator_id = do2.operator_id
WHERE sub.total_tons = (
    SELECT MAX(sub2.total_tons)
    FROM   (
        SELECT mine_id, operator_id,
               SUM(tons_mined) AS total_tons
        FROM   fact_production
        GROUP BY mine_id, operator_id
    ) sub2
    WHERE sub2.mine_id = sub.mine_id   -- корреляция
)
ORDER BY sub.total_tons DESC;

Тема 4. EXISTS и NOT EXISTS

Предикат EXISTS

Возвращает TRUE, если подзапрос возвращает хотя бы одну строку.

Возвращает FALSE, если подзапрос возвращает пустой набор.

Важно: EXISTS проверяет только наличие строк, а не их значения. Поэтому в подзапросе часто пишут SELECT 1

EXISTS: оборудование с аварийной телеметрией

-- Оборудование, у которого были тревожные показания датчиков
SELECT de.equipment_name,
       de.inventory_number,
       dm.mine_name
FROM   dim_equipment de
JOIN   dim_mine dm ON de.mine_id = dm.mine_id
WHERE  EXISTS (
           SELECT 1
           FROM   fact_equipment_telemetry ft
           WHERE  ft.equipment_id = de.equipment_id
           AND    ft.is_alarm = TRUE
       )
ORDER BY dm.mine_name, de.equipment_name;
Оптимизация: EXISTS прекращает поиск сразу после нахождения первой совпадающей строки (short-circuit evaluation)

NOT EXISTS: безопаснее, чем NOT IN

Задача: Найти шахты, где не было анализов качества руды

-- Шахты без проб качества руды
SELECT dm.mine_name, dm.region
FROM   dim_mine dm
WHERE  NOT EXISTS (
           SELECT 1
           FROM   fact_ore_quality foq
           WHERE  foq.mine_id = dm.mine_id
       );

-- Сравните с NOT IN (потенциальная проблема с NULL):
SELECT mine_name, region
FROM   dim_mine
WHERE  mine_id NOT IN (
           SELECT mine_id   -- если mine_id может быть NULL!
           FROM   fact_ore_quality
       );
Рекомендация: Всегда используйте NOT EXISTS вместо NOT IN — безопаснее и часто быстрее

EXISTS: шахты с проблемами качества

Задача: Шахты, где есть пробы с содержанием Fe ниже нормы (< 30%)

SELECT dm.mine_name,
       dm.region,
       dm.max_depth_m
FROM   dim_mine dm
WHERE  EXISTS (
           SELECT 1
           FROM   fact_ore_quality foq
           WHERE  foq.mine_id = dm.mine_id
           AND    foq.fe_content < 30.0
       )
AND    dm.status = 'active'
ORDER BY dm.mine_name;
Бизнес-кейс: MES-система «Руда+» отслеживает качество руды. Шахты с низким содержанием Fe требуют корректировки технологии добычи.

Двойной NOT EXISTS: «реляционное деление»

Задача: Операторы, которые работали на всех активных шахтах

-- Оператор, для которого НЕ СУЩЕСТВУЕТ шахты,
-- где он НЕ работал
SELECT o.last_name || ' ' || o.first_name AS operator_name
FROM   dim_operator o
WHERE  NOT EXISTS (
    SELECT 1
    FROM   dim_mine dm
    WHERE  dm.status = 'active'
    AND    NOT EXISTS (
        SELECT 1
        FROM   fact_production fp
        WHERE  fp.operator_id = o.operator_id
        AND    fp.mine_id = dm.mine_id
    )
);
Реляционное деление — один из самых мощных паттернов с NOT EXISTS. Читается: «нет такой шахты, где оператор НЕ работал».

Тема 5. Подзапросы в FROM (Inline Views)

Производная таблица (Derived Table)

Подзапрос в секции FROM создаёт временную таблицу, к которой можно обращаться как к обычной

SELECT sub.mine_name, sub.total_tons
FROM   (
    SELECT dm.mine_name,
           SUM(fp.tons_mined) AS total_tons
    FROM   fact_production fp
    JOIN   dim_mine dm ON fp.mine_id = dm.mine_id
    GROUP BY dm.mine_name
) sub                        -- обязательный псевдоним!
WHERE  sub.total_tons > 10000
ORDER BY sub.total_tons DESC;
Обязательно: Inline View всегда требует псевдоним (AS sub)

Агрегация агрегатов

Задача: Средняя добыча по шахтам (среднее от суммы по сменам)

-- Средняя сменная добыча по каждой шахте
SELECT mine_name,
       ROUND(AVG(shift_tons), 2) AS avg_shift_production,
       COUNT(*) AS total_shifts
FROM (
    SELECT dm.mine_name,
           fp.date_id,
           fp.shift_id,
           SUM(fp.tons_mined) AS shift_tons
    FROM   fact_production fp
    JOIN   dim_mine dm ON fp.mine_id = dm.mine_id
    GROUP BY dm.mine_name, fp.date_id, fp.shift_id
) shift_totals
GROUP BY mine_name
ORDER BY avg_shift_production DESC;
Inline View позволяет применить агрегацию к агрегированным данным — двухуровневая агрегация

Тема 6. Подзапросы vs JOIN

Когда что использовать?

СитуацияРекомендация
Проверка существования связи EXISTS (или SEMI JOIN)
Фильтрация по списку значений IN или EXISTS
Нужны столбцы из обеих таблиц JOIN
Сравнение с агрегатом по группе Коррелированный подзапрос или оконная функция
Двухуровневая агрегация Inline View или CTE

Один запрос — три варианта

Задача: Оборудование, у которого были простои

-- Вариант 1: IN
SELECT equipment_name FROM dim_equipment
WHERE equipment_id IN (SELECT equipment_id FROM fact_equipment_downtime);

-- Вариант 2: EXISTS
SELECT equipment_name FROM dim_equipment de
WHERE EXISTS (
    SELECT 1 FROM fact_equipment_downtime fd
    WHERE fd.equipment_id = de.equipment_id
);

-- Вариант 3: JOIN (DISTINCT нужен!)
SELECT DISTINCT de.equipment_name
FROM dim_equipment de
JOIN fact_equipment_downtime fd ON fd.equipment_id = de.equipment_id;
Все три вернут одинаковый результат. PostgreSQL часто преобразует их в один план (Semi Join).

SQL vs DAX: скалярный подзапрос

SQL (PostgreSQL)

-- Добыча выше средней
SELECT equipment_name,
       tons_mined
FROM   fact_production fp
JOIN   dim_equipment de
  ON   fp.equipment_id
     = de.equipment_id
WHERE  tons_mined > (
    SELECT AVG(tons_mined)
    FROM   fact_production
);

DAX

// Мера: средняя добыча
Avg Production =
    AVERAGE(
      fact_production[tons_mined]
    )

// Таблица: выше средней
EVALUATE
FILTER(
    ADDCOLUMNS(
        fact_production,
        "EquipName",
            RELATED(
              dim_equipment[equipment_name]
            )
    ),
    [tons_mined]
        > [Avg Production]
)

SQL vs DAX: коррелированный подзапрос

SQL (PostgreSQL)

-- Добыча выше средней
-- по своей шахте
SELECT mine_id,
       tons_mined
FROM fact_production fp
WHERE tons_mined > (
    SELECT AVG(fp2.tons_mined)
    FROM fact_production fp2
    WHERE fp2.mine_id
        = fp.mine_id
);

DAX (EARLIER)

// Вычисляемый столбец
AboveAvgForMine =
VAR CurrentMine =
    fact_production[mine_id]
VAR MineAvg =
    CALCULATE(
        AVERAGE(
          fact_production[tons_mined]
        ),
        FILTER(
            ALL(fact_production),
            fact_production[mine_id]
                = CurrentMine
        )
    )
RETURN
    fact_production[tons_mined]
        > MineAvg
EARLIER / переменные VAR — аналог корреляции. VAR CurrentMine хранит значение «текущей строки», как fp.mine_id в SQL.

SQL vs DAX: EXISTS

SQL (PostgreSQL)

-- Шахты с проблемами
-- качества
SELECT mine_name
FROM dim_mine dm
WHERE EXISTS (
    SELECT 1
    FROM fact_ore_quality foq
    WHERE foq.mine_id
        = dm.mine_id
    AND foq.fe_content < 30
);

DAX

// Шахты с проблемами
EVALUATE
FILTER(
    dim_mine,
    COUNTROWS(
        FILTER(
            RELATEDTABLE(
                fact_ore_quality
            ),
            fact_ore_quality[fe_content]
                < 30
        )
    ) > 0
)
В DAX нет прямого EXISTS. Используем COUNTROWS(FILTER(...)) > 0 или NOT ISEMPTY(...).

JOIN fact_equipment_downtime fd ON fd.equipment_id = de.equipment_id

SELECT mine_name FROM dim_mine

SELECT * FROM fact_production fp
WHERE tons_mined > (

Вопрос 1

Какой подзапрос выполняется один раз, а какой — для каждой строки?

A) Замкнутый — один раз, коррелированный — для каждой строки
B) Коррелированный — один раз, замкнутый — для каждой строки
C) Оба выполняются один раз
D) Зависит от оптимизатора
Ответ: A — Концептуально замкнутый выполняется один раз, коррелированный — для каждой строки (хотя оптимизатор может оптимизировать).

Вопрос 2

Что вернёт запрос, если в подзапросе NOT IN есть NULL?

SELECT * FROM dim_mine
WHERE mine_id NOT IN (SELECT mine_id FROM fact_ore_quality);
A) Все шахты без анализов качества
B) Пустой результат (0 строк)
C) Ошибку SQL
D) Все шахты
Ответ: B — Если в списке есть NULL, NOT IN возвращает UNKNOWN для каждой строки → 0 строк. Используйте NOT EXISTS!

A) SUMMARIZE
B) TOPN

D) UNION

Вопрос 4

Обязателен ли псевдоним для подзапроса в секции FROM?

SELECT * FROM (SELECT mine_id, SUM(tons_mined) FROM fact_production GROUP BY mine_id);
A) Нет, псевдоним необязателен
B) Да, PostgreSQL требует псевдоним для Inline View
C) Только если к подзапросу обращаются по имени
D) Только для Oracle, в PostgreSQL не нужен
Ответ: B — PostgreSQL требует обязательный псевдоним для подзапроса в FROM. Без него будет ошибка синтаксиса.

Вопрос 5

Какая конструкция наиболее эффективна для поиска «записей, у которых нет связанных»?

A) LEFT JOIN ... WHERE ... IS NULL
B) NOT IN (подзапрос)
C) NOT EXISTS (подзапрос)
D) Все три примерно одинаковы в PostgreSQL
Ответ: D — В PostgreSQL оптимизатор обычно приводит все три варианта к одному плану (Anti Join). Но NOT EXISTS безопаснее из-за поведения с NULL.
