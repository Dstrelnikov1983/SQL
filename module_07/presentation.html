7.1 Планы выполнения запросов

Как PostgreSQL обрабатывает ваш запрос

?

Как PostgreSQL обрабатывает запрос

SQL-запрос → Парсер → Перезапись → Планировщик → Исполнитель → Результат
  • Парсер — проверяет синтаксис, строит дерево разбора
  • Перезапись — применяет правила (views, rules)
  • Планировщик (оптимизатор) — выбирает наилучший план выполнения на основе статистики
  • Исполнитель — выполняет план, возвращает результат
Ключевой этап: Планировщик оценивает стоимость разных стратегий и выбирает самую дешёвую.

Что такое план выполнения?

Определение

План выполнения — это дерево операций (узлов), которое PostgreSQL построит для выполнения запроса. Каждый узел описывает одну операцию: сканирование таблицы, соединение, сортировку и т.д.

-- Пример: простой запрос к «Руда+»
EXPLAIN
SELECT * FROM fact_production
WHERE date_id = 20240315;
Index Scan using idx_fact_production_date on fact_production
  Index Cond: (date_id = 20240315)
  cost=0.29..8.30 rows=14 width=76

План читается снизу вверх и изнутри наружу: сначала выполняются вложенные узлы.

Предполагаемый vs фактический план

EXPLAIN

Предполагаемый план

  • Запрос не выполняется
  • Показывает оценку стоимости (cost)
  • Показывает оценку строк (rows)
  • Безопасен для DELETE/UPDATE

EXPLAIN ANALYZE

Фактический план

  • Запрос выполняется
  • Показывает реальное время (actual time)
  • Показывает реальное количество строк
  • Осторожно с UPDATE/DELETE!
Совет: Для безопасного анализа UPDATE/DELETE оберните в транзакцию: BEGIN; EXPLAIN ANALYZE DELETE ...; ROLLBACK;

Индекс в базе данных

Определение

Индекс — это отдельная структура данных, которая хранит отсортированные значения одного или нескольких столбцов с указателями на соответствующие строки таблицы.

  • Ускоряет операции поиска (SELECT с WHERE)
  • Ускоряет сортировку (ORDER BY) и соединения (JOIN)
  • Замедляет вставку, обновление и удаление (INSERT, UPDATE, DELETE)
  • Занимает дополнительное дисковое пространство

Когда индекс помогает?

Операция Без индекса С индексом
SELECT ... WHERE equipment_id = 5 Seq Scan: просмотр всех строк Index Scan: переход к нужной строке
SELECT ... ORDER BY date_id Чтение + сортировка в памяти Данные уже отсортированы в индексе
JOIN fact_production ON equipment_id Nested Loop с полным сканированием Быстрый поиск по ключу
INSERT INTO fact_production ... Просто добавить строку Добавить строку + обновить все индексы
Вывод: индексы — это компромисс между скоростью чтения и стоимостью записи.

Избирательность (Selectivity)

Определение

Избирательность = количество уникальных значений / общее число строк

Чем ближе к 1 — тем лучше индекс работает

Столбец (Руда+)УникальныхВсего строкИзбирательностьИндекс?
production_id (PK)10 00010 0001.0Отлично
date_id54710 0000.055Хорошо
equipment_id810 0000.0008Средне
mine_id210 0000.0002Плохо
shift_id210 0000.0002Плохо
Правило: Индекс эффективен, если запрос возвращает < 10-15% строк таблицы.

Плотность (Density) и кардинальность

Плотность

Плотность = 1 / количество уникальных значений

Показывает долю строк, приходящуюся на одно значение.

Кардинальность

Высокая — много уникальных значений (date_id, telemetry_id)

Низкая — мало уникальных значений (mine_id, shift_id)

-- Расчёт селективности в PostgreSQL
SELECT
    'equipment_id' AS column_name,
    COUNT(DISTINCT equipment_id) AS unique_values,
    COUNT(*) AS total_rows,
    ROUND(COUNT(DISTINCT equipment_id)::numeric / COUNT(*)::numeric, 6) AS selectivity
FROM fact_production;
Вывод: для столбцов с низкой кардинальностью используйте частичные индексы или составные индексы.

Глубина индекса

Что это?

Глубина B-tree — количество уровней от корня до листа. Определяет, сколько страниц индекса нужно прочитать для поиска одного значения.

Количество строкГлубина B-treeЧтений страниц
до 4001 (только корень)1
до ~150 00022
до ~60 000 00033
до ~25 000 000 00044
Вывод: Даже для таблицы с 60 миллионами строк B-tree индекс требует всего 3 чтения! Это O(log N).

Контекст «Руда+»

Проблема: В MES-системе «Руда+» таблица fact_equipment_telemetry содержит миллионы показаний датчиков. Запрос аварийных показаний за конкретную дату выполняется несколько секунд. 
-- Без индекса: Seq Scan по всей таблице
SELECT * FROM fact_equipment_telemetry
WHERE date_id = 20240315
  AND is_alarm = TRUE;

-- Время: ~3200 мс на 1 000 000 строк
-- С индексом: Index Scan — мгновенно
CREATE INDEX idx_telemetry_date_alarm
ON fact_equipment_telemetry(date_id)
WHERE is_alarm = TRUE;

-- Время: ~12 мс

7.3 Как PostgreSQL хранит данные

Heap-таблицы, страницы и кортежи

Heap-таблица (куча)

Что такое heap?

В PostgreSQL все обычные таблицы хранятся как heap (куча) — неупорядоченная коллекция строк. Новые строки добавляются туда, где есть свободное место.

Страница 0
equip_id=3, ПДМ-03
equip_id=1, ПДМ-01
equip_id=7, Самосвал-02
Страница 1
equip_id=12, Скип-01
equip_id=5, ПДМ-05
equip_id=2, ПДМ-02
Страница 2
equip_id=9, Вагон-03
equip_id=4, ПДМ-04
equip_id=15, Скип-03

Строки расположены в произвольном порядке. Чтобы найти equip_id=9, нужно просмотреть все страницы.

Структура страницы (8 КБ)

Page Header (24 байта)
ItemId 1 | ItemId 2 | ItemId 3 | ... (массив указателей)
Свободное пространство
Tuple 3 (строка данных)
Tuple 2 (строка данных)
Tuple 1 (строка данных)
Special Space
  • Размер страницы по умолчанию: 8 КБ
  • Кортежи растут с конца страницы, указатели — с начала
  • Адрес строки: ctid = (номер_страницы, номер_элемента)

ctid — физический адрес строки

-- Посмотрим физический адрес строк в таблице оборудования
SELECT ctid, equipment_id, equipment_name
FROM dim_equipment
ORDER BY ctid
LIMIT 10;
ctidequipment_idequipment_name
(0,1)1ПДМ-001
(0,2)2ПДМ-002
(0,3)3ПДМ-003
(0,4)4ПДМ-004
ctid = (0, 1) означает: страница 0, элемент 1. Это внутренний указатель, который используют индексы для быстрого доступа к строке.

Вопрос для обсуждения

Подумайте: Почему PostgreSQL не хранит данные в отсортированном виде, как словарь? Какие были бы преимущества и недостатки?
  • Heap быстрее на вставку — строка добавляется в первое свободное место
  • Сортированное хранение потребовало бы перемещения строк при каждом INSERT
  • PostgreSQL использует MVCC (многоверсионность) — старые версии строк остаются на месте
  • Для упорядоченного доступа есть индексы — отдельные структуры

7.4 Типы индексов в PostgreSQL

B-tree, Hash, GiST, SP-GiST, GIN, BRIN

Обзор всех типов индексов

Тип Когда использовать Операции Пример (Руда+)
B-tree Универсальный (по умолчанию) =, <, >, BETWEEN, LIKE 'x%' equipment_id, date_id
Hash Только точное равенство = (только) sensor_code (если всегда =)
GiST Геометрия, полнотекст, диапазоны @>, <@, &&, расстояние Координаты навигации (x, y, z)
SP-GiST Несбалансированные деревья, квадродеревья <<, >>, ~= IP-адреса, телефонные номера
GIN Массивы, JSONB, полнотекст @>, ?, ?|, @@ JSONB-метаданные, теги оборудования
BRIN Очень большие упорядоченные таблицы =, <, >, BETWEEN date_id в fact_telemetry
В 90% случаев используется B-tree. Остальные типы — для специализированных задач.

Hash-индекс

CREATE INDEX idx_sensor_code_hash
ON dim_sensor USING hash (sensor_code);
Плюсы:
  • Компактнее B-tree для длинных ключей
  • O(1) для точного поиска
Минусы:
  • Не поддерживает диапазоны (<, >)
  • Не поддерживает сортировку
  • Не поддерживает UNIQUE
Вывод: Hash-индекс редко оправдан. B-tree в большинстве случаев не менее эффективен для операции =.

GiST — обобщённое поисковое дерево

Generalized Search Tree

Сбалансированное дерево поиска, поддерживающее произвольные схемы разбиения пространства. Каждый внутренний узел содержит «ограничивающий прямоугольник» (или предикат), который описывает все данные в поддереве.

ПрименениеТип данныхОператоры
Геометрия / PostGISgeometry, point, box, circle@>, <@, &&, <-> (расстояние)
Диапазоныint4range, tsrange, daterange@>, <@, &&, -|-
Полнотекстовый поискtsvector@@
Сетевые адресаinet, cidr>>, <<, &&
Исключающие ограничениялюбые с оператором &&EXCLUDE USING gist
-- Пространственные данные навигации «Руда+» (PostGIS)
CREATE INDEX idx_location_geom
ON dim_location USING gist (geom);

-- Поиск всех локаций в радиусе 100 м от ПДМ
SELECT * FROM dim_location
WHERE ST_DWithin(geom, ST_MakePoint(45.12, 67.34)::geography, 100);

-- Диапазонный тип: смены как интервалы времени
CREATE INDEX idx_shift_range
ON dim_shift USING gist (tsrange(start_time, end_time));
Руда+: Подземная навигация ПДМ — поиск ближайших зон, определение участка по координатам.

SP-GiST — разделённое пространственное дерево

Space-Partitioned GiST

В отличие от GiST строит несбалансированные деревья: квадродеревья (quad-tree), k-d деревья, префиксные деревья (radix tree). Эффективен, когда данные имеют естественную иерархическую структуру разбиения.

СтруктураТип данныхКогда лучше GiST
Quad-treepoint, boxРавномерно распределённые 2D-точки
k-d treepointПоиск ближайших соседей в k измерениях
Radix treetextСтроки с общими префиксами (коды, URL)
Prefix treeinet, cidrIP-адреса, сетевые подсети
-- SP-GiST для текстового поиска по префиксу
-- (инвентарные номера оборудования: INV-LHD-001, INV-TRK-002...)
CREATE INDEX idx_equip_inventory_spgist
ON dim_equipment USING spgist (inventory_number);

-- SP-GiST для IP-адресов бортовых компьютеров
CREATE INDEX idx_sensor_ip_spgist
ON dim_sensor USING spgist (ip_address inet_ops);
Когда выбирать SP-GiST вместо GiST: данные с естественной иерархией (префиксы, координатные квадранты). SP-GiST компактнее и быстрее, но поддерживает меньше операторов.

GIN — обобщённый инвертированный индекс

Generalized Inverted Index

Хранит инвертированный список: для каждого элемента (слово, ключ, тег) — список строк, в которых он встречается. Идеален для составных значений, где одна строка содержит много элементов.

ПрименениеТип данныхОператоры
Полнотекстовый поискtsvector@@ (совпадение tsquery)
JSONBjsonb@>, ?, ?|, ?&
Массивыanyarray@>, <@, &&, =
Триграммыtext (pg_trgm)%, <->, LIKE '%abc%'
-- 1. Полнотекстовый поиск по комментариям к простоям
CREATE INDEX idx_downtime_comment_gin
ON fact_equipment_downtime
USING gin (to_tsvector('russian', comment));

SELECT * FROM fact_equipment_downtime
WHERE to_tsvector('russian', comment)
      @@ to_tsquery('russian', 'гидравлик* | насос*');

-- 2. JSONB-метаданные датчиков (параметры калибровки и т.д.)
-- ALTER TABLE dim_sensor ADD COLUMN metadata jsonb;
CREATE INDEX idx_sensor_metadata_gin
ON dim_sensor USING gin (metadata);

SELECT * FROM dim_sensor
WHERE metadata @> '{"calibrated": true}';

-- 3. Нечёткий поиск по LIKE '%текст%' (расширение pg_trgm)
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_equip_name_trgm
ON dim_equipment USING gin (equipment_name gin_trgm_ops);

SELECT * FROM dim_equipment
WHERE equipment_name LIKE '%самосвал%';

GIN: варианты классов операторов

jsonb_ops (по умолчанию)

  • Поддерживает @>, ?, ?|, ?&
  • Индексирует все ключи и значения
  • Больше по размеру
CREATE INDEX ON t
USING gin (data);

jsonb_path_ops

  • Поддерживает только @>
  • Индексирует хеш пути + значение
  • Компактнее в 2-3 раза
CREATE INDEX ON t
USING gin (data jsonb_path_ops);
Важно: Тип json (не jsonb) не поддерживает GIN-индексы напрямую. Для индексирования JSON-полей используйте тип jsonb или индексы по выражениям.
XML: Нативных GIN/GiST индексов для типа xml в PostgreSQL нет. Используйте индексы по выражениям (xpath()) или конвертируйте в jsonb.

BRIN — блочный диапазонный индекс

Идея BRIN

Хранит минимум и максимум значений для каждого блока из N страниц. Очень компактный, но эффективен только для физически упорядоченных данных.

-- BRIN идеален для телеметрии: данные вставляются по порядку дат
CREATE INDEX idx_telemetry_date_brin
ON fact_equipment_telemetry USING brin (date_id)
WITH (pages_per_range = 128);

-- Размер: ~48 КБ vs ~21 МБ для B-tree (на 10М строк)
B-tree: 21 МБ, Index Scan: 0.5 мс
BRIN: 48 КБ, Bitmap Scan: 3 мс

BRIN в 400 раз меньше, но в 6 раз медленнее. Идеален для огромных таблиц, где размер критичен.

А где Columnstore? Колоночное хранение в PostgreSQL

Вопрос от студентов: В SQL Server есть Columnstore Index, в Oracle — In-Memory Column Store. Есть ли аналог в PostgreSQL?

Ответ: нативных колоночных индексов в PostgreSQL нет

PostgreSQL хранит данные построчно (row-oriented). Но есть альтернативы:

РешениеТипОписание
BRIN Встроенный индекс Ближайший аналог zone maps — min/max по блокам. Для упорядоченных аналитических данных
Citus Columnar Расширение Колоночный формат хранения таблиц. Сжатие в 5-10x, быстрые агрегации
TimescaleDB Hypercore Расширение Гибридное row/columnar хранение для временных рядов 
-- Citus Columnar: колоночная таблица для архива телеметрии
CREATE TABLE telemetry_archive (LIKE fact_equipment_telemetry)
USING columnar;

-- Сжатие в 5-10 раз, быстрые SUM/AVG/COUNT
-- Но: нет UPDATE/DELETE, медленные точечные запросы
Citus Columnar будет подробно рассмотрен в отдельном модуле курса.

Квиз: выбор типа индекса

Ситуация: Таблица телеметрии — 50 млн строк, данные всегда вставляются последовательно по дате. Нужен индекс по date_id, но дисковое пространство ограничено.
A) B-tree
B) Hash
C) BRIN
D) GiST
Ответ: C) BRIN — минимальный размер, данные физически упорядочены — идеальный случай для BRIN.

7.5 B-tree индекс

Основной тип индекса в PostgreSQL

7.5.1 Структура B-tree

10 | 20 | 30
↙         ↓         ↘
3 | 5 | 8 12 | 15 | 18 22 | 25 | 28 32 | 35 | 40
↓        ↓        ↓        ↓
1,2,3 → 5,6,8 → 10,11,12 → ... →
  • Корень — верхний уровень, один узел
  • Внутренние узлы — навигация по диапазонам
  • Листья — значения ключей + ctid (указатели на heap)
  • Листья связаны между собой (двусвязный список) — для диапазонных запросов

7.5.2 Поиск по B-tree: пример

Задача: найти equipment_id = 15 в индексе idx_fact_production_equip
5 | 10 | 15 ← Шаг 1: 15 ≥ 10, идём вправо
11 | 13 | 14 15 | 16 | 17 ← Шаг 2: нашли узел
15 → ctid(4,7) ← Шаг 3: читаем heap по ctid
Сложность: O(log N). Для таблицы в 1 000 000 строк — всего ~3 чтения страниц вместо тысяч.

7.5.4 Синтаксис CREATE INDEX

CREATE [ UNIQUE ] INDEX [ CONCURRENTLY ] [ IF NOT EXISTS ]
    имя_индекса
ON имя_таблицы
    [ USING метод ]  -- btree (по умолчанию), hash, gist, gin, brin
    ( столбец1 [ ASC | DESC ] [ NULLS { FIRST | LAST } ]
      [, столбец2 ... ] )
    [ INCLUDE ( столбец, ... ) ]    -- покрывающий индекс (PG 11+)
    [ WHERE условие ]               -- частичный индекс
    [ WITH ( параметр = значение ) ] -- fillfactor, и др.
    [ TABLESPACE имя ];             -- табличное пространство
Имя индекса: рекомендуемый шаблон — idx_<таблица>_<столбец(ы)>, например idx_prod_date.

7.5.5 Обычный индекс

-- Индекс на один столбец
CREATE INDEX idx_prod_tons_mined
ON fact_production(tons_mined);

-- Проверяем: теперь Index Scan вместо Seq Scan
EXPLAIN ANALYZE
SELECT * FROM fact_production
WHERE tons_mined > 150;

-- Индекс с направлением сортировки
CREATE INDEX idx_prod_date_desc
ON fact_production(date_id DESC NULLS LAST);

-- Используется для ORDER BY DESC
EXPLAIN ANALYZE
SELECT date_id, equipment_id, tons_mined
FROM fact_production
ORDER BY date_id DESC
LIMIT 20;

7.5.6 UNIQUE индекс

-- Уникальный индекс: гарантирует уникальность значений
CREATE UNIQUE INDEX idx_equip_inventory_unique
ON dim_equipment(inventory_number);

-- Попытка вставить дубликат:
INSERT INTO dim_equipment (equipment_type_id, mine_id,
    equipment_name, inventory_number, manufacturer)
VALUES (1, 1, 'ПДМ-новый', 'INV-LHD-001', 'Caterpillar');

-- Ошибка: duplicate key value violates
-- unique constraint "idx_equip_inventory_unique"
Важно: Ограничения PRIMARY KEY и UNIQUE автоматически создают B-tree индекс. Не нужно создавать их отдельно!

7.5.7 Частичный индекс (Partial Index)

Идея

Индексируем только подмножество строк, соответствующих условию WHERE. Индекс меньше, быстрее обновляется.

-- Индекс только по аварийным показаниям телеметрии
CREATE INDEX idx_telemetry_alarm
ON fact_equipment_telemetry(date_id, equipment_id)
WHERE is_alarm = TRUE;

-- Используется при запросе:
SELECT * FROM fact_equipment_telemetry
WHERE date_id = 20240315
  AND equipment_id = 3
  AND is_alarm = TRUE;  -- условие совпадает с WHERE индекса
Руда+: Аварийные показания — менее 2% данных. Частичный индекс будет в 50 раз компактнее полного.

7.5.8 Индекс по выражению (Expression Index)

-- Индекс по вычисляемому выражению
CREATE INDEX idx_prod_year_month
ON fact_production ((date_id / 100));
-- date_id = YYYYMMDD, date_id / 100 = YYYYMM

-- Используется при запросе:
SELECT * FROM fact_production
WHERE date_id / 100 = 202403;

-- Индекс по LOWER для регистронезависимого поиска
CREATE INDEX idx_operator_lastname_lower
ON dim_operator (LOWER(last_name));

SELECT * FROM dim_operator
WHERE LOWER(last_name) = 'петров';
Важно: выражение в запросе должно точно совпадать с выражением в определении индекса.

7.5.9 Покрывающий индекс (INCLUDE)

-- Покрывающий индекс: дополнительные столбцы хранятся в листьях
CREATE INDEX idx_prod_date_covering
ON fact_production(date_id)
INCLUDE (equipment_id, tons_mined);

-- Index Only Scan: не обращается к heap!
EXPLAIN ANALYZE
SELECT date_id, equipment_id, tons_mined
FROM fact_production
WHERE date_id = 20240315;
INCLUDE vs ключевые столбцы: столбцы в INCLUDE не сортируются и не участвуют в поиске — они просто «приложены» к листьям для избежания обращения к heap. 
-- Запрос с дополнительным столбцом — уже НЕ Index Only Scan
SELECT date_id, equipment_id, tons_mined, fuel_consumed_l
FROM fact_production
WHERE date_id = 20240315;

7.5.10 Index Scan vs Index Only Scan

Index Scan

  1. Найти ключ в B-tree
  2. Получить ctid
  3. Обратиться к heap за остальными столбцами

Index Only Scan

  1. Найти ключ в B-tree
  2. Все нужные данные уже в индексе!
  3. Проверить Visibility Map

Heap Fetches: 0 = всё из индекса

Visibility Map: PostgreSQL проверяет, «видимы» ли строки для текущей транзакции. Если Heap Fetches > 0 — нужен VACUUM.

7.5.11 CONCURRENTLY — без блокировки

-- Стандартное создание: блокирует INSERT/UPDATE/DELETE
CREATE INDEX idx_example ON big_table(column);

-- CONCURRENTLY: не блокирует, но дольше создаётся
CREATE INDEX CONCURRENTLY idx_example
ON big_table(column);
Параметр Обычный CONCURRENTLY
Блокировка таблицыДа (ShareLock)Нет
Скорость созданияБыстрееМедленнее (2 прохода)
Можно в транзакции?ДаНет
ПрименениеРазработка, миграцииПродуктив
Совет: На продуктивных базах всегда используйте CONCURRENTLY, чтобы не блокировать работу приложения.

7.5.12 Композитные индексы

-- Один индекс на несколько столбцов
CREATE INDEX idx_prod_equip_date
ON fact_production(equipment_id, date_id);

Правило «левого префикса»

Композитный индекс (A, B, C) эффективно используется для запросов:

  • WHERE A = ? — да
  • WHERE A = ? AND B = ? — да
  • WHERE A = ? AND B = ? AND C = ? — да
  • WHERE B = ? — нет (A не указан)
  • WHERE B = ? AND C = ? — нет
  • WHERE A = ? AND C = ? — частично (A используется)

Порядок столбцов: наглядно

Аналогия с телефонной книгой: отсортирована по Фамилии, затем Имени.
ЗапросТелефонная книгаИндекс (A, B)
Найти «Петров» Перейти к букве «П» WHERE A = 'Петров' — быстро
Найти «Петров Иван» «П» → «Иван» WHERE A = 'Петров' AND B = 'Иван' — ещё быстрее
Найти всех «Иван» Просмотреть всю книгу! WHERE B = 'Иван' — индекс бесполезен

Как выбрать порядок столбцов?

Правило: сначала столбцы равенства, затем — столбцы диапазонов, потом — столбцы сортировки. 
-- Частый запрос в «Руда+»:
SELECT * FROM fact_production
WHERE equipment_id = 5           -- равенство
  AND date_id BETWEEN 20240301   -- диапазон
                   AND 20240331;

-- Правильный порядок: сначала равенство, потом диапазон
CREATE INDEX idx_prod_equip_date
ON fact_production(equipment_id, date_id);

-- Неправильный порядок:
CREATE INDEX idx_prod_date_equip
ON fact_production(date_id, equipment_id);
-- date_id в диапазоне — сканируется больше поддеревьев

Bitmap Index Scan — объединение индексов

-- Если есть два отдельных индекса: idx_equip и idx_shift
EXPLAIN ANALYZE
SELECT * FROM fact_production
WHERE equipment_id = 5
  AND shift_id = 2;
Bitmap Heap Scan on fact_production
  Recheck Cond: ((equipment_id = 5) AND (shift_id = 2))
  -> BitmapAnd
    -> Bitmap Index Scan on idx_fact_production_equip
    -> Bitmap Index Scan on idx_fact_production_shift
PostgreSQL может объединить несколько одноколоночных индексов через BitmapAnd/BitmapOr, но это менее эффективно, чем один композитный индекс.

7.5.13 Команда CLUSTER

CLUSTER

Перезаписывает таблицу, физически упорядочивая строки по указанному индексу. Аналог кластеризованного индекса в SQL Server, но однократная операция.

-- Физически упорядочить добычу по date_id
CLUSTER fact_production USING idx_fact_production_date;

-- Проверить: ctid теперь отражает порядок date_id
SELECT ctid, date_id, equipment_id, tons_mined
FROM fact_production
ORDER BY ctid
LIMIT 10;

-- Обновить статистику после CLUSTER
ANALYZE fact_production;

CLUSTER: особенности

Характеристика SQL Server (Clustered Index) PostgreSQL (CLUSTER)
Автоматическое поддержание порядкаДаНет
Новые строки в порядкеДаНет (в свободное место)
Блокировка таблицыПри созданииACCESS EXCLUSIVE lock
Повторный запускНе нужноПериодически
Когда полезен: аналитические таблицы «Руда+», которые загружаются пакетно (ETL) и редко обновляются. После загрузки данных — CLUSTER, затем ANALYZE.

7.6 Влияние индексов на INSERT/UPDATE/DELETE

Цена, которую мы платим за скорость чтения

Индексы замедляют запись

INSERT без индексов

Быстро

Добавить строку в heap

INSERT с 5 индексами

Медленнее в 2-3 раза

Добавить в heap + обновить каждый из 5 индексов

ОперацияВлияние индексов
INSERTКаждый индекс обновляется при каждой вставке
UPDATEВ PostgreSQL UPDATE = DELETE + INSERT (из-за MVCC). Все индексы обновляются, даже если изменён неиндексированный столбец
DELETEСтрока помечается как удалённая, индексы обновляются при VACUUM
HOT Update: Если изменённый столбец не входит ни в один индекс и на странице есть место — PostgreSQL может обновить строку без обновления индексов (Heap-Only Tuple).

Баланс: чтение vs запись

Сценарий Рекомендация
OLAP (аналитика) — много чтения, мало записи Больше индексов — ускорят аналитические запросы
OLTP (транзакции) — много записи Минимум индексов — только самые необходимые
ETL-загрузка (пакетная) Удалить индексы → загрузить → пересоздать
Смешанная нагрузка Мониторинг и оптимизация по метрикам
MES «Руда+»: аналитическая БД (OLAP) — индексы на факт-таблицах оправданы, данные загружаются пакетно.
