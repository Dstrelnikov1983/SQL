Тема 1. Типы данных PostgreSQL

PostgreSQL — одна из самых типоразнообразных СУБД

Числовые

INTEGER, BIGINT, NUMERIC, REAL, DOUBLE PRECISION, SERIAL

Символьные

CHAR(n), VARCHAR(n), TEXT

Дата и время

DATE, TIME, TIMESTAMP, INTERVAL

Логический

BOOLEAN — TRUE / FALSE / NULL

Специальные

JSON / JSONB, UUID, массивы, XML, hstore

Сетевые и геометрические

INET, CIDR, MACADDR, POINT, LINE, POLYGON

Зачем инженеру-аналитику знать типы данных?

  • Корректность запросов: оператор = работает по-разному для числа и строки
  • Производительность: INTEGER (4 байта) vs BIGINT (8 байт) vs TEXT
  • Точность вычислений: NUMERIC(10,2) для финансовых данных vs REAL
  • Валидация данных: СУБД не позволит записать «abc» в поле INTEGER
  • Правильные агрегации: SUM, AVG требуют числовой тип
В MES-системе «Руда+» используются: NUMERIC для тоннажа, TIMESTAMP для событий, BOOLEAN для флагов, VARCHAR для кодов.

Вопрос для обсуждения

Ситуация из «Руда+»

Инженер создаёт таблицу для хранения координат подземной навигации. Какой тип данных лучше выбрать для координат X, Y, Z?

  • A) INTEGER
  • B) REAL
  • C) NUMERIC(12,4)
  • D) TEXT
Ответ: C) NUMERIC(12,4) — точное хранение дробных координат без потерь округления. В dim_location именно так и сделано!

Приведение типов (CAST)

-- Стандартный синтаксис CAST
SELECT CAST('2024-03-15' AS DATE);
SELECT CAST(42 AS TEXT);
SELECT CAST('123.45' AS NUMERIC(10,2));

-- Синтаксис PostgreSQL (двойное двоеточие)
SELECT '2024-03-15'::DATE;
SELECT 42::TEXT;
SELECT '123.45'::NUMERIC(10,2);

-- Пример из «Руда+»: date_id хранится как INTEGER (20240315)
-- Преобразуем в строку для конкатенации
SELECT 'Отчёт за дату: ' || date_id::TEXT
FROM fact_production
LIMIT 5;
Оператор :: — это «синтаксический сахар» PostgreSQL, эквивалентен CAST(... AS ...).

Тема 2. Символьные данные

CHAR, VARCHAR, TEXT — три символьных типа PostgreSQL

ТипОписаниеМакс. длинаКогда использовать
CHAR(n) Фиксированная длина, дополняется пробелами n символов Коды фиксированной длины (ISO, валюты)
VARCHAR(n) Переменная длина с ограничением до n символов Названия, номера, коды
TEXT Переменная длина без ограничения ~1 ГБ Описания, комментарии, журналы
В PostgreSQL VARCHAR без указания длины и TEXT — идентичны по производительности. Разница лишь в ограничении длины.

Разбор строк: SPLIT_PART

Задача: разобрать инвентарный номер оборудования

-- Формат: "INV-LHD-001"
-- Часть 1: "INV" (префикс), Часть 2: "LHD" (тип), Часть 3: "001" (номер)

SELECT inventory_number,
       SPLIT_PART(inventory_number, '-', 1) AS prefix,
       SPLIT_PART(inventory_number, '-', 2) AS eq_type,
       SPLIT_PART(inventory_number, '-', 3) AS serial_no,
       -- Преобразуем номер в целое число (убирая ведущие нули)
       CAST(SPLIT_PART(inventory_number, '-', 3) AS INTEGER) AS serial_int
FROM dim_equipment;
SPLIT_PART — функция PostgreSQL (нет в стандарте SQL). В DAX аналога нет — приходится комбинировать LEFT, MID, SEARCH.

Тема 3. Шаблонный поиск (Pattern Matching)

Три уровня поиска по шаблону в PostgreSQL:

LIKE / ILIKE

Простые шаблоны: % (любые символы), _ (один символ)

SIMILAR TO

SQL-стандарт регулярных выражений (смесь LIKE и regex)

~ (POSIX regex)

Полноценные регулярные выражения. Мощнее, но сложнее

REGEXP_MATCHES и др.

Извлечение групп, замена по шаблону

LIKE и ILIKE

-- Оборудование, название которого начинается с "ПДМ"
SELECT equipment_name, model
FROM dim_equipment
WHERE equipment_name LIKE 'ПДМ%';

-- Оборудование, содержащее "Северная" (без учёта регистра)
SELECT mine_name
FROM dim_mine
WHERE mine_name ILIKE '%северная%';

-- Инвентарный номер вида INV-???-001 (точно 3 символа типа)
SELECT inventory_number
FROM dim_equipment
WHERE inventory_number LIKE 'INV-___-001';

-- Операторы с фамилией на "К" или "С"
SELECT last_name, first_name
FROM dim_operator
WHERE last_name LIKE 'К%' OR last_name LIKE 'С%';
ILIKE — расширение PostgreSQL для поиска без учёта регистра. В стандартном SQL — только LIKE.

SIMILAR TO и POSIX regex

-- SIMILAR TO: инвентарные номера ПДМ или самосвалов
SELECT inventory_number, equipment_name
FROM dim_equipment
WHERE inventory_number SIMILAR TO 'INV-(LHD|TRUCK)-%';

-- POSIX regex (~): извлечь числовую часть из кода
SELECT shaft_code,
       (REGEXP_MATCH(shaft_code, '(\d+)'))[1] AS numeric_part
FROM dim_shaft;

-- Найти все локации типа "забой" с номером
SELECT location_name
FROM dim_location
WHERE location_name ~ 'Забой\s*#?\d+';

-- REGEXP_REPLACE: нормализовать пробелы в комментариях
SELECT REGEXP_REPLACE(comment, '\s+', ' ', 'g') AS clean_comment
FROM fact_equipment_downtime
WHERE comment IS NOT NULL
LIMIT 5;

Шаблонный поиск: SQL vs DAX

SQL (PostgreSQL)

-- LIKE
SELECT equipment_name
FROM dim_equipment
WHERE equipment_name
      LIKE 'ПДМ%';

-- Регулярные выражения
SELECT equipment_name
FROM dim_equipment
WHERE inventory_number
      ~ '^INV-LHD-\d{3}$';

DAX

    dim_equipment,

Важно: DAX не поддерживает регулярные выражения. PostgreSQL значительно мощнее в обработке текста.

Усечение даты: DATE_TRUNC

-- Усечение до месяца (для группировки по месяцам)
SELECT DATE_TRUNC('month', full_date) AS month_start,
       COUNT(*) AS days_count
FROM dim_date
WHERE year = 2024
GROUP BY DATE_TRUNC('month', full_date)
ORDER BY month_start;

-- Усечение начала простоя до часа
SELECT DATE_TRUNC('hour', start_time) AS hour_start,
       COUNT(*) AS downtimes_count
FROM fact_equipment_downtime
GROUP BY DATE_TRUNC('hour', start_time)
ORDER BY hour_start;

-- Доступные уровни: microsecond, millisecond, second,
-- minute, hour, day, week, month, quarter, year, decade,
-- century, millennium
В DAX нет прямого аналога DATE_TRUNC. Используют DATE(YEAR(d), MONTH(d), 1) для усечения до месяца.

Пример: длительность простоев

Вычисление продолжительности через TIMESTAMP

-- Длительность простоя из временных меток
SELECT d.equipment_id,
       e.equipment_name,
       d.start_time,
       d.end_time,
       -- Разница в минутах
       EXTRACT(EPOCH FROM (d.end_time - d.start_time)) / 60
           AS calc_duration_min,
       d.duration_min AS stored_duration_min
FROM fact_equipment_downtime d
JOIN dim_equipment e ON d.equipment_id = e.equipment_id
WHERE d.end_time IS NOT NULL
ORDER BY calc_duration_min DESC
LIMIT 10;
EXTRACT(EPOCH FROM interval) возвращает количество секунд. Делим на 60 для минут, на 3600 для часов.

SELECT EXTRACT(EPOCH FROM INTERVAL '2 hours 30 minutes');
A) 2.5
B) 150
C) 9000
D) Ошибка
Ответ: C) 9000 — EXTRACT(EPOCH FROM ...) возвращает секунды: 2*3600 + 30*60 = 9000.

Бонус: специальные типы PostgreSQL

JSON / JSONB

-- Пример: телеметрия в формате JSONB
CREATE TABLE telemetry_raw (
    id         BIGSERIAL PRIMARY KEY,
    device_id  INT,
    payload    JSONB,
    received   TIMESTAMP DEFAULT NOW()
);

-- Вставка JSON-данных
INSERT INTO telemetry_raw (device_id, payload) VALUES
(1, '{"temperature": 45.2, "vibration": 3.1, "rpm": 1500}');

-- Извлечение полей
SELECT payload->>'temperature' AS temp,
       payload->>'vibration'   AS vib,
       (payload->>'rpm')::INT  AS rpm
FROM telemetry_raw;

-- Фильтрация по JSON-полю
SELECT * FROM telemetry_raw
WHERE (payload->>'temperature')::NUMERIC > 40;

Бонус: массивы и UUID

-- Массивы PostgreSQL
SELECT ARRAY['LHD', 'TRUCK', 'CART', 'SKIP'] AS equipment_types;

-- Проверка вхождения
SELECT equipment_name
FROM dim_equipment
WHERE equipment_type_id = ANY(ARRAY[1, 2]);

-- UUID: генерация уникального идентификатора
SELECT gen_random_uuid();
-- Результат: 'a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11'

-- UUID часто используется для распределённых систем,
-- где SERIAL может вызвать конфликты между узлами
Эти типы данных отсутствуют в DAX / Power BI. Если нужно работать с JSON-телеметрией, обрабатывайте данные на уровне PostgreSQL или Power Query (M).
